11. 틀릴 수 있는 AI 답변을 어떻게 보여줄 것인가

DataNexus Chat UI에서 답변 근거를 보여주는 방법. 국내외 7개 시스템을 살펴본 뒤 SQL과 실행 순서, fallback 여부, 출처를 화면에 남긴 과정.
5월 24, 2026 · 7 분 · Junho Lee | 번역:
  • En
목차
  • 먼저 실제 제품을 봤다
  • 정답처럼 보이는 답도 숫자만으로는 판단하기 어렵다
  • SQL은 숨기지 않는다
  • Trace는 답이 어떤 순서로 나왔는지 보여준다
  • Fallback은 실패를 숨기지 않는다
  • 출처가 있어야 업무에서 쓸 수 있는 답변이 된다
  • 정확도와 검토 가능성은 따로 본다
  • 이 원칙을 화면으로 옮기면
  • 화면에 남겨야 할 것

10편까지는 SQL 생성이 어디에서 자주 틀리는지 확인했다.

BIRD와 Spider를 돌리고 모델도 바꿔봤다. 오답은 COLUMN_BINDING, VALUE_BINDING, DERIVED_METRIC으로 나눠 봤다. 그런데 SQL 생성 오류를 보던 중 다른 질문이 생겼다.

그럼 맞은 SQL은 화면에 어떻게 보여줘야 할까.

정확도 중심으로 실험하던 동안 놓친 게 있었다. 데이터 분석 AI를 실제 업무 화면에 올리려면 근거를 확인할 방법이 필요하다. 숫자가 맞아도 어디서 나온 값인지 알 수 없으면 불안하다. SQL은 맞는데 화면이 보기 불편하면 다시 쿼리 에디터로 돌아간다.

DataNexus Chat UI에서는 답변 문장보다 검토 가능성이 더 중요했다. 숫자와 문장 뒤에 있는 근거가 화면에 보여야 했다.

먼저 실제 제품을 봤다#

화면 설계에 들어가기 전에 운영 중인 시스템부터 뜯어봤다. 해외 제품은 Databricks Genie, Snowflake Cortex Analyst, MicroStrategy AUTO, Vanna.ai였다. 국내 사례는 Qurify(SmartMind), MISO(GS Neotek & 52g), DARVIS(Dfinite)였다.

볼 때마다 같은 걸 확인했다. AI 답변을 어떻게 믿게 만들었는가.

제품마다 표현 방식은 달랐지만 답변만 던지지 않는다는 점은 비슷했다. Databricks Genie 는 ‘Trusted’ 배지와 ‘Fix it’, ‘Request review’ 동선을 답변 옆에 둔다. Snowflake Cortex Analyst 는 UI보다 API에 가깝지만 verified_query_used, question_category 같은 검증 필드를 응답 JSON에 남긴다. 확신이 없으면 SQL 대신 suggestion을 돌려준다.

MicroStrategy AUTO 는 Interpretation 패널로 질문 해석과 참조한 Metric/Dimension을 보여준다. 데이터 반환에 실패해도 SQL과 해석 과정이 사라지지 않는다. 차트도 질문 성격에 따라 막대그래프, 지도, 파이 차트로 고른다.

Vanna.ai 에서 참고할 만했던 건 4블록 구성과 피드백 루프였다. SQL/Table/Chart/Summary를 따로 조립하고 thumbs up/down 피드백은 Training Data로 보낸다. RAG가 틀렸을 때 고칠 재료가 화면에서 바로 생긴다.

국내 제품들은 접근 방식이 조금 달랐다. Qurify 는 SQL 실행 오류를 Self-Correction 루프로 다시 푼다. 단순 질문은 가벼운 모델로 보내고 복잡한 추론은 고성능 모델로 넘긴다. MISO 는 Ask → Look → Investigate → Voice → Evolve 흐름으로 성급한 판단을 막는다. DARVIS 는 기존 ERP나 MES를 건드리지 않고 그 위에 별도 레이어를 둔다. 보안이 3단계로 짜여 있고 사용자가 질문하기 전에 데이터 이상 징후를 먼저 알린다.

이 제품들을 보면서 DataNexus에도 필요한 레이어가 분명해졌다. 답변을 예쁘게 쓰는 것보다 SQL과 출처, 검증 상태와 fallback 여부를 남기는 일이 먼저였다.

정답처럼 보이는 답도 숫자만으로는 판단하기 어렵다#

실제 제품을 보면서 화면에 근거가 남아야 한다는 생각이 더 분명해졌다.

사용자가 이렇게 묻는다고 해보자.

2024년 총 주문 건수는?

AI가 이렇게 답한다.

2024년 총 주문 건수는 42건입니다.

겉으로 보면 별문제 없어 보인다. 하지만 업무 화면에서는 바로 확인하고 싶은 게 생긴다.

어떤 테이블을 조회했지?
2024년 조건은 어떻게 걸었지?
취소 주문은 포함했나?
이 숫자는 샘플 DB에서 나온 건가, 운영 DB에서 나온 건가?
SQL이 실행된 건가, 아니면 모델이 문장으로 만든 건가?

“42건"이라는 숫자 하나로는 판단하기 어렵다.

Genie가 ‘Trusted’ 배지를 달고 Snowflake가 응답 JSON에 검증 필드를 남기는 목적도 같다. 화면에 근거가 없으면 사용자는 답을 무비판적으로 믿거나 아예 무시한다. 업무 화면에서는 둘 다 곤란하다.

그래서 DataNexus Chat UI는 답변 텍스트만 보여주지 않는다. SQL과 실행 순서, 컨텍스트 출처, 대체 응답(fallback) 사용 여부와 검증 상태를 함께 남긴다.

SQL은 숨기지 않는다#

NL2SQL 제품을 만들다 보면 SQL을 감추고 싶다.

비즈니스 사용자는 보통 SQL을 직접 보지 않는다. 화면도 복잡해지고 AI가 대신 분석해준다는 인상도 약해진다.

Snowflake는 접기/펼치기 패널로 풀었다. 기본 화면은 단순하게 두고 필요할 때만 SQL을 연다.

DataNexus도 같은 방향이다. SQL을 개발자만 보는 내부 로그로 두지 않는다. 답변 안에서 접거나 펼칠 수 있는 형태로 노출한다.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

그 아래에는 사람이 읽는 답변이 나온다.

2024년 총 주문 건수는 42건입니다.

사용자는 결과를 먼저 보고 싶어 한다. SQL이 답변보다 먼저 보이면 안 된다. 그렇다고 SQL이 없어져도 안 된다.

엔터프라이즈 데이터 환경에서 SQL은 중요한 추적 근거다. 직접 읽지 않더라도 나중에 확인돼야 한다. 데이터 분석가나 DBA가 필요할 때 확인하고 오류가 발생했을 때 같은 쿼리로 재현한다.

SQL은 “이 숫자는 어디서 왔는가"를 따질 때 가장 먼저 잡히는 근거다.

Trace는 답이 어떤 순서로 나왔는지 보여준다#

AI 답변은 하나의 모델 호출로 끝나지 않는다.

DataNexus에는 온톨로지와 DataHub가 있다. PostgreSQL과 NL2SQL, 검증 로직도 함께 돈다. 답변은 자연스럽게 여러 단계를 거친다. Snowflake가 응답 JSON에 verified_query_used, question_category 같은 필드를 담는 것도 외부에서 이 순서가 추적되게 만들려는 거다.

사용자 질문이 들어오면 먼저 질문 의도를 분류한다. 지표 정의인지 실제 숫자 계산인지, 데이터 위치인지 영향 분석인지에 따라 처리 흐름이 달라진다.

“순매출이 뭐야?“는 SQL 실행보다 DataHub 용어집 조회가 먼저다. “2024년 총 주문 건수는?“은 PostgreSQL sample DB에서 SELECT를 실행한다. 사용자는 이 차이를 화면에서 봐야 한다.

모든 내부 로그를 다 보여줄 필요는 없다. source, validation, fallback 세 가지만 있어도 PostgreSQL 답변인지 DataHub 답변인지 구분된다.

Runtime Trace
source: PostgreSQL
validation: PASS
fallback: false

DataHub에서 온 답이라면 이렇게 보인다.

Runtime Trace
source: DataHub Glossary
validation: PASS
fallback: false

MicroStrategy AUTO는 AI가 어떤 Metric/Dimension을 참조했는지 Interpretation 패널로 보여준다. 엔터프라이즈 환경에서 “AI가 어떤 기준으로 이 답을 냈는가"가 불투명하면 채택이 어렵다. Trace가 없으면 사용자는 답의 출처를 추측해야 한다.

Fallback은 실패를 숨기지 않는다#

실패했는데 성공처럼 보이면 더 위험하다.

DB 연결이 끊기거나 SQL 실행이 실패했는데 화면에는 맞는 것처럼 보이는 문장이 나온다.

이러면 사용자는 틀린 답을 맞다고 믿게 된다.

Snowflake는 AI가 확신하지 못하는 질문에 SQL을 억지로 만들지 않는다. 대신 suggestion 콘텐츠를 반환해서 명시적으로 알린다. Qurify는 SQL 실패 시 Self-Correction 루프로 오류를 분석하고 다시 시도한다. 두 제품 모두 실패를 숨기지 않는다.

DataNexus에서도 대체 응답(fallback)을 숨기지 않는다. 실패했을 때도 답변을 줄 때가 있다. 데모 환경에서는 미리 준비한 답을 보여줄 수도 있다. 그렇다면 더더욱 표시해야 한다. 실제 DB 결과인지, fallback인지.

Runtime Trace
source: fallback demo response
validation: REVIEW
fallback: true

데모에서는 fallback 표시가 없으면 실제 DB 결과처럼 오해될 수 있었다.

사용자는 시스템이 무엇을 했고 무엇을 하지 못했는지 알게 된다. 운영자와 개발자에게는 장애 지점과 fallback 빈도가 그대로 추적 가능한 신호가 된다.

출처가 있어야 업무에서 쓸 수 있는 답변이 된다#

DataNexus에서 화면에 자주 띄워야 하는 출처는 DataHub다.

예를 들어 “순매출"이라는 용어가 있다. 회사마다 정의가 다르다. 반품과 할인만 빼는 곳이 있고 에누리나 세금 처리까지 포함하는 곳도 있다.

DDL에는 이런 정의가 없다.

테이블에는 amount, discount_amount, refund_amount 같은 컬럼이 있을 뿐이다. 순매출이라는 업무 개념은 데이터베이스 바깥에 있다.

그래서 용어 정의는 DataHub에서 가져와야 했다.

Chat UI에서 “순매출이 뭐야?“라고 물었을 때 답변이 DataHub 용어집에서 왔다면 source: DataHub Glossary로 남겨야 한다.

source: DataHub Glossary
term: 순매출

사용자는 이 답이 모델의 일반 지식이 아니라 조직이 관리하는 용어집에서 왔음을 안다.

온톨로지 기반 NL2SQL은 모델이 알아서 맞히는 방식이 아니다. 조직이 정의한 의미를 런타임에 가져와서 답한다.

DARVIS가 기존 ERP와 MES를 건드리지 않고 그 위에서 작동하는 것처럼 DataNexus도 기존 Data Warehouse를 바꾸지 않는다. DataHub와 Vanna 사이에 읽기 전용 레이어로 끼어든다.

정확도와 검토 가능성은 따로 본다#

NL2SQL 평가에서는 EX(Execution Accuracy)를 본다. 생성된 SQL을 실행했을 때 결과가 정답과 일치하는지를 본다.

하지만 EX가 통과해도 화면에서 믿고 쓰기 어려울 때가 있다.

SQL은 맞는데 문장이 너무 딱딱하다.

숫자도 맞다. 그런데 어떤 기준으로 낸 값인지 화면에서 안 보이면 사용자는 다음 판단을 못 한다.

출처가 있어도 찾기 어려우면 다시 물어봐야 한다.

그래서 DataNexus에서는 정확도 지표와 화면 검토성을 분리해서 본다.

지표보는 것
EXSQL 실행 결과가 정답과 같은가
CQ온톨로지가 Competency Question을 견디는가
Schema ComplianceSQL이 허용된 스키마와 정책 안에 있는가
Hallucination Rate없는 컬럼·용어·출처를 만들지 않는가
Query Router Accuracy질문이 SQL/Graph/Glossary/Vector 중 맞는 흐름으로 갔는가
Taste Gate사용자가 답을 검토하고 다음 판단으로 넘어갈 수 있는가

여기서는 이 검토 가능성을 Taste Gate라고 불렀다.

EX는 맞았는데 사용자가 “그래서 이걸 믿어도 돼?“라고 묻는다면 좋은 제품 경험이라고 보기 어렵다.

반대로 SQL이 실패했더라도 fallback임을 명확히 보여주고 검토할 여지를 남겨두면 사용자는 시스템 상태를 파악한다.

그래서 EX와 화면 검토성을 같은 지표로 묶지 않기로 했다.

이 원칙을 화면으로 옮기면#

화면에서는 이렇게 잡았다.

가장 먼저 손댄 곳은 답변 카드다. 상단은 신뢰 상태와 출처. 중간은 비즈니스 요약과 차트. SQL과 Trace는 하단 아코디언으로 여닫는다.

+-----------------------------------------------------------+
| [Status: PASS / REVIEW / FALLBACK] | [Source: DataHub]   |
+-----------------------------------------------------------+
| "이번 분기 순매출은 1,200억 원입니다."                      |
| [Chart: Bar / Line 자동 렌더링]                            |
+-----------------------------------------------------------+
| ▶ Show Evidence: SQL & Trace      ← 접기/펼치기 아코디언   |
+-----------------------------------------------------------+

임원은 위 두 블록만 보고 지나간다. 분석가는 아코디언을 열어 SQL을 확인한다. DBA는 데이터 계보 탭에서 어떤 테이블이 거쳐갔는지 본다.

하단 피드백 버튼은 Vanna의 Self-Learning 루프와 같은 통로다. 수정 피드백을 학습 데이터 후보로 남긴다.

기존 사내 ERP나 MES 위에 DataNexus를 붙일 때는 기존 화면을 갈아엎기보다 위에 얹는 방식이 맞다. DARVIS가 택한 방식이다.

별도 분석 페이지로 이동하게 하지 않는다. 실무자가 지금 보고 있는 화면 위에서 바로 조회한다. 기존 시스템을 건드리지 않으니 도입 협의도 단순해진다.

처음에는 요약만 보여준다. 더 알고 싶을 때 SQL과 데이터 계보를 펼친다. 한꺼번에 다 쏟아붓지 않는다.

출처가 없는 답변은 내보내지 않는다. 최소한 SQL, 문서 ID, 용어집 항목 중 하나는 붙어야 한다.

상태 라벨은 처음에 세 가지로 나눴다. PASS/REVIEW/FALLBACK. 사용자는 라벨만 보고도 다음 판단으로 넘어가야 한다.

화면에 남겨야 할 것#

정리하면 이렇다.

답변 텍스트는 짧을수록 좋다. 보고서 문장 말고 옆자리 분석가가 짧게 설명하듯이. AI가 바로 결론을 던지기보다 단계를 나눠야 사용자도 성급하게 결론 내리지 않는다.

SQL은 접어둘 수는 있어도 없애면 안 된다. DataHub 용어 정의처럼 SQL이 없는 답변이라면 억지로 만들지 않는다.

Trace와 상태 라벨은 세트다. PostgreSQL에서 온 건지, DataHub에서 온 건지, fallback인지 구분해야 한다. 상태는 PASS, REVIEW, FALLBACK 세 가지면 된다.

출처는 답변에 사용된 지식이 어디서 왔는지다. 현재 데모에서는 PostgreSQL 테이블과 DataHub Glossary Term을 출처로 쓴다. 이후에는 문서와 계보 정보까지 붙일 계획이다.

증거를 화면에 남기지 않으면 그냥 챗봇이 된다.
