11. 틀릴 수 있는 AI 답변, 화면에서는 어떻게 보여야 할까

DataNexus Chat UI에서 답변 근거를 보여주는 방법. SQL, Runtime Trace, fallback 여부, 권한 범위, 출처와 지식 그래프까지 답변 근거로 함께 보여주기로 했다
5월 23, 2026 · 18 분 · Junho Lee | 번역:
  • En
목차
  • 다른 제품은 근거를 어떻게 보여줬나
  • 지식 그래프는 출처를 확인하기 위한 화면이다
  • 그래프에서도 출처를 확인할 수 있어야 한다
  • 결과 숫자만으로는 답을 신뢰하기 어렵다
  • SQL을 화면에 남겨두는 이유
  • Trace로 답변 처리 과정을 확인한다
    • Route는 실행 경로별로 나눠야 한다
  • 답변 전에 모호성을 먼저 줄인다
  • 권한 범위까지 보여야 근거가 된다
  • Fallback 상태도 화면에 보여준다
  • 출처가 보여야 업무에서 쓸 수 있다
  • 한 답변 카드 안에서 값과 관계를 함께 본다
  • 대화형 편집은 곧바로 반영하지 않고 검토를 거친다
  • 정확도와 검토할 수 있느냐는 다른 문제다
  • 이 원칙을 실제 화면 설계에 적용한 결과

10편까지는 SQL 생성이 어디에서 자주 틀리는지 확인했다.

BIRD와 Spider를 돌리고 모델도 바꿔보면서 오답을 COLUMN_BINDING, VALUE_BINDING, DERIVED_METRIC 같은 유형으로 나눴다. SQL 오답을 계속 살펴보다 보니 질문이 바뀌었다.

그렇다면 생성된 SQL과 답변의 근거는 화면에서 어떻게 보여야 할까.

정확도 중심으로 테스트하다 보니 화면에서 사용자가 무엇을 확인하는지 놓쳤다. 업무에서 쓰는 데이터 분석 AI라면 숫자가 어디서 왔는지 바로 확인할 수 있어야 한다. 근거를 확인할 길이 없으면 숫자가 맞아 보여도 결국 SQL을 다시 돌린다.

DataNexus Chat UI에서는 답변 문장을 매끄럽게 쓰는 것만으로는 부족했다. 숫자와 문장에는 사용자가 확인할 근거를 함께 보여줘야 했다.

결국 사용자가 화면에서 직접 확인해야 신뢰가 생긴다. 사용자는 답변과 함께 SQL과 출처를 보고, 권한 범위와 검증 상태, fallback 여부까지 살핀다. 이 근거가 빠지면 맞는 숫자도 업무에 쓰기 어렵다.

다른 제품은 근거를 어떻게 보여줬나#

2026년 5월 기준으로 국내외 제품 문서와 데모 화면을 다시 확인했다.

해외 제품

  • Databricks Genie
  • Snowflake Cortex Analyst
  • Strategy(MSTR) AUTO
  • ThoughtSpot Spotter 3
  • Vanna.ai

국내 제품

  • Qurify(SmartMind)
  • MISO(GS Neotek & 52g)

제품 문서와 데모를 볼 때는 한 가지를 계속 확인했다.

사용자는 무엇을 확인해야 이 답을 업무에 쓸 수 있을까.

제품마다 표현은 달랐다. 그래도 답변 문장 하나만 던져두는 제품은 없었다.

  • Databricks Genie 는 Trusted assets 로 검증된 SQL query나 SQL function이 사용된 답변을 Trusted로 표시한다. Genie Space에서는 회사 데이터에 자연어로 질의한다. Genie Space의 Data 탭과 Instructions 탭에서는 테이블과 컬럼, instruction, SQL query, function 같은 컨텍스트를 확인한다. 또 Unity Catalog 권한이 적용되어 사용자가 볼 수 있는 데이터만 답변에 쓰인다.

  • Snowflake Cortex Analyst 는 API 응답 구조를 보고 화면 설계에 대한 직접적인 단서를 얻었다. 화면에 무엇을 보여줄지 판단할 수 있는 필드가 명시되어 있다. REST API 는 응답 content를 text, suggestion, sql로 구분한다. SQL content에는 verified_query_used 같은 필드가 포함된다. 응답에는 warnings와 response_metadata.question_category도 함께 담긴다. SQL을 만들기 어려운 질문에는 SQL 대신 suggestion을 반환한다.

  • Strategy(MSTR) AUTO 의 경우, Auto Dashboards와 Strategy AI 관련 문서, 데모 화면을 확인했다. 공개 문서를 보면 derived metric, selector, auto narratives, auto answers 같은 기능을 계속 넓히고 있다. 데모의 Interpretation 영역도 눈여겨봤다. 사용자가 던진 질문을 어떤 Metric과 Dimension으로 해석했는지 풀어 보여주는 방식은 DataNexus Trace UI에도 필요했다.

  • ThoughtSpot Spotter 3 는 답변 근거를 화면에 어떻게 보여줄 것인가라는 이 글의 주제와 가장 가까웠다. Spotter 3 는 구조화된 warehouse 데이터뿐 아니라 Slack, Salesforce, Jira, SharePoint 같은 업무 도구에 흩어진 정보까지 다룬다. 공개 자료에서는 방향이 비교적 또렷했다. 업무 도구에 흩어진 정보를 답변 맥락으로 가져오고, 사용자가 결과를 점검한 뒤 follow-up 분석으로 넘어가게 만든다.

  • Vanna.ai 는 학습 루프와 권한 처리 흐름 위주로 살펴봤다. Vanna 2.0 문서 는 LLM에 SQL 실행과 chart 생성 같은 tool을 연결한다. Tool Memory는 성공한 질문-SQL-결과 묶음을 다시 쓸 수 있는 형태로 저장한다. 사용자별 권한을 전체 흐름에 적용하는 방식도 설명한다. 공개 README에서도 진행 상태, SQL 코드, 표, 차트, 자연어 요약이 한 흐름에 놓인다.

국내 제품은 접근 방식이 조금 달랐다. 내부 자료는 제외하고 공개 소개자료와 릴리스 노트 기준으로만 봤다.

  • Qurify 는 자연어로 DB와 대화하는 Text-to-SQL 제품으로 볼 수 있다. 0.4.4 릴리스 노트 기준으로 생성된 SQL을 직접 편집하고 실행하거나 복사한다. 실행 결과를 인용해 후속 질문도 이어간다. 결과 CSV 다운로드와 tool call 인자 표시도 제공한다. DataNexus에서는 SQL을 밖으로 드러내는 흐름과 후속 질문 처리 방식을 참고했다.

  • MISO 는 NL2SQL 전용 제품보다는 업무 자동화형 AI 앱 빌더 성격이 더 강하다. 공개 사이트 기준으로 온프레미스 엔터프라이즈 AI 플랫폼과 Workflow App, Agent App 사례를 강조한다. 반복 보고서 자동화와 문서 검색 사례를 보며 DataNexus의 운영형 AI 화면을 구체화했다.

제품별 방식을 정리하고 나니 DataNexus 화면에 무엇을 남길지 정할 수 있었다. 답변 문장을 다듬기 전에 사용자가 SQL, 출처, 검증 상태, 권한 범위, fallback 여부를 먼저 확인할 수 있어야 했다.

semantic layer도 여기서 다시 정리됐다. 보통 semantic layer는 업무 용어와 지표 정의를 SQL이나 BI 화면에 연결하는 중간 계층으로 쓰인다. DataNexus의 Glossary와 Ontology, Knowledge Graph는 컬럼과 테이블 수준의 메타데이터에 업무 규칙, 개념 사이의 관계까지 묶는다.

그 관계가 답변 뒤에만 숨어 있으면 사용자는 다시 문서와 SQL을 찾아야 한다. 그래서 답변 카드 안에서 관계를 펼쳐 볼 화면이 필요했다.

지식 그래프는 출처를 확인하기 위한 화면이다#

여기서 말하는 근거는 답변에 쓰인 개념에서 출발해 출처, 관련 테이블과 컬럼까지 이어지는 연결망을 가리킨다. 처음에는 지식 그래프를 별도 분석 화면 정도로 생각했다.

노드와 엣지를 확대하고 이동하면서 관계를 탐색하는 화면을 떠올렸다. 기술 데모로는 충분히 인상적이다. 업무 화면에서는 예쁜 그래프보다 클릭했을 때 무엇을 확인할 수 있는지가 더 중요했다.

그래프를 업무 화면에 넣으려면 사용자가 이런 질문에 답을 얻을 수 있어야 한다.

  • 이 답변은 어떤 개념을 기반으로 도출됐나?
  • 그 개념은 어떤 데이터와 연결되어 있나?
  • 이 관계는 누가 정의했고 근거는 어디에 있나?
  • 계산식이나 지표 정의가 바뀌면 어떤 답변이 영향을 받나?

전통적인 검색은 키워드를 잘 찾는다. 문서 안에 “순매출"이라는 단어가 있으면 해당 문서를 찾아준다. 하지만 “순매출이 총매출, 반품, 할인, 에누리와 어떤 관계인가"까지 알기는 어렵다.

업무 답변에서는 단어가 들어 있는 문서보다 그 단어가 어떤 개념과 연결되는지가 더 중요할 때가 있다. 이 연결을 화면에서 따라가려면 온톨로지와 지식 그래프가 필요하다.

온톨로지라고 하면 무거워 보이지만 여기서는 조직이 쓰는 개념과 관계를 정리한 기준표 정도라고 보면 된다.

예를 들면 이런 식이다.

  • Concept: 고객, 제품, 주문, 순매출
  • Relationship: 고객이 주문을 생성한다, 순매출은 총매출에서 계산된다
  • Property: 주문일시, 결제수단, 주문상태
  • Rule: 환불은 결제 완료 이후에만 발생한다
  • Metric: 평균 주문 금액, 순매출, 전년 동기 대비 성장률

지식 그래프는 그 기준표를 실제 데이터와 연결한다. 개념은 노드가 되고 관계는 엣지가 된다. 문서와 테이블, 컬럼과 지표 정의, 계산식이 하나의 연결망으로 묶인다.

RDBMS를 지식 그래프로 대체하려는 시도와는 다르다.

값을 저장하고 트랜잭션을 처리하는 일은 RDBMS가 잘한다. 지식 그래프는 그 값이 업무에서 어떤 의미를 갖는지, 다른 개념과 어떻게 이어지는지를 설명한다.

DataNexus에서는 이 구분이 중요했다.

SQL만 있으면 값은 계산된다. 하지만 그 값이 어떤 업무 개념에서 나온 것인지 설명하려면 지식 그래프가 필요했다. DataNexus에서는 둘을 나란히 봐야 했다.

그래프에서도 출처를 확인할 수 있어야 한다#

지식 그래프 화면에서 개념만 시각화하면 부족하다. 그 개념을 뒷받침하는 출처와 연결 정보도 함께 보여야 한다.

사용자가 노드나 관계를 클릭하면 그 관계가 어디서 나왔는지 바로 열려야 한다. “순매출” 노드도 연결된 노드만 보여주면 부족하다. 그 정의가 나온 용어집 항목, 문서 문장, 관련 컬럼까지 이어서 열린다.

그래프를 이렇게 쓰면 사용자는 노드 몇 개를 보는 데서 멈추지 않는다. 답변에 쓰인 정의와 컬럼, 문서 근거까지 같은 화면에서 확인할 수 있다.

“순매출이 뭐야?“라고 물었는데 답변이 이렇게 끝나면 사용자는 더 볼 게 없다.

순매출은 총매출에서 반품, 할인, 에누리를 차감한 금액입니다.

이 문장이 맞더라도 사용자는 다시 묻게 된다.

  • 이 정의는 어디서 왔지?
  • 우리 회사 기준이 맞나?
  • 어떤 컬럼과 연결되어 있나?
  • 이 정의가 바뀌면 어떤 대시보드가 영향을 받나?

그래서 답변에는 최소한 이런 정보가 함께 있어야 한다.

source: Data Catalog Glossary
term: 순매출
source_id: urn:li:glossaryTerm:net_sales
related concepts: 총매출, 반품, 할인, 에누리
linked columns: orders.amount, orders.refund_amount, orders.discount_amount
validation: PASS

사용자는 이 정보를 바탕으로 지식 그래프 안에서 답변의 출처를 확인한다.

결과 숫자만으로는 답을 신뢰하기 어렵다#

Genie의 Trusted 표시, Snowflake의 검증 필드, Qurify의 SQL 노출 방식을 보며 결과 화면에서 근거를 함께 보여야 한다는 판단이 더 분명해졌다.

사용자가 이렇게 물었다고 해보자.

2024년 총 주문 건수는?

시스템이 이렇게만 답하면,

2024년 총 주문 건수는 12,486건입니다.

문장만 보면 답이 나온 것 같다. 그런데 실무자는 여기서 멈추지 않는다. 바로 이런 질문이 따라붙는다.

  • 어떤 테이블을 조회했지?
  • 2024년 조건은 어떻게 걸었지?
  • 취소 주문은 포함했나?
  • 이 숫자는 샘플 DB에서 나온 건가, 운영 DB에서 나온 건가?
  • SQL이 실행된 건가, 아니면 모델이 문장으로 만든 건가?

“12,486건"이라는 숫자 하나만으로는 이 값을 회의자료에 써도 되는지 판단하기 어렵다.

Genie의 Trusted 표시와 Snowflake의 응답 JSON 검증 필드도 같은 고민에서 나온 설계로 보였다. 그래도 표시만 붙이는 것으로는 부족하다. 화면에 근거가 없으면 사용자는 표시만 믿고 넘어가거나, 반대로 제품 자체를 믿지 않는다. 둘 다 업무용 도구에는 좋지 않다.

그래서 DataNexus의 Chat UI에서는 답변 결과만 내놓지 않는다. SQL 쿼리와 실행 순서를 함께 제공한다. 출처와 fallback 여부, 검증 상태, 권한 범위도 답변 카드 안에서 확인할 수 있게 한다.

SQL을 화면에 남겨두는 이유#

NL2SQL 제품을 설계하다 보면 SQL을 화면에서 숨기고 싶어지기 마련이다.

현업 사용자는 보통 SQL을 열어보지 않는다. 화면도 복잡해진다. AI가 대신 분석해 준다는 인상도 약해진다.

그래도 엔터프라이즈 환경에서 SQL을 완전히 숨기기는 어렵다.

DataNexus에서는 SQL을 개발자만 들여다보는 내부 로그처럼 다루지 않는다. 답변 안에서 접거나 펼칠 수 있게 보여준다.

답변 영역에는 자연어 답변을 먼저 보여준다.

2024년 총 주문 건수는 12,486건입니다.

그 아래에는 접고 펼 수 있는 SQL 블록을 배치한다.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

사용자는 결과를 먼저 본다. SQL이 답변보다 위에 있으면 화면이 복잡해 보이니 접어둔다. 대신 숫자가 이상해 보이는 순간에는 같은 자리에서 열 수 있어야 한다.

“이 숫자는 어디서 왔는가"를 따질 때 분석가와 DBA가 가장 먼저 여는 것이 SQL이다. 오류가 나도 같은 쿼리로 다시 실행해 오류의 범위를 좁힌다.

SQL만 보여줘도 문제가 남는다. 모든 사용자가 SQL을 읽지는 않는다. 먼저 볼 것은 질문이 어떤 값으로 매핑됐는지다.

해석된 기준

기간: 2024-01-01 - 2024-12-31
지표: 순매출
계산식: 총매출 - 반품 - 할인 - 에누리
데이터: orders
권한: 현재 사용자 권한 기준

이 표는 DataNexus에서 값 매핑 오류(VALUE_BINDING)를 줄이기 위한 설계다. 사용자는 “2024년"이 어떤 날짜 조건으로 바뀌었는지, “순매출"이 어떤 계산식으로 풀렸는지, “VIP 고객"이 어떤 세그먼트 규칙에 묶였는지 먼저 본다. 값이 틀렸다면 SQL 전체를 읽기 전에 기간이나 지표 정의를 화면에서 바로 고쳐 다시 실행한다.

SQL 탭에는 실행 전 점검 정보도 필요하다. dry-run 정보다. 쿼리만 봐서는 실행 위험을 판단하기 어렵다. 예상 행 수, 풀스캔 여부, 정책 검사 결과, 비용 수준이 한 화면에 있으면 실행 전 위험을 먼저 볼 수 있다.

SQL Inspect

EXPLAIN 요약: Index Scan
예상 행 수: 124,503
풀스캔 경고: 없음
예상 비용: 낮음
정책 검사: 통과

Trace로 답변 처리 과정을 확인한다#

Trace는 사용자 질문이 어떤 경로로 처리됐는지 보여주는 실행 기록이다. DataNexus에서는 AI 답변이 모델 호출 한 번으로 끝나지 않는다. 온톨로지와 카탈로그를 거친 뒤 질문 유형에 따라 DW/DM 조회, NL2SQL, 검증 로직으로 넘어간다.

앞에서 본 Snowflake의 verified_query_used, question_category, warnings도 비슷한 역할을 한다. 응답을 받은 시스템은 이 필드를 통해 처리 과정을 추적한다.

사용자 질문이 들어오면 시스템은 먼저 질문 의도를 분류한다. 처리 순서는 질문 유형에 따라 달라진다. 사용자가 지표 정의를 물으면 시스템은 카탈로그를 먼저 조회한다. 실제 숫자 계산이 필요하면 DW/DM으로 향한다.

“순매출이 뭐야?“는 SQL 실행보다 카탈로그 용어집 조회가 먼저다.

“2024년 총 주문 건수는?“은 DW/DM DB에서 SELECT를 실행한다.

화면은 이 차이를 바로 드러내야 한다.

모든 내부 로그를 다 노출할 필요는 없다. 사용자에게는 source, validation, fallback 세 가지만 제공해도 일차적인 확인으로는 충분하다. 다만 운영자와 분석가가 열어 볼 상세 Trace에는 더 많은 정보가 있어야 한다.

예를 들어 DB 조회 답변이라면 이런 기록이 필요하다.

Runtime Trace
trace_id: dnq_20260523_001
route: NL2SQL
source: DW/DM
source_id: mart.orders
semantic_context: Data Catalog Glossary + Ontology
verified_query: none
sql_execution: success
validation:
  schema_compliance: pass
  policy_check: pass
  source_binding: pass
permission_scope:
  user: sales_analyst
  rls_policy: sales_region_allowed
  masked_columns: []
fallback: false
warnings: []

카탈로그에서 온 답이라면 이렇게 보인다.

Runtime Trace
trace_id: dnq_20260523_002
route: Glossary
source: Data Catalog Glossary
source_id: urn:li:glossaryTerm:net_sales
sql_execution: not_applicable
validation:
  source_binding: pass
  policy_check: pass
permission_scope:
  glossary_visibility: internal
fallback: false
warnings: []

사용자 화면에는 이 중 일부만 꺼내면 된다.

source: DW/DM
validation: PASS
fallback: false

Route는 실행 경로별로 나눠야 한다#

초기 설계에서는 route를 NL2SQL과 Glossary 중심으로 뒀다. 운영 화면에서는 route를 더 세분화해야 했다. 같은 PASS라도 검증된 쿼리를 쓴 답과 새로 생성한 raw SQL을 실행한 답은 다르게 읽힌다.

route enum v2

semantic_routed
verified_query_matched
raw_sql_generated
raw_sql_fallback
glossary_only
graph_traversal
fallback_demo
unsupported_metric

이 구분을 단순 신뢰 배지처럼 쓰면 의미가 흐려진다. 이번 답변이 semantic layer를 거쳤는지, verified query와 매칭됐는지, raw SQL fallback이 발생했는지를 화면에 남긴다. 실제 운영 환경에서는 raw fallback이 생길 수밖에 없다. 좋은 경로를 탔다고 포장하기보다 실제 경로를 드러내는 편이 낫다.

앞에서 언급한 Strategy(MSTR) AUTO의 Interpretation 영역도 같은 맥락에서 만들어진 화면으로 보인다. 사용자가 던진 질문을 어떤 Metric과 Dimension으로 해석했는지 남겨야 나중에 따져볼 수 있다. 도출 기준이 불분명한 AI 답변은 업무에 그대로 쓰기 어렵다. Trace가 없으면 사용자는 출처를 확인하지 못한 채 다시 SQL이나 문서를 찾아야 한다.

답변 전에 모호성을 먼저 줄인다#

답변 뒤에 근거를 덧붙이는 것만으로는 부족하다. 실제 업무 질문은 처음부터 흐릿한 경우가 많다. “순매출 보여줘"라는 말 안에는 기간, 회계 기준, 차감 항목, 데이터 소스가 숨어 있다. 이럴 때 DataNexus는 바로 SQL을 만들지 않고 해석 후보를 먼저 제시한다.

[질문] 2024년 순매출 알려줘

DataNexus가 한 번 더 확인합니다.

○ 순매출_v1: 총매출 - 반품 - 할인 - 에누리
○ 순매출_KR: 국내 회계 기준, 세금 처리 포함
○ 순매출_global: IFRS 기준

[이 기준으로 실행] [직접 선택]

이 확인 창은 업무 기준이 여러 개일 때 사용자의 의도를 좁히는 단계다. 시스템은 온톨로지에서 후보 정의와 연결 컬럼, 권한 범위, 검증 상태를 가져와 선택지로 만든다.

선택한 기준은 같은 대화 안에서 유지된다.

{
  "carry_forward": {
    "accepted_metric": "urn:li:glossaryTerm:net_sales",
    "accepted_period": "2024",
    "accepted_filters": ["region = KR"],
    "rejected_interpretations": ["gross_sales_only"],
    "scope": "current_thread_only"
  }
}

대화가 길어질수록 사용자는 “방금 선택한 순매출 기준"이 유지된다고 기대한다. 그래서 DataNexus는 다음 질문에서도 이 기준을 유지하는지, 새 질문으로 보고 다시 해석하는지 화면에 표시한다.

권한 범위까지 보여야 근거가 된다#

출처만으로는 부족하다.

사용자가 볼 수 없는 테이블이나 문서가 근거로 쓰이면 안 된다. 같은 질문이라도 영업팀, 재무팀, 임원 계정마다 접근 범위가 다르다. 답변과 근거도 그 범위 안에서 달라진다.

그래서 Trace에는 source 옆에 permission scope도 기록한다.

permission_scope:
  user: sales_analyst
  allowed_domains: [Sales]
  row_level_filter: region IN ('KR-Seoul', 'KR-Gyeonggi')
  masked_columns: [customer_phone, customer_email]

이 정보를 모든 사용자에게까지 펼쳐 보일 필요는 없다. 다만 운영자와 감사 담당자는 나중에 열람할 수 있어야 한다.

신뢰 가능한 답변은 어느 데이터에서 왔는지 먼저 말한다. 그리고 그 답이 사용자의 권한 범위 안에서 나온 것인지도 남긴다.

Fallback 상태도 화면에 보여준다#

실패한 결과가 성공한 것처럼 보이는 경우가 가장 위험하다.

예를 들어 DB 연결이 끊겼는데 화면에는 “2024년 총 주문 건수는 12,486건입니다” 같은 문장이 그대로 나올 수 있다. 실패 표시가 없으면 사용자는 이 값을 실제 조회 결과로 받아들인다.

Snowflake는 AI가 확신하지 못하는 질문에서 SQL을 억지로 만들지 않고 suggestion 콘텐츠로 돌린다. Qurify는 사용자가 생성된 SQL을 편집하고 실행하는 흐름을 둔다. 방식은 다르지만 실패나 불확실성을 화면에 드러낸다.

DB 조회가 실패하면 DataNexus도 fallback 상태를 화면에 표시한다.

실패 후에도 답변이 필요한 상황은 있다. 특히 데모 환경에서는 미리 준비한 답을 띄우기도 한다. 이럴수록 실제 DB 결과인지 fallback인지 분명히 표시한다.

Runtime Trace
source: fallback demo response
sql_execution: failed
validation: FALLBACK
fallback: true
warnings:
  - DB connection failed. Demo response was used.

데모 환경인데 fallback 표시가 없으면 실제 DB 결과로 오해하기 쉽다.

사용자는 시스템이 어디까지 처리했는지 알아야 한다. 운영자와 개발자는 실패 지점과 fallback 발생 빈도까지 추적한다.

운영 라벨은 여기서 한 단계 더 나눈다. FALLBACK · demo_response와 FALLBACK · sql_execution_failed는 사용자에게 전혀 다른 의미로 다가온다. 정보가 부족하거나 권한 때문에 답할 수 없는 경우는 ABSTAIN으로 분리하는 편이 낫다.

출처가 보여야 업무에서 쓸 수 있다#

DataNexus에서 가장 자주 꺼내야 할 출처는 데이터 카탈로그다.

예를 들어 “순매출"이라는 용어가 있다. 회사마다 기준이 조금씩 다르다. 어떤 회사는 반품과 할인만 차감한다. 다른 회사는 에누리나 세금 처리까지 계산에 넣는다.

DDL은 테이블과 컬럼의 모양을 설명하지만 이런 비즈니스 규칙까지는 담지 않는다.

테이블에서 보이는 것은 amount, discount_amount, refund_amount 같은 컬럼이다. 이 컬럼들을 어떻게 묶어 순매출로 볼지는 별도의 업무 기준으로 정의해야 한다.

그래서 용어 정의는 카탈로그에서 가져온다.

Chat UI에서 “순매출이 뭐야?“라고 물었고 답변이 카탈로그 용어집에서 왔다면 source: Data Catalog Glossary도 표시한다.

source: Data Catalog Glossary
term: 순매출
source_id: urn:li:glossaryTerm:net_sales
owner: Sales Data Domain

사용자는 이 답이 조직이 관리하는 용어집에서 온 것인지 본다.

온톨로지 기반 NL2SQL에서는 의미 해석을 모델 추측에만 맡기지 않는다. 조직이 정의한 의미를 런타임에 가져와 답변에 연결한다.

한 답변 카드 안에서 값과 관계를 함께 본다#

지식 그래프를 화면에 넣어도 모든 사용자가 그래프부터 보지는 않는다. 대부분은 답변과 숫자를 먼저 본다. 그래프는 그다음 순서다.

DataNexus에서는 같은 답변 카드 안에서 요약, SQL, 카탈로그, 그래프 탭을 전환하도록 잡았다. 사용자는 먼저 요약 값을 보고 계산 방식이 궁금해질 때 SQL 탭을 열어 확인한다.

2024년 총 주문 건수는 12,486건입니다.

계산 방법은 SQL 탭에서 본다.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

의미 관계를 따져봐야 할 때는 그래프 탭으로 이동한다.

주문(Order)
 ├─ has property: 주문일자(order_date)
 ├─ has property: 주문상태(order_status)
 └─ counted as: 주문 건수(Order Count)

회사 기준이 걸리면 카탈로그 탭에서 용어 정의를 본다.

Glossary Term: 주문 건수
Definition: 기준 기간 내 생성된 주문 레코드 수
Owner: Sales Data Domain

요약, SQL, 카탈로그, 그래프를 한 화면에 모두 펼치면 화면이 무거워진다. 처음에는 답변 요약만 보여준다. 그러다 숫자가 이상하면 SQL을 열고 업무 정의가 걸리면 카탈로그 탭으로 이동한다. 관계와 영향 범위까지 따져야 할 때는 그래프를 열면 된다.

대화형 편집은 곧바로 반영하지 않고 검토를 거친다#

사용자가 “이 노드의 유형을 전략 고객으로 바꿔줘” 또는 “이 관계에 긴급 키워드를 추가해줘"처럼 요청하면, 시스템은 운영 지식에 반영할 변경 후보를 만들 수 있다.

이 방향이 맞다. 다만 엔터프라이즈 환경에서 변경 사항을 즉시 반영하면 위험하다. 용어 하나가 바뀌면 SQL 생성부터 대시보드와 리포트, 권한 정책까지 영향을 받는다.

그래서 DataNexus에서는 대화형 편집을 곧바로 운영 지식에 넣지 않고 다음과 같은 절차를 거친다.

사용자 수정 요청
  → 변경 후보 생성
  → 영향 범위 분석
  → 데이터 오너 검토
  → 승인 후 Glossary / Ontology 반영
  → 관련 인덱스 재동기화

AI에게 지식 수정을 요청할 수 있으면 운영은 편해진다. 그래도 운영 지식 베이스를 개인 메모장처럼 다루면 승인 흐름이 무너진다. 용어 하나가 조직의 기준을 바꿀 수 있으니 승인 절차와 변경 이력을 함께 남긴다.

그래서 운영 화면의 그래프는 관계만 보여주고 끝나지 않는다. 변경 영향 범위를 확인하는 검토 흐름까지 만든다.

정확도와 검토할 수 있느냐는 다른 문제다#

NL2SQL 평가에서는 보통 EX(Execution Accuracy)를 측정한다. 생성된 SQL을 실행했을 때 결과가 정답과 일치하는지 확인하는 지표다.

하지만 EX는 오프라인 평가 지표다. 정답 SQL이나 정답 실행 결과가 있는 벤치마크에서 측정한다.

운영 런타임에서는 보통 gold answer가 없다. 사용자가 실시간으로 던진 질문에는 정답 SQL이 미리 준비되어 있지 않기 때문이다.

그래서 운영 화면의 PASS를 EX 통과와 같은 뜻으로 쓰면 안 된다. EX는 벤치마크 정답과 실행 결과를 비교하는 오프라인 지표다. runtime validation은 지금 들어온 질문이 실행, 정책, 출처 검증을 통과했는지 보는 운영 기준이다.

PASS
- SQL 실행 또는 카탈로그 조회 성공
- Schema Compliance 통과
- 권한/RLS 정책 통과
- 출처 또는 verified query / glossary / graph node 매칭
- 치명적 warning 없음

REVIEW
- SQL은 실행됐지만 binding이 불확실함
- verified query 미사용
- 출처 매칭이 약함
- 질문이 모호하거나 router가 우회함
- 데이터 오너 또는 분석가 검토 필요

FALLBACK
- SQL 실행 실패
- DB 미조회
- suggestion only
- demo response
- 실제 운영 데이터가 아닌 대체 응답

SQL 실행 결과가 맞아도 업무에 바로 쓰기 어려운 경우가 있다. 자연어 답변이 지나치게 보고서 문장처럼 딱딱하면 읽는 사람이 다시 해석한다. 숫자가 맞더라도 어떤 기준으로 계산했는지 화면에 없으면 그 값을 보고서나 회의자료에 넣기 어렵다. 출처가 있어도 찾기 어렵다면 사용자는 데이터 오너에게 묻거나 시스템에 다시 질문한다.

TrustSQL 은 Text-to-SQL reliability를 다루며 답할 수 있는 질문에는 올바른 SQL을 만들어야 한다고 강조한다. 답할 수 없는 질문에는 보류할 줄 알아야 한다는 점도 함께 짚는다. LinkedIn의 enterprise Text-to-SQL 사례 에서도 schema와 문서, query log, wiki, code, domain knowledge를 지식 그래프로 구성한다. 이 사례에서는 다양한 사용자 의도와 후속 질문까지 chatbot UI에서 함께 다룬다.

DataNexus에서는 이 기준을 Taste Gate라고 부른다. EX가 통과해도 근거가 없거나 출처가 불분명하면 Taste Gate를 통과하지 못한다.

지표보는 것
EX오프라인 벤치마크에서 SQL 실행 결과가 정답과 같은가
CQ온톨로지가 Competency Question을 통과하는가
Schema ComplianceSQL이 허용된 스키마와 정책 안에 있는가
Hallucination Rate없는 컬럼, 용어, 출처를 만들지 않는가
Query Router Accuracy질문이 SQL, Graph, Glossary, Vector 중 적절한 경로로 라우팅됐는가
Runtime Validation운영 시점에 실행, 정책, 출처, warning 검증을 통과했는가
Taste Gate사용자가 답을 검토하고 다음 판단으로 넘어갈 수 있는가

EX가 통과했는데도 사용자가 곧바로 믿지 못한다면 화면에서 확인 경로를 충분히 제공하지 못한 것이다. SQL이 실패한 경우라도 fallback 표시와 확인할 근거가 있으면 사용자는 지금 시스템 상태를 이해한다. 그래서 EX가 통과한 답도 Taste Gate에서 한 번 더 걸러낸다.

이 원칙을 실제 화면 설계에 적용한 결과#

답변 카드 레이아웃부터 다시 짰다. 카드 상단에는 신뢰 상태와 출처를 둔다. 가운데에는 비즈니스 요약과 차트를 배치한다. SQL과 Trace는 카드 하단에서 접었다 펼 수 있게 했다.

+---------------------------------------------------------------+
| [Status: PASS / REVIEW / FALLBACK / ABSTAIN] | [Source: Data Catalog] |
+---------------------------------------------------------------+
| "이번 분기 순매출은 1,200억 원입니다."                        |
| [Chart: Bar / Line 자동 렌더링]                                |
+---------------------------------------------------------------+
| ▶ Show Evidence: SQL & Trace                                  |
+---------------------------------------------------------------+
| [Tabs] 1. SQL  2. Data Lineage  3. Metric Logic  4. Graph      |
+---------------------------------------------------------------+
| [👍 / 👎]       [수정 요청하기]  [쿼리 내보내기]                |
+---------------------------------------------------------------+

임원은 위쪽의 상태와 요약만 보고 업무 판단을 시작한다. 더 따져봐야 하는 분석가, DBA, 데이터 오너는 아래 탭으로 내려간다. 분석가는 SQL 탭을, DBA는 데이터 계보 탭을, 데이터 오너는 그래프 탭을 본다. 각자 쿼리와 테이블 흐름, 용어와 지표 관계를 따지는 식이다.

하단 피드백 버튼은 Vanna의 Self-Learning 루프처럼 사용자의 수정 의견을 다시 학습 재료로 모으는 자리다. DataNexus에서는 이 피드백을 바로 학습에 넣지 않고 검토 큐에 후보로 보낸다.

이 구조에서는 기본 카드와 Inspect 레이어를 분리한다. 기본 카드는 핵심 답변, PASS/REVIEW/FALLBACK/ABSTAIN 상태, 주요 출처, 차트만 담는다. Inspect 레이어에는 SQL, Trace, 권한 범위, Lineage, Metric Logic, Graph, 출처 하이라이트를 둔다.

레이어내용
1차 카드핵심 답변, 상태 표시, 주요 출처, 차트
Inspect 레이어SQL, Trace, 권한 범위, Lineage, Metric Logic, Graph, 출처 하이라이트

출처도 문서 하나를 연결하는 수준에서 끝나면 부족하다. 근거를 어느 단위로 제공할지 정한다. 문서 단위에서 멈출지, 문단과 문장, 숫자 단위까지 좁힐지에 따라 사용자가 검토하는 방식도 달라진다. DataNexus는 DataHub URN과 table.column FQN을 쓸 수 있다. 그래서 숫자와 문장 단위까지 근거를 좁혀 보여주기로 했다.

다만 Trusted, PASS, verified 같은 단어를 쉽게 붙이면 사용자는 내용을 확인하지 않고 표시만 믿는다. 상태 표시는 조건을 만족할 때만 사용한다. 출처를 얼마나 믿을 수 있는지, 잘못된 문서가 섞였을 가능성이 있는지도 운영 지표로 함께 본다.

처음에는 요약만 보인다. 더 알고 싶은 사용자가 SQL과 데이터 계보, 지식 그래프를 차례로 열어 보게 한다. 모든 근거를 처음부터 펼쳐 놓으면 답변보다 검토 도구가 먼저 보인다.

출처가 없는 답변은 화면에 띄우지 않도록 설계했다. 최소한 SQL, 문서 ID, 용어집 항목, 그래프 노드 중 하나는 연결되어 있어야 화면에 표시했다.

기본 상태는 PASS, REVIEW, FALLBACK이다. 답변하지 않는 편이 안전한 상태는 ABSTAIN으로 분리한다.

사용자는 이 표시만 보고도 다음 행동을 대략 정한다.

PASS: 지금 업무 판단에 사용 가능
REVIEW: 결과를 열어보고 한 번 확인해야 함
FALLBACK: 실제 DB 결과가 아니거나 정상 실행 결과가 아님
ABSTAIN: 정보 부족이나 권한 제한으로 답변하지 않음

여기서 PASS는 EX 통과를 뜻하지 않는다. 실행 성공, 정책 검증, 출처 매칭, 치명적 warning 없음까지 만족할 때 PASS로 표시한다.

모델 내부의 confidence score를 그대로 노출하는 방법도 있다. 보정되지 않은 점수는 사용자가 해석하기 어렵다. 그래서 PASS, REVIEW, FALLBACK 세 상태로 바로 써도 되는 답인지, 한 번 열어봐야 하는 답인지, 실제 DB 결과가 아닌지를 나누는 편이 명확하다.

답변 텍스트는 짧을수록 좋다. 옆자리 분석가가 짧게 설명하듯 쓰는 편이 낫다. SQL은 접어둘 수는 있지만 화면에서 완전히 없애서는 안 된다. Trace와 상태 표시는 답변이 DB에서 왔는지, 카탈로그에서 왔는지, fallback인지 구분해 준다.

운영자 화면에서는 답변 카드보다 더 긴 시간을 머무르며 봐야 한다. 사용자가 어떤 답을 열어봤는지, 어떤 citation을 눌렀는지, 어떤 route에서 fallback이 자주 발생하는지, REVIEW 상태가 실제 오류로 이어졌는지를 봐야 한다.

묶음지표 예시
모델 성능정답률, selective accuracy, abstention rate, fallback precision
근거 품질citation faithfulness, source coverage, verified-query hit rate
사용자 행동trace open rate, citation click-through, verification time
의존 보정over-reliance, under-reliance, trust calibration gap
안전poisoned-source susceptibility, misleading trusted-label rate
운영question_category 분포, warnings 발생률, fallback 발생률

이 지표는 사용자를 추적하려는 숫자가 아니다. AI 답변을 어디까지 자동화할지, 어디서부터 사람이 열어볼지 정하기 위한 운영 대시보드다.

DataNexus Chat UI를 설계하면서 계속 고민한 질문은 하나였다. 사용자가 이 답을 업무에 써도 되는지 판단하려면 무엇을 바로 확인해야 할까. 그 질문에 답하지 못하면 문장을 아무리 다듬어도 화면은 믿기 어렵다.
