11. 틀릴 수 있는 AI 답변, 화면에서는 어떻게 보여줘야 할까

DataNexus Chat UI에서 답변 근거를 보여주는 방법. 국내외 제품을 살펴본 뒤 SQL, Runtime Trace, fallback 여부, 출처, 지식 그래프까지 답변 근거로 함께 보여주기로 했다
5월 23, 2026 · 11 분 · Junho Lee | 번역:
  • En
목차
  • 먼저 타사 제품을 살펴봤다
  • 지식 그래프는 보기 좋은 그림이 아니라 근거 구조에 가깝다
  • 그래프에도 근거가 붙어야 한다
  • 정답처럼 보여도 결과 숫자만으로 신뢰 여부를 판단하기는 어렵다
  • SQL 쿼리도 숨기지 않는다
  • Trace는 답이 만들어진 경로를 보여준다
  • Fallback은 실패를 숨기지 않는다
  • 출처가 있어야 업무에 활용된다
  • 데이터 뷰와 그래프 뷰는 서로 보완해야 한다
  • 대화형 편집은 바로 반영보다 검토 큐가 먼저다
  • 정확도와 화면에서의 검토 가능성은 따로 본다
  • 이 원칙을 화면으로 옮기면
  • 결과 화면에서 보여줘야 할 것

10편까지는 SQL 생성이 어디에서 자주 틀리는지 확인했다.

BIRD와 Spider를 돌리고 모델도 바꿔봤다. 오답은 COLUMN_BINDING, VALUE_BINDING, DERIVED_METRIC 같은 유형으로 나눴다. 그렇게 SQL 생성 오류를 보던 중, 다른 질문이 생겼다.

그럼 정답 SQL은 화면에서 어떻게 보여줘야 할까.

정확도 중심으로 실험하던 동안 놓친 게 있었다. 데이터 분석 AI를 업무에 적용하려면 사용자가 근거를 확인할 수 있어야 한다. 숫자가 맞아도 어디서 나온 값인지 알 수 없으면 불안하다. SQL은 맞는데 화면이 보기 불편하면 다시 쿼리 에디터로 돌아간다.

DataNexus Chat UI에서는 답변 문장만큼 검토할 수 있는 근거가 중요했다. 숫자와 문장 뒤에 있는 근거가 화면에 남아야 했다.

먼저 타사 제품을 살펴봤다#

화면 설계에 들어가기 전에 운영 중인 타사 제품을 살펴봤다.

해외 제품

  • Databricks Genie
  • Snowflake Cortex Analyst
  • Strategy(MSTR) AUTO
  • Vanna.ai

국내 제품

  • Qurify(SmartMind)
  • MISO(GS Neotek & 52g)

살펴볼 때 기준은 하나였다.

사용자는 무엇을 봐야 AI 답변을 믿을 수 있을까.

제품마다 표현은 달랐지만, 단순히 답변만 내놓지 않는다는 점은 같았다.

  • Databricks Genie는 ‘Trusted’ 배지와 ‘Fix it’, ‘Request review’ 동선을 답변 옆에 둔다.

  • Snowflake Cortex Analyst는 UI보다 API에 가깝지만 verified_query_used, question_category 같은 검증 필드를 응답 JSON에 남긴다. 정확도가 낮다고 판단되면 SQL 대신 suggestion을 돌려준다.

  • Strategy(MSTR) AUTO는 Interpretation 영역에서 질문 해석과 참조한 Metric/Dimension을 확인하게 한다. 데이터 반환에 실패해도 SQL과 해석 결과를 남긴다.

  • Vanna.ai에서 참고할 만했던 건 4블록 구성과 피드백 루프였다. SQL/Table/Chart/Summary를 따로 조립하고 thumbs up/down 피드백은 Training Data로 보낸다. RAG가 틀렸을 때 결과 화면에서 바로 고칠 재료가 생긴다.

국내 제품은 접근 방식이 조금 달랐다.

  • Qurify는 SQL 실행 오류를 Self-Correction 루프로 다시 점검한다. 또 지식 그래프 탭에서 데이터 간 관계를 시각적으로 탐색하게 한다는 점이 눈에 들어왔다.

  • MISO는 Ask → Look → Investigate → Voice → Evolve 흐름으로 성급한 판단을 막는다.

이 제품들을 보면서 DataNexus 화면에 꼭 넣어야 할 요소가 보였다. 답변을 예쁘게 쓰는 것보다 SQL과 출처, 검증 상태와 fallback 여부를 남기는 일이 먼저였다.

그리고 한 가지가 더 필요했다.

관계를 직접 확인하는 화면이다.

지식 그래프는 보기 좋은 그림이 아니라 근거 구조에 가깝다#

처음에는 지식 그래프를 별도 분석 화면 정도로 생각했다.

노드와 엣지를 확대하고 이동하면서 관계를 탐색하는 화면을 떠올렸다. 기술 데모로는 멋있다. 하지만 실제 업무에서는 그래프가 예쁘게 보이는 것만으로는 부족하다.

중요한 건 이런 질문에 답할 수 있는지다.

  • 이 답변은 어떤 개념을 참조했나?
  • 그 개념은 어떤 데이터와 연결되어 있나?
  • 관계는 누가 정의했나?
  • 이 관계의 근거는 어디에 있나?
  • 계산식이나 지표 정의가 바뀌면 어떤 답변이 영향을 받나?

전통적인 검색은 키워드를 잘 찾는다. 문서 안에 “순매출"이라는 단어가 있으면 해당 문서를 찾아준다. 하지만 “순매출이 총매출, 반품, 할인, 에누리와 어떤 관계인가"까지 알기는 어렵다.

그래서 온톨로지와 지식 그래프가 필요하다.

온톨로지는 조직 안에서 쓰는 개념과 관계를 정리한 설계도에 가깝다.

예를 들면 이런 식이다.

  • Concept: 고객, 제품, 주문, 순매출
  • Relationship: 고객이 주문을 생성한다, 순매출은 총매출에서 계산된다
  • Property: 주문일시, 결제수단, 주문상태
  • Rule: 환불은 결제 완료 이후에만 발생한다
  • Metric: 평균 주문 금액, 순매출, 전년 동기 대비 성장률

지식 그래프는 이 설계도를 실제 데이터와 연결한 구조다. 개념은 노드, 관계는 엣지가 된다. 문서와 테이블, 컬럼과 지표 정의, 계산식이 하나의 연결망으로 묶인다.

여기서 중요한 건 RDBMS를 대체한다는 뜻이 아니다.

RDBMS는 값을 저장하고 트랜잭션을 처리하는 데 강하다. 지식 그래프는 의미와 관계를 드러내는 데 강하다. 둘은 경쟁 관계가 아니라 서로 다른 역할을 맡는다.

DataNexus에서는 이 구분이 중요했다.

SQL은 값을 계산하고, 지식 그래프는 그 값이 어떤 업무 개념에서 나온 것인지 설명한다. 둘 중 하나만으로는 부족했다.

그래프에도 근거가 붙어야 한다#

Qurify의 지식 그래프 기능에서 눈에 들어온 건 그래프 자체가 아니었다. 개념과 근거를 함께 보여주는 방식이었다.

사용자가 노드나 관계를 클릭했을 때, 그 관계가 어디서 나온 것인지 확인할 수 있어야 한다. 예를 들어 “순매출” 노드를 클릭하면 단순히 연결된 노드만 보여주는 게 아니라, 그 정의가 나온 용어집 항목이나 문서 문장, 관련 컬럼을 함께 확인해야 한다.

그래프가 이렇게 쓰이면 단순한 관계망이 아니라 답변 근거를 따라가는 UI가 된다.

예를 들어 “순매출이 뭐야?“라고 물었을 때 답변이 이렇게 끝나면 부족하다.

순매출은 총매출에서 반품, 할인, 에누리를 차감한 금액입니다.

이 문장이 맞더라도 사용자는 다시 묻게 된다.

  • 이 정의는 어디서 왔지?
  • 우리 회사 기준이 맞나?
  • 어떤 컬럼과 연결되어 있나?
  • 이 정의가 바뀌면 어떤 대시보드가 영향을 받나?

그래서 답변 옆에는 최소한 이런 정보가 있어야 한다.

source: Data Catalog Glossary
term: 순매출
related concepts: 총매출, 반품, 할인, 에누리
linked columns: orders.amount, orders.refund_amount, orders.discount_amount
validation: PASS

여기서 지식 그래프는 장식이 아니다. 답변의 출처를 따라가는 입구다.

정답처럼 보여도 결과 숫자만으로 신뢰 여부를 판단하기는 어렵다#

타사 제품을 보면서 결과 화면에 근거가 남아야 한다는 생각이 더 분명해졌다.

예를 들어 사용자가 이렇게 물었다고 하자.

2024년 총 주문 건수는?

AI가 이렇게만 답한다면,

2024년 총 주문 건수는 42건입니다.

겉으로는 별문제 없어 보인다. 하지만 바로 이런 질문이 남는다.

  • 어떤 테이블을 조회했지?
  • 2024년 조건은 어떻게 걸었지?
  • 취소 주문은 포함했나?
  • 이 숫자는 샘플 DB에서 나온 건가, 운영 DB에서 나온 건가?
  • SQL이 실행된 건가, 아니면 모델이 문장으로 만든 건가?

결국 “42건"이라는 숫자 하나로는 판단이 어렵다.

Genie가 ‘Trusted’ 배지를 달고 Snowflake가 응답 JSON에 검증 필드를 남기는 목적도 같다. 화면에 근거가 없으면 사용자는 답을 근거 없이 믿거나 아예 무시하게 된다. 둘 다 실제 업무에서는 곤란하다.

그래서 DataNexus의 Chat UI에서는 답변 결과만 보여주지 않는다. SQL 쿼리와 실행 순서, 어떤 출처를 참고했는지, fallback을 썼는지, 검증 상태가 어떤지도 함께 남기기로 했다.

SQL 쿼리도 숨기지 않는다#

NL2SQL 제품을 만들다 보면 SQL을 감추고 싶다.

현업 사용자는 보통 SQL을 직접 확인하지 않는다. 화면도 복잡해진다. 무엇보다 AI가 대신 분석한다는 제품 경험도 약해진다.

하지만 엔터프라이즈 데이터 환경에서는 SQL을 완전히 숨기면 안 된다.

Snowflake는 접기/펼치기 패널로 풀었다. 기본 화면은 단순하게 두고 필요할 때만 SQL을 연다.

DataNexus도 같은 방향이다. SQL을 개발자만 보는 내부 로그로 두지 않고, 답변 안에서 접거나 펼칠 수 있게 보여준다.

답변 카드에는 먼저 사람이 읽는 답변이 나온다.

2024년 총 주문 건수는 42건입니다.

그 아래에 SQL을 접거나 펼칠 수 있게 둔다.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

사용자는 결과를 먼저 보고 싶어 한다. SQL이 답변보다 먼저 보이면 안 된다. 그렇다고 SQL이 없어져도 안 된다.

SQL은 “이 숫자는 어디서 왔는가"를 따질 때 가장 먼저 확인할 수 있는 근거다. 분석가나 DBA가 필요할 때 열어보고, 오류가 발생했을 때 같은 쿼리로 재현해야 한다.

Trace는 답이 만들어진 경로를 보여준다#

AI 답변은 하나의 모델 호출로 끝나지 않는다.

DataNexus에는 온톨로지와 카탈로그가 있다. DW/DM, NL2SQL, 검증 로직도 함께 작동한다. 답변은 자연스럽게 여러 단계를 거친다.

Snowflake가 응답 JSON에 verified_query_used, question_category 같은 필드를 담는 것도 그 과정을 밖에서도 추적할 수 있게 만들려는 것이다.

사용자 질문이 들어오면 먼저 질문 의도를 분류한다. 지표 정의인지, 실제 숫자 계산인지, 데이터 위치인지, 영향 분석인지에 따라 처리 순서가 달라진다.

“순매출이 뭐야?“는 SQL 실행보다 카탈로그 용어집 조회가 먼저다.

“2024년 총 주문 건수는?“은 DW/DM DB에서 SELECT를 실행한다.

이 차이는 화면에서 바로 구분되어야 한다.

모든 내부 로그를 다 보여줄 필요는 없다. Runtime Trace, 즉 답변이 만들어진 경로 기록에는 source, validation, fallback 세 가지만 있어도 된다. 이 정도만 있어도 DB 조회 결과인지 카탈로그 답변인지 구분된다.

Runtime Trace
source: DW/DM
validation: PASS
fallback: false

카탈로그에서 온 답이라면 이렇게 보인다.

Runtime Trace
source: Data Catalog Glossary
validation: PASS
fallback: false

Strategy(MSTR) AUTO는 AI가 어떤 Metric/Dimension을 참조했는지 Interpretation 영역에서 확인하게 한다. 엔터프라이즈 환경에서는 “AI가 어떤 기준으로 이 답을 냈는가"가 불투명하면 믿고 사용하기 어렵다.

Trace가 없으면 사용자는 답의 출처를 추측해야 한다.

Fallback은 실패를 숨기지 않는다#

실패했는데 성공처럼 보이는 건 더 위험하다.

DB 연결이 끊기거나 SQL 실행이 실패했는데 화면에는 정답처럼 보이는 문장이 나온다.

이러면 사용자는 틀린 답을 맞다고 믿게 된다.

Snowflake는 AI가 확신하지 못하는 질문에 SQL을 억지로 만들지 않는다. 대신 suggestion 콘텐츠를 반환해서 명시적으로 알린다. Qurify는 SQL 실패 시 Self-Correction 루프로 오류를 분석하고 다시 시도한다. 두 제품 모두 실패를 숨기지 않는다.

DataNexus에서도 fallback, 즉 DB 조회 실패 시 사용하는 대체 응답을 숨기지 않는다.

실패했을 때도 답변을 보여줘야 할 때가 있다. 데모 환경에서는 미리 준비한 답을 보여줄 수도 있다. 그럴수록 실제 DB 결과인지 fallback인지 분명히 표시해야 한다.

Runtime Trace
source: fallback demo response
validation: REVIEW
fallback: true

데모에서는 fallback 표시가 없으면 실제 DB 결과처럼 오해될 수 있었다.

사용자는 시스템이 무엇을 했고 무엇을 하지 못했는지 알 수 있어야 한다. 운영자와 개발자는 어디서 실패했는지, fallback이 얼마나 자주 발생하는지 추적할 수 있어야 한다.

출처가 있어야 업무에 활용된다#

DataNexus에서 가장 자주 보여줘야 할 출처는 데이터 카탈로그다.

예를 들어 “순매출"이라는 용어가 있다. 회사마다 정의가 다르다. 반품과 할인만 빼는 곳이 있고 에누리나 세금 처리까지 포함하는 곳도 있다.

DDL에는 이런 정의가 없다.

테이블에는 amount, discount_amount, refund_amount 같은 컬럼이 있을 뿐이다. 순매출이라는 업무 개념은 데이터베이스 바깥에 있다.

그래서 용어 정의는 카탈로그에서 가져와야 한다.

Chat UI에서 “순매출이 뭐야?“라고 물었을 때 답변이 카탈로그 용어집에서 왔다면 source: Data Catalog Glossary로 남겨야 한다.

source: Data Catalog Glossary
term: 순매출

사용자는 이 답이 모델의 일반 지식이 아니라 조직이 관리하는 용어집에서 왔다는 점을 확인한다.

온톨로지 기반 NL2SQL은 모델이 알아서 맞히는 방식이 아니다. 조직이 정의한 의미를 런타임에 가져와서 답하는 방식이다.

데이터 뷰와 그래프 뷰는 서로 보완해야 한다#

지식 그래프를 화면에 넣는다고 해서 모든 사용자가 그래프부터 봐야 하는 것은 아니다.

오히려 대부분의 사용자는 답변과 숫자를 먼저 본다. 그래프는 그 다음이다.

그래서 DataNexus 화면에서는 데이터 뷰와 그래프 뷰를 따로 열 수 있게 하되, 서로 이어지게 만들기로 했다.

데이터 뷰는 정확한 값을 보여준다.

2024년 총 주문 건수는 42건입니다.

SQL 뷰는 계산 방법을 남긴다.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

그래프 뷰는 의미 관계를 확인하게 한다.

주문(Order)
 ├─ has property: 주문일자(order_date)
 ├─ has property: 주문상태(order_status)
 └─ counted as: 주문 건수(Order Count)

카탈로그 뷰는 조직 기준을 확인하게 한다.

Glossary Term: 주문 건수
Definition: 기준 기간 내 생성된 주문 레코드 수
Owner: Sales Data Domain

이 네 가지를 한 화면에 한꺼번에 다 펼치면 복잡하다. 하지만 필요할 때 열 수 있어야 한다.

처음에는 답변만 보여준다. 검토가 필요하면 SQL을 연다. 업무 정의가 궁금하면 카탈로그를 연다. 관계와 영향 범위가 궁금하면 그래프를 연다.

이 순서가 자연스럽다.

대화형 편집은 바로 반영보다 검토 큐가 먼저다#

Qurify의 지식 그래프 접근에서 또 하나 참고할 부분은 대화형 편집이다.

사용자가 “이 노드의 유형을 전략 고객으로 바꿔줘” 또는 “이 관계에 긴급 키워드를 추가해줘"처럼 요청하면 지식 체계를 계속 다듬을 수 있다.

이 방향 자체는 맞다.

하지만 엔터프라이즈 환경에서는 바로 반영하면 위험하다. 용어 하나가 바뀌면 SQL 생성, 대시보드, 리포트, 권한 정책까지 영향을 받는다.

그래서 DataNexus에서는 대화형 편집을 곧바로 운영 지식에 반영하기보다는 다음 흐름이 맞다고 봤다.

사용자 수정 요청
  → 변경 후보 생성
  → 영향 범위 분석
  → 데이터 오너 검토
  → 승인 후 Glossary / Ontology 반영
  → 관련 인덱스 재동기화

사용자가 AI에게 지식 수정을 요청하는 기능은 필요하다. 다만 운영 지식 베이스는 개인 메모장이 아니다. 조직의 기준이므로 승인과 이력이 남아야 한다.

그래서 지식 그래프 UI는 신뢰를 만드는 화면 설계의 일부가 된다.

그래프는 관계를 보여주는 데서 끝나지 않고, 변경 영향 범위를 확인하는 안전장치가 된다.

정확도와 화면에서의 검토 가능성은 따로 본다#

NL2SQL 평가에서는 EX(Execution Accuracy)를 본다. 생성된 SQL을 실행했을 때 결과가 정답과 일치하는지 확인한다.

하지만 EX가 통과해도 화면에서 믿고 쓰기 어려울 때가 있다.

SQL은 맞는데 문장이 너무 딱딱하다.

숫자도 맞다. 그런데 어떤 기준으로 낸 값인지 화면에서 안 보이면 사용자는 다음 판단을 못 한다.

출처가 있어도 찾기 어려우면 다시 물어봐야 한다.

그래서 DataNexus에서는 정확도와 화면에서의 검토 가능성을 따로 본다.

나는 이 기준을 임시로 Taste Gate라고 불렀다. 정확한 답인지와 별개로, 사용자가 그 답을 믿고 검토할 수 있는지를 보는 기준이다.

지표보는 것
EXSQL 실행 결과가 정답과 같은가
CQ온톨로지가 Competency Question을 통과하는가
Schema ComplianceSQL이 허용된 스키마와 정책 안에 있는가
Hallucination Rate없는 컬럼·용어·출처를 만들지 않는가
Query Router Accuracy질문이 SQL/Graph/Glossary/Vector 중 맞는 흐름으로 갔는가
Taste Gate사용자가 답을 검토하고 다음 판단으로 넘어갈 수 있는가

EX는 맞았는데 사용자가 “그래서 이걸 믿어도 돼?“라고 묻는다면 좋은 제품 경험이라고 보기 어렵다.

반대로 SQL이 실패했더라도 fallback임을 명확히 보여주고 검토할 여지를 남겨두면 사용자는 시스템 상태를 파악한다.

그래서 EX가 통과해도 Taste Gate에서 한 번 더 점검한다.

이 원칙을 화면으로 옮기면#

화면 구조는 이렇게 정리했다.

먼저 답변 카드부터 고쳤다. 상단은 신뢰 상태와 출처, 중간은 비즈니스 요약과 차트다. SQL과 Trace는 하단 아코디언으로 여닫는다.

+-----------------------------------------------------------+
| [Status: PASS / REVIEW / FALLBACK] | [Source: Data Catalog] |
+-----------------------------------------------------------+
| "이번 분기 순매출은 1,200억 원입니다."                    |
| [Chart: Bar / Line 자동 렌더링]                            |
+-----------------------------------------------------------+
| ▶ Show Evidence: SQL & Trace                              |
+-----------------------------------------------------------+
|  [Tabs] 1. SQL 구문  2. 데이터 계보  3. 연산 로직  4. 그래프 |
+-----------------------------------------------------------+
| [👍 / 👎]       [수정 요청하기]  [쿼리 내보내기]            |
+-----------------------------------------------------------+

임원은 위 두 블록만 확인해도 된다. 분석가는 아코디언을 열어 SQL을 본다. DBA는 데이터 계보 탭에서 거쳐간 테이블을 확인한다. 데이터 오너는 그래프 탭에서 용어와 지표 관계를 본다.

하단 피드백 버튼은 Vanna의 Self-Learning 루프로 연결되는 지점이다. 수정 피드백을 학습 데이터 후보로 남긴다.

처음에는 요약만 보여준다. 더 알고 싶을 때 SQL과 데이터 계보, 지식 그래프를 펼친다. 한꺼번에 다 쏟아붓지 않는다.

출처가 없는 답변은 화면에 내지 않는다. 최소한 SQL, 문서 ID, 용어집 항목, 그래프 노드 중 하나는 붙어야 한다.

상태 라벨은 처음에 세 가지로 나눴다. PASS/REVIEW/FALLBACK. 사용자는 라벨만 보고도 다음 판단으로 넘어가야 한다.

결과 화면에서 보여줘야 할 것#

정리하면 이렇다.

  • 답변 텍스트는 짧을수록 좋다. 보고서 문장보다 옆자리 분석가가 짧게 설명하듯이 보여준다. AI가 결론만 내놓기보다 근거를 확인할 수 있는 단계를 남겨야 한다.

  • SQL은 접어둘 수는 있어도 없애면 안 된다. 단, 카탈로그 용어 정의처럼 SQL이 없는 답변인 경우 억지로 만들지는 않는다.

  • Trace와 상태 라벨은 세트다. DB 조회 결과인지, 카탈로그에서 가져온 건지, fallback인지 구분해야 한다. 상태는 PASS, REVIEW, FALLBACK 세 가지면 충분하다.

  • 출처는 이 답이 어디에 근거했는지를 알려준다. 현재는 DW/DM 테이블과 Data Catalog Glossary Term을 출처로 쓴다. 이후에는 문서와 데이터 계보, 지식 그래프 노드까지 추가할 계획이다.

  • 지식 그래프는 장식용 시각화가 아니라 답변 근거를 탐색하는 경로다. Concept, Relationship, Property, Rule, Metric이 실제 데이터와 어떻게 연결되는지 확인할 수 있어야 한다.

  • 대화형 편집은 유용하지만 바로 운영 지식에 반영하면 안 된다. 변경 후보, 영향 범위, 데이터 오너 승인, 재동기화 흐름이 필요하다.

답변 근거를 함께 보여주지 않으면 결국 그냥 그럴듯한 챗봇이 된다.

DataNexus Chat UI에서 풀고 싶은 문제는 “AI가 답을 잘 쓰게 만드는 것"이 아니다.

사용자가 그 답을 검토하고 의심하고 필요하면 다시 추적하게 만드는 것이다.
