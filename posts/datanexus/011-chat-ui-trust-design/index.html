11. 틀릴 수 있는 AI 답변, 화면에서는 어떻게 보여줘야 할까

DataNexus Chat UI에서 답변 근거를 보여주는 방법. SQL, Runtime Trace, fallback 여부, 권한 범위, 출처와 지식 그래프까지 답변 근거로 함께 보여주기로 했다
5월 23, 2026 · 13 분 · Junho Lee | 번역:
  • En
목차
  • 먼저 타사 제품을 살펴봤다
  • 지식 그래프는 보기 좋은 그림이 아니라 근거 구조에 가깝다
  • 그래프에도 근거가 붙어야 한다
  • 결과 숫자만 봐서는 믿을지 정하기 어렵다
  • SQL 쿼리도 숨기지 않는다
  • Trace는 답이 만들어진 경로를 보여준다
  • 권한 범위도 근거의 일부다
  • Fallback 여부를 숨기지 않는다
  • 출처가 있어야 업무에 활용된다
  • 데이터 뷰와 그래프 뷰는 서로 보완해야 한다
  • 대화형 편집은 바로 반영하기보다 검토가 우선이다
  • 정확도와 검토 가능성은 따로 본다
  • 이 원칙을 화면으로 옮기면

10편까지는 SQL 생성이 어디에서 자주 틀리는지 확인했다.

BIRD와 Spider를 돌리고 모델도 바꿔봤다. 오답은 COLUMN_BINDING, VALUE_BINDING, DERIVED_METRIC 같은 유형으로 나눴다. 그렇게 SQL 생성 오류를 보던 중 다른 질문이 생겼다.

그럼 생성된 SQL과 답변 근거는 화면에서 어떻게 보여줘야 할까.

정확도 중심으로 테스트하던 동안 놓친 게 있었다. 데이터 분석 AI를 업무에 적용하려면 사용자가 근거를 확인해야 한다. 숫자가 맞아도 어디서 나온 값인지 알 수 없으면 확신할 수 없다. 그러면 결국 다시 쿼리를 돌려보게 된다.

DataNexus Chat UI에서는 답변 문장뿐만 아니라 검토 가능한 근거도 중요하다. 숫자와 문장 뒤에는 근거가 화면에서 함께 제공되어야 한다.

먼저 타사 제품을 살펴봤다#

2026년 5월 기준으로 국내외 제품을 다시 봤다.

해외 제품

  • Databricks Genie
  • Snowflake Cortex Analyst
  • Strategy(MSTR) AUTO
  • ThoughtSpot Spotter 3
  • Vanna.ai

국내 제품

  • Qurify(SmartMind)
  • MISO(GS Neotek & 52g)

살펴볼 때 기준은 하나였다.

사용자는 무엇을 봐야 AI 답변을 믿을 수 있을까.

제품마다 표현은 달랐지만 단순히 답변만 내놓지는 않았다.

  • Databricks Genie 는 Trusted assets 로 검증된 SQL query나 SQL function이 사용된 답변에 Trusted 라벨을 붙인다. Genie Space 사용자는 조직 데이터에 자연어로 질문한다. 공간의 Data 탭과 Instructions 탭에서 테이블과 컬럼, instruction, SQL query, function 같은 컨텍스트도 확인한다. 또 Unity Catalog 권한이 적용되어 사용자가 볼 수 있는 데이터만 응답에 반영된다.

  • Snowflake Cortex Analyst 는 UI보다 API 구조가 더 눈에 들어왔다. REST API 는 응답 content를 text, suggestion, sql로 구분한다. SQL content에는 verified_query_used 같은 필드가 붙는다. 응답에는 warnings와 response_metadata.question_category도 남는다. SQL을 만들기 어려운 질문에는 SQL 대신 suggestion을 반환한다.

  • Strategy(MSTR) AUTO 는 Auto Dashboards와 Strategy AI 쪽을 봤다. 공개 문서 기준으로는 derived metric, selector, auto narratives, auto answers 같은 기능이 강화되고 있다. 데모에서 본 Interpretation 영역도 참고가 됐다. 사용자가 던진 질문을 어떤 Metric과 Dimension으로 해석했는지 남기는 방식은 DataNexus의 Trace UI와 맞닿아 있다.

  • ThoughtSpot Spotter 3 는 이 글의 주제와 더 직접적으로 맞닿아 있었다. Spotter 3 는 구조화된 warehouse 데이터와 Slack, Salesforce, Jira, SharePoint 같은 비정형 업무 컨텍스트를 함께 다루겠다고 설명한다. MCP host, 쉽게 말해 외부 업무 도구와 연결되는 실행 주체로 컨텍스트를 끌어온다. 답을 낸 뒤 다시 점검하고 follow-up 분석을 이어가는 흐름도 강조한다.

  • Vanna.ai 는 학습 루프와 권한 처리 측면을 봤다. Vanna 2.0 문서 는 LLM에 SQL 실행과 chart 생성 같은 tool을 연결한다. Tool Memory로 성공한 상호작용을 저장하고 사용자별 권한을 전체 흐름에 반영하는 구조도 설명한다. 공개 README 기준으로도 streaming progress와 SQL code block, table, chart, natural language summary를 함께 제공한다.

국내 제품은 접근 방식이 조금 달랐다. 공개 소개자료와 릴리스 노트에서 확인 가능한 범위 안에서 봤다.

  • Qurify 는 자연어로 DB와 대화하는 Text-to-SQL 제품에 가깝다. 0.4.4 릴리스 노트 기준으로 생성된 SQL을 직접 편집하고 실행하거나 복사한다. 실행 결과를 인용해 후속 질문도 이어간다. 결과 CSV 다운로드와 tool call 인자 표시도 제공한다. DataNexus 입장에서는 SQL을 숨기지 않는 흐름과 후속 질문 처리 방식이 참고됐다.

  • MISO 는 NL2SQL 전용 제품이라기보다 업무 자동화형 AI 앱 빌더에 가깝다. 공개 사이트 기준으로 온프레미스 엔터프라이즈 AI 플랫폼과 Workflow App, Agent App 사례를 강조한다. 반복 보고서 자동화와 문서 검색 사례는 DataNexus가 운영형 AI 화면을 설계할 때 참고할 만하다.

이 제품들을 보면서 DataNexus 화면에 꼭 넣어야 할 요소가 보였다. 답변을 예쁘게 쓰는 것보다 SQL과 출처, 검증 상태, 권한 범위, fallback 여부를 남기는 일이 먼저였다.

흔히 semantic layer라고 부르는 층도 여기서 다시 보게 됐다. DataNexus의 Glossary와 Ontology, Knowledge Graph는 semantic layer를 업무 지식 쪽으로 더 밀어붙인 형태에 가깝다.

그리고 한 가지가 더 필요했다. 관계를 직접 확인하는 화면이다.

지식 그래프는 보기 좋은 그림이 아니라 근거 구조에 가깝다#

처음에는 지식 그래프를 별도 분석 화면 정도로 생각했다.

노드와 엣지를 확대하고 이동하면서 관계를 탐색하는 화면을 떠올렸다. 기술 데모로는 멋있다. 하지만 실제 업무에서는 그래프가 예쁘게 보이는 것만으로는 부족하다.

그래프가 제 역할을 하려면 이런 질문에 답해야 한다.

  • 이 답변은 어떤 개념을 참조했나?
  • 그 개념은 어떤 데이터와 연결되어 있나?
  • 이 관계는 누가 정의했고 근거는 어디에 있나?
  • 계산식이나 지표 정의가 바뀌면 어떤 답변이 영향을 받나?

전통적인 검색은 키워드를 잘 찾는다. 문서 안에 “순매출"이라는 단어가 있으면 해당 문서를 찾아준다. 하지만 “순매출이 총매출, 반품, 할인, 에누리와 어떤 관계인가"까지 알기는 어렵다.

그래서 온톨로지와 지식 그래프가 필요하다.

온톨로지는 조직 안에서 쓰는 개념과 관계를 정리한 설계도에 가깝다.

예를 들면 이런 식이다.

  • Concept: 고객, 제품, 주문, 순매출
  • Relationship: 고객이 주문을 생성한다, 순매출은 총매출에서 계산된다
  • Property: 주문일시, 결제수단, 주문상태
  • Rule: 환불은 결제 완료 이후에만 발생한다
  • Metric: 평균 주문 금액, 순매출, 전년 동기 대비 성장률

지식 그래프는 이 설계도를 실제 데이터와 연결한 구조다. 개념은 노드가 되고 관계는 엣지가 된다. 문서와 테이블, 컬럼과 지표 정의, 계산식이 하나의 연결망으로 묶인다.

오해하면 안 되는 게, 기존 RDBMS를 지식 그래프로 대체하자는 얘기가 아니다.

RDBMS는 값을 저장하고 트랜잭션을 처리하는 데 강하다. 지식 그래프는 값의 의미와 업무 관계를 설명하는 데 강하다. 둘은 서로 다른 역할을 맡는다.

DataNexus에서는 이 구분이 중요했다.

SQL은 값을 계산한다. 지식 그래프는 그 값이 어떤 업무 개념에서 나온 것인지 설명한다. 둘 중 하나만으로는 부족했다.

그래프에도 근거가 붙어야 한다#

지식 그래프 기능에서는 개념과 근거를 함께 보여줘야 한다.

사용자가 노드나 관계를 클릭했을 때 그 관계가 어디서 나온 것인지 바로 봐야 한다. 예를 들어 “순매출” 노드를 클릭하면 단순히 연결된 노드만 보여주는 게 아니라, 그 정의가 나온 용어집 항목이나 문서 문장, 관련 컬럼을 함께 확인해야 한다.

그래프가 이렇게 쓰이면 단순한 관계망이 아니라 답변 근거를 따라가는 UI가 된다.

예를 들어 “순매출이 뭐야?“라고 물었을 때 답변이 이렇게 끝나면 부족하다.

순매출은 총매출에서 반품, 할인, 에누리를 차감한 금액입니다.

이 문장이 맞더라도 사용자는 다시 묻게 된다.

  • 이 정의는 어디서 왔지?
  • 우리 회사 기준이 맞나?
  • 어떤 컬럼과 연결되어 있나?
  • 이 정의가 바뀌면 어떤 대시보드가 영향을 받나?

그래서 답변 옆에는 최소한 이런 정보가 있어야 한다.

source: Data Catalog Glossary
term: 순매출
source_id: urn:li:glossaryTerm:net_sales
related concepts: 총매출, 반품, 할인, 에누리
linked columns: orders.amount, orders.refund_amount, orders.discount_amount
validation: PASS

여기서 지식 그래프는 답변의 출처를 따라가는 입구에 가깝다.

결과 숫자만 봐서는 믿을지 정하기 어렵다#

타사 제품을 보면서 결과 화면에서 근거를 같이 제공해야 한다는 생각이 더 분명해졌다.

예를 들어 사용자가 이렇게 물었다고 하자.

2024년 총 주문 건수는?

AI가 이렇게만 답한다면,

2024년 총 주문 건수는 42건입니다.

겉으로는 별문제 없어 보인다. 하지만 바로 이런 질문이 남는다.

  • 어떤 테이블을 조회했지?
  • 2024년 조건은 어떻게 걸었지?
  • 취소 주문은 포함했나?
  • 이 숫자는 샘플 DB에서 나온 건가, 운영 DB에서 나온 건가?
  • SQL이 실행된 건가, 아니면 모델이 문장으로 만든 건가?

“42건"이라는 숫자 하나로는 판단이 어렵다.

Genie가 Trusted 라벨을 달고 Snowflake가 응답 JSON에 검증 필드를 남기는 목적도 같다. 화면에 근거가 없으면 사용자는 답을 그냥 믿거나 아예 무시하는 양극단으로 갈린다. 업무에 적용하려면 둘 다 좋지 않은 결과다.

그래서 DataNexus의 Chat UI에서는 답변 결과만 보여주지 않고 SQL 쿼리와 실행 순서를 함께 보여준다. 출처와 fallback 여부, 검증 상태, 권한 범위도 같은 자리에서 보이게 한다.

SQL 쿼리도 숨기지 않는다#

NL2SQL 제품을 만들다 보면 SQL을 감추고 싶다.

현업 사용자는 보통 SQL을 직접 확인하지 않는다. 화면도 복잡해진다. AI가 대신 분석해 준다는 제품 경험도 약해진다.

하지만 엔터프라이즈 기업 데이터 환경에서는 SQL을 완전히 숨기면 안 된다.

DataNexus에서는 SQL을 개발자만 보는 내부 로그로 두지 않는다. 답변 안에서 접거나 펼칠 수 있게 보여준다.

답변 영역에는 먼저 사람이 읽을 수 있는 답변이 나온다.

2024년 총 주문 건수는 42건입니다.

그 아래에 SQL을 접거나 펼칠 수 있게 둔다.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

사용자는 결과를 먼저 보고 싶어 하므로 SQL이 답변보다 먼저 보이면 안 된다. 그렇다고 SQL이 없어져서도 안 된다.

SQL은 “이 숫자는 어디서 왔는가"를 따질 때 가장 먼저 잡히는 근거다. 분석가나 DBA가 필요할 때 열어본다. 오류가 발생했을 때도 같은 쿼리로 재현해야 한다.

Trace는 답이 만들어진 경로를 보여준다#

AI 답변은 하나의 모델 호출로 끝나지 않는다.

DataNexus에는 온톨로지와 카탈로그가 있다. DW/DM, NL2SQL, 검증 로직도 함께 작동한다. 답변은 자연스럽게 여러 단계를 거친다.

Snowflake가 응답 JSON에 verified_query_used, question_category, warnings 같은 필드를 담는 것도 그 과정을 밖에서도 추적하려는 것이다.

사용자 질문이 들어오면 먼저 질문 의도를 분류한다. 처리 순서는 질문 유형에 따라 달라진다. 지표 정의를 물으면 카탈로그를 먼저 보고 실제 숫자 계산이면 DW/DM DB로 간다.

“순매출이 뭐야?“는 SQL 실행보다 카탈로그 용어집 조회가 먼저다.

“2024년 총 주문 건수는?“은 DW/DM DB에서 SELECT를 실행한다.

화면이 이 차이를 바로 보여줘야 한다.

모든 내부 로그를 다 보여줄 필요는 없다. 사용자에게는 source, validation, fallback 세 가지부터 보여주면 된다. 다만 운영자와 분석가가 펼쳐볼 상세 Trace에는 더 많은 정보가 있어야 한다.

예를 들어 DB 조회 답변이라면 이렇게 남긴다.

Runtime Trace
trace_id: dnq_20260523_001
route: NL2SQL
source: DW/DM
source_id: mart.orders
semantic_context: Data Catalog Glossary + Ontology
verified_query: none
sql_execution: success
validation:
  schema_compliance: pass
  policy_check: pass
  source_binding: pass
permission_scope:
  user: sales_analyst
  rls_policy: sales_region_allowed
  masked_columns: []
fallback: false
warnings: []

카탈로그에서 온 답이라면 이렇게 보인다.

Runtime Trace
trace_id: dnq_20260523_002
route: Glossary
source: Data Catalog Glossary
source_id: urn:li:glossaryTerm:net_sales
sql_execution: not_applicable
validation:
  source_binding: pass
  policy_check: pass
permission_scope:
  glossary_visibility: internal
fallback: false
warnings: []

사용자 화면에는 이 중 일부만 보여줘도 된다.

source: DW/DM
validation: PASS
fallback: false

Strategy(MSTR) AUTO가 어떤 Metric과 Dimension을 참조했는지 Interpretation 영역에 남기는 것도 같은 문제를 다룬다. 엔터프라이즈 환경에서는 “AI가 어떤 기준으로 이 답을 냈는가"가 불투명하면 믿고 사용하기 어렵다.

Trace가 없으면 사용자는 답의 출처를 추측해야 한다.

권한 범위도 근거의 일부다#

출처만 보여준다고 충분하지는 않다.

사용자가 볼 수 없는 테이블이나 문서가 근거로 쓰이면 안 된다. 같은 질문이라도 영업팀, 재무팀, 임원 계정의 권한 범위가 다르면 답변과 근거가 달라질 수 있다.

그래서 Trace에는 source뿐 아니라 permission scope도 남겨야 한다.

permission_scope:
  user: sales_analyst
  allowed_domains: [Sales]
  row_level_filter: region IN ('KR-Seoul', 'KR-Gyeonggi')
  masked_columns: [customer_phone, customer_email]

이 정보는 모든 사용자에게 그대로 노출할 필요는 없다. 하지만 운영자와 감사 담당자는 확인해야 한다.

신뢰 가능한 답변은 “어느 데이터에서 왔는가"뿐 아니라 “이 사용자가 볼 수 있는 범위 안에서 나온 답인가"까지 설명해야 한다.

Fallback 여부를 숨기지 않는다#

실패했는데 성공처럼 보이는 건 더 위험하다.

DB 연결이 끊기거나 SQL 실행이 실패했는데 화면에는 정답처럼 보이는 문장이 나온다. 이러면 사용자는 틀린 답을 정답으로 믿는다.

Snowflake는 AI가 확신하지 못하는 질문에 SQL을 억지로 만들지 않고 suggestion 콘텐츠를 반환한다. Qurify는 생성된 SQL을 사용자가 직접 편집하고 실행하게 한다. 두 방향 모두 실패나 불확실성을 화면 밖으로 숨기지 않는다는 점에서 참고할 만하다.

DB 조회가 실패하면 fallback을 보여줘야 한다. DataNexus에서는 이 fallback도 숨기지 않는다.

실패했을 때도 답변을 보여줘야 할 때가 있다. 데모 환경에서는 미리 준비한 답을 띄워야 할 때도 있다. 그럴수록 실제 DB 결과인지 fallback인지 분명히 표시해야 한다.

Runtime Trace
source: fallback demo response
sql_execution: failed
validation: FALLBACK
fallback: true
warnings:
  - DB connection failed. Demo response was used.

데모 환경인데 fallback 표시가 없으면 실제 DB 결과로 오해하기 쉽다.

사용자는 시스템이 무엇을 했고 무엇을 하지 못했는지 알아야 한다. 운영자와 개발자는 어디서 실패했는지, fallback이 얼마나 자주 발생하는지 추적해야 한다.

출처가 있어야 업무에 활용된다#

DataNexus에서 가장 자주 보여줘야 할 출처는 데이터 카탈로그다.

예를 들어 “순매출"이라는 용어가 있다. 회사마다 정의가 다르다. 반품과 할인만 빼는 곳이 있고 에누리나 세금 처리까지 포함하는 곳도 있다.

DDL에는 이런 정의가 없다.

테이블에는 amount, discount_amount, refund_amount 같은 컬럼이 있을 뿐이다. 순매출이라는 업무 개념은 데이터베이스 바깥에 있다.

그래서 용어 정의는 카탈로그에서 가져와야 한다.

Chat UI에서 “순매출이 뭐야?“라고 물었을 때 답변이 카탈로그 용어집에서 왔다면 source: Data Catalog Glossary로 남겨야 한다.

source: Data Catalog Glossary
term: 순매출
source_id: urn:li:glossaryTerm:net_sales
owner: Sales Data Domain

사용자는 이 답이 모델의 일반 지식이 아니라 조직이 관리하는 용어집에서 왔다는 점을 확인한다.

온톨로지 기반 NL2SQL은 모델이 알아서 맞히는 방식이 아니다. 조직이 정의한 의미를 런타임에 가져와서 답하는 방식이다.

데이터 뷰와 그래프 뷰는 서로 보완해야 한다#

지식 그래프를 화면에 넣는다고 해서 모든 사용자가 그래프부터 봐야 하는 것은 아니다.

대부분의 사용자는 답변과 숫자를 먼저 본다. 그래프는 그 다음이다.

그래서 DataNexus 화면에서는 데이터 뷰와 그래프 뷰를 따로 열 수 있게 하되 서로 이어지게 만들었다.

데이터 뷰는 정확한 값을 보여주고 SQL 뷰는 그 값이 어떻게 나왔는지 남긴다.

2024년 총 주문 건수는 42건입니다.

계산 방법은 SQL로 확인한다.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

의미 관계가 궁금하면 그래프 뷰를 연다.

주문(Order)
 ├─ has property: 주문일자(order_date)
 ├─ has property: 주문상태(order_status)
 └─ counted as: 주문 건수(Order Count)

회사 기준이 궁금하면 카탈로그 뷰를 연다.

Glossary Term: 주문 건수
Definition: 기준 기간 내 생성된 주문 레코드 수
Owner: Sales Data Domain

이 네 가지를 한 화면에 한꺼번에 다 펼치면 복잡하다. 하지만 필요할 때 열 수 있어야 한다.

처음에는 답변만 보여준다. 숫자가 이상하면 SQL을 연다. 업무 정의가 궁금하면 카탈로그로 들어간다. 관계나 영향 범위까지 봐야 할 때만 그래프를 연다.

이 순서가 필요한 이유는 단순하다. 대부분의 사용자는 먼저 결과를 확인한다. 의심이 생겼을 때 근거를 따라간다.

대화형 편집은 바로 반영하기보다 검토가 우선이다#

사용자가 “이 노드의 유형을 전략 고객으로 바꿔줘” 또는 “이 관계에 긴급 키워드를 추가해줘"처럼 요청한다면 지식 체계를 계속 다듬을 수 있다.

이 방향은 맞다. 다만 엔터프라이즈 환경에서는 바로 반영하면 위험하다. 용어 하나가 바뀌면 SQL 생성부터 대시보드와 리포트, 권한 정책까지 영향을 받는다.

그래서 DataNexus에서는 대화형 편집을 곧바로 운영 지식에 반영하지 않는다. 다음 흐름으로 처리한다.

사용자 수정 요청
  → 변경 후보 생성
  → 영향 범위 분석
  → 데이터 오너 검토
  → 승인 후 Glossary / Ontology 반영
  → 관련 인덱스 재동기화

사용자가 AI에게 지식 수정을 요청하는 기능은 필요하다. 다만 운영 지식 베이스를 개인 메모장처럼 다루면 안 된다. 조직의 기준이므로 승인과 이력이 남아야 한다.

그래서 지식 그래프 UI는 신뢰를 만드는 화면 설계의 일부가 된다.

그래프는 관계를 보여주는 데서 끝나지 않는다. 변경 영향 범위를 확인하는 안전장치가 되어야 한다.

정확도와 검토 가능성은 따로 본다#

NL2SQL 평가에서는 EX(Execution Accuracy)를 본다. 생성된 SQL을 실행했을 때 결과가 정답과 일치하는지 확인한다.

하지만 EX는 오프라인 평가 지표다. 정답 SQL이나 정답 실행 결과가 있는 벤치마크에서 측정한다.

운영 런타임에서는 보통 gold answer가 없다. 사용자가 방금 던진 질문에는 정답 SQL이 미리 준비되어 있지 않기 때문이다.

따라서 운영 화면에서 PASS를 EX 통과로 정의하면 안 된다. PASS는 runtime validation 기준으로 판단해야 한다.

PASS
- SQL 실행 또는 카탈로그 조회 성공
- Schema Compliance 통과
- 권한/RLS 정책 통과
- 출처 또는 verified query / glossary / graph node 매칭
- 치명적 warning 없음

REVIEW
- SQL은 실행됐지만 binding이 불확실함
- verified query 미사용
- 출처 매칭이 약함
- 질문이 모호하거나 router가 우회함
- 데이터 오너 또는 분석가 검토 필요

FALLBACK
- SQL 실행 실패
- DB 미조회
- suggestion only
- demo response
- 실제 운영 데이터가 아닌 대체 응답

정확도와 검토 가능성도 따로 봐야 한다. SQL 결과가 맞아도 화면에서 믿고 쓰기 어려울 때가 있다.

SQL은 맞는데 문장이 너무 딱딱하다.

숫자도 맞다. 그런데 어떤 기준으로 낸 값인지 화면에서 안 보이면 사용자는 다음 판단을 못 한다.

출처가 있어도 찾기 어려우면 다시 물어봐야 한다.

TrustSQL 은 Text-to-SQL reliability를 다룰 때 가능한 질문에는 올바른 SQL을 만들어야 한다고 본다. 불가능한 질문에는 답변을 보류하는 능력도 중요하게 본다. LinkedIn의 enterprise Text-to-SQL 사례 도 schema와 문서, query log, wiki, code, domain knowledge를 지식 그래프로 구성한다. chatbot UI에서 다양한 사용자 의도와 follow-up을 지원하는 방향도 보여준다.

이 기준을 DataNexus에서는 Taste Gate라고 부른다. EX가 통과해도 근거가 없거나 출처가 불분명하면 Taste Gate에서는 실패다.

지표보는 것
EX오프라인 벤치마크에서 SQL 실행 결과가 정답과 같은가
CQ온톨로지가 Competency Question을 통과하는가
Schema ComplianceSQL이 허용된 스키마와 정책 안에 있는가
Hallucination Rate없는 컬럼, 용어, 출처를 만들지 않는가
Query Router Accuracy질문이 SQL, Graph, Glossary, Vector 중 맞는 흐름으로 갔는가
Runtime Validation운영 시점에 실행, 정책, 출처, warning 검증을 통과했는가
Taste Gate사용자가 답을 검토하고 다음 판단으로 넘어갈 수 있는가

EX는 맞았는데 사용자가 “그래서 이걸 믿어도 돼?“라고 묻는다면 제품 경험으로는 실패다.

반대로 SQL이 실패했더라도 fallback임을 명확히 보여주고 검토할 여지를 남겨두면 사용자는 시스템 상태를 파악한다.

그래서 EX가 통과해도 Taste Gate에서 한 번 더 점검한다.

이 원칙을 화면으로 옮기면#

화면 구조는 이렇게 정리했다.

먼저 답변 카드부터 고쳤다. 상단은 신뢰 상태와 출처, 중간은 비즈니스 요약과 차트다. SQL과 Trace는 하단 아코디언으로 여닫는다.

+---------------------------------------------------------------+
| [Status: PASS / REVIEW / FALLBACK] | [Source: Data Catalog]   |
+---------------------------------------------------------------+
| "이번 분기 순매출은 1,200억 원입니다."                        |
| [Chart: Bar / Line 자동 렌더링]                                |
+---------------------------------------------------------------+
| ▶ Show Evidence: SQL & Trace                                  |
+---------------------------------------------------------------+
| [Tabs] 1. SQL  2. Data Lineage  3. Metric Logic  4. Graph      |
+---------------------------------------------------------------+
| [👍 / 👎]       [수정 요청하기]  [쿼리 내보내기]                |
+---------------------------------------------------------------+

임원은 위 두 블록만 확인해도 된다. 분석가는 아코디언을 열어 SQL을 본다. DBA는 데이터 계보 탭에서 거쳐간 테이블을 확인한다. 데이터 오너는 그래프 탭에서 용어와 지표 관계를 본다.

하단 피드백 버튼은 Vanna의 Self-Learning 루프와 DataNexus의 검토 큐로 연결되는 지점이다. 수정 피드백을 학습 데이터 후보로 남긴다.

처음에는 요약만 보여준다. 더 알고 싶을 때 SQL과 데이터 계보, 지식 그래프를 펼친다. 한꺼번에 다 쏟아붓지 않는다.

출처가 없는 답변은 화면에 내지 않는다. 최소한 SQL, 문서 ID, 용어집 항목, 그래프 노드 중 하나는 붙어야 한다.

상태 라벨은 세 가지로 나눈다. PASS, REVIEW, FALLBACK.

사용자는 라벨만 보고도 다음 행동을 정해야 한다.

PASS: 지금 업무 판단에 사용 가능
REVIEW: 결과를 열어보고 한 번 확인해야 함
FALLBACK: 실제 DB 결과가 아니거나 정상 실행 결과가 아님

여기서 PASS는 EX 통과가 아니다.

PASS는 실행 성공, 정책 검증, 출처 매칭, 치명적 warning 없음으로 판단한다.

모델 내부의 confidence score를 그대로 노출하는 방법도 있다. 하지만 보정되지 않은 수치는 사용자에게 해석 부담을 준다. 라벨 세 개로 “지금 바로 써도 된다”, “한 번 확인해야 한다”, “실제 결과가 아니다"를 구분하는 편이 낫다.

답변 텍스트는 짧을수록 좋다. 보고서 문장보다 옆자리 분석가가 짧게 설명하듯이 보여준다. SQL은 접어둘 수는 있어도 없애면 안 된다. Trace와 상태 라벨은 DB 조회 결과인지, 카탈로그에서 가져온 건지, fallback인지 구분하는 최소 장치다.

답변 근거를 함께 보여주지 않으면 그냥 그럴듯한 챗봇이 된다.

DataNexus Chat UI에서 풀고 싶은 문제는 “AI가 답을 잘 쓰게 만드는 것"보다 답이 어디서 왔는지 사용자가 직접 따져보게 하는 데 있다.
