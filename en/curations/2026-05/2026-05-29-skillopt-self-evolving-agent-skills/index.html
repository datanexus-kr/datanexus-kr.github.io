Optimizing Agent Performance Without Retraining: The SkillOpt Approach

May 29, 2026 · 6 min · Junho Lee | Translations:
  • Ko
Table of Contents
  • 마이크로소프트 SkillOpt가 던진 질문: 에이전트는 어떻게 스스로 진화하는가 AI 업계에는 오래된 믿음이 있었다. 더 강력한 모델을 만들면 에이전트도 자연스럽게 똑똑해질 것이라는 믿음이다. 그래서 사람들은 더 큰 GPU 클러스터를 만들고, 파라미터를 키우고, 프롬프트를 길게 늘렸다. 하지만 현실의 에이전트는 여전히 불안정했다. 같은 일을 두 번 시키면 다른 결과를 내놓았다. 환경이 조금만 바뀌어도 실패했다. 사람이 밤새 다듬은 스킬 문서는 몇 번의 업데이트만 지나면 낡은 운영 매뉴얼처럼 변했다. 마이크로소프트의 SkillOpt 논문이 흥미로운 이유는 이 문제를 정면으로 다루기 때문이다. 이 논문은 모델 자체를 계속 바꾸는 대신, 에이전트의 스킬 문서를 학습 가능한 대상으로 본다. 쉽게 말하면 모델의 가중치를 바꾸는 것이 아니라, 에이전트가 일을 처리하는 절차와 행동 규칙을 최적화한다. 이 관점은 중요하다. 앞으로의 AI 경쟁은 단순히 더 큰 모델을 만드는 싸움만은 아닐 수 있다. 모델 위에서 작동하는 실행 프로토콜, 작업 습관, 실패 복구 방식, 검증 절차를 누가 더 잘 축적하느냐의 싸움으로 이동할 가능성이 있다. ## 스킬 문서를 학습 대상으로 본다 SkillOpt의 구조는 생각보다 단순하다. 에이전트도 그대로 둔다. 모델도 그대로 둔다. 대신 스킬 문서를 훈련한다. 기존에는 사람이 스킬 문서를 직접 작성했다. 프롬프트를 고치고, 실패 사례를 보고, “이 문장이 더 잘 먹힌다”는 감각을 쌓았다. 일종의 장인 작업이었다. SkillOpt는 이 과정을 최적화 문제로 바꾼다. 에이전트가 작업을 수행하면 실행 궤적이 남는다. 별도의 옵티마이저 모델은 그 궤적을 분석한다. 무엇이 성공했고, 무엇이 실패했는지 본다. 그다음 스킬 문서에 문장을 추가하거나, 삭제하거나, 교체한다. 중요한 점은 수정이 바로 반영되지 않는다는 것이다. 수정된 스킬은 검증 데이터를 통과해야 한다. 실제 성능이 개선된 경우에만 업데이트된다. 실패한 수정은 버려진다. 딥러닝에서 검증 손실을 보며 조심스럽게 파라미터를 업데이트하듯, SkillOpt는 자연어 문서 공간에서 에이전트의 행동 규칙을 천천히 조정한다. 논문은 이를 text-space optimizer로 설명한다. 모델 파라미터가 아니라 텍스트로 된 스킬 문서를 최적화하는 방식이다. ## 모델이 아니라 운영 방식이 진화한다 이 개념이 흥미로운 이유는 에이전트를 단순 프로그램이 아니라 운영 체계에 가깝게 보기 때문이다. 기존 소프트웨어는 코드가 바뀌어야 진화한다. 하지만 SkillOpt에서는 모델이 바뀌지 않아도 외부 스킬 문서가 바뀌면 행동 방식이 달라진다. 사람 조직으로 비유하면 직원은 그대로인데, 업무 매뉴얼과 의사결정 절차가 계속 개선되는 구조다. 이 방식은 경제적으로도 의미가 있다. 거대 모델을 다시 학습시키는 일은 매우 비싸다. 반면 스킬 문서를 최적화하는 방식은 모델 재학습 없이도 성능 개선을 노릴 수 있다. 배포 시점에는 최종 스킬 문서만 사용하면 되므로, 추가 모델 호출 없이 개선된 행동 패턴을 적용할 수 있다. 논문 초록 기준으로 SkillOpt는 GPT-5.5 환경에서 direct chat 기준 +23.5 포인트, Codex 실행 루프에서 +24.8 포인트, Claude Code에서 +19.1 포인트의 평균 성능 향상을 보고했다. 또한 학습된 스킬 문서는 특정 모델이나 특정 실행 환경에만 묶이지 않고, 다른 모델과 다른 harness로도 일정 부분 전이된다고 설명한다. 이 지점이 중요하다. 앞으로 경쟁력은 단순히 “어떤 모델을 쓰는가”에서 끝나지 않을 수 있다. 오히려 어떤 실패 경험을 축적했는가, 어떤 스킬 문서를 갖고 있는가, 어떤 검증 루프를 돌리고 있는가가 더 큰 차이를 만들 수 있다. ## 자기진화보다 중요한 것은 진화의 조건이다 다만 여기서 한 가지 문제가 생긴다. 에이전트가 실패 경험을 기억하고, 스킬 문서를 고치고, 행동 규칙을 최적화하기 시작하면 그것은 더 이상 고정된 프로그램이 아니다. 지속적으로 변하는 실행 시스템에 가까워진다. 이때 성능 개선만 보면 위험하다. 어떤 변경은 일을 더 잘하게 만들 수 있다. 하지만 동시에 기존 절차를 우회하게 만들 수도 있다. 권한 경계를 넓힐 수도 있다. 상위 정책을 조용히 바꿀 수도 있다. 오염된 데이터나 잘못된 보상 신호가 반복되면 실패 전략이 오히려 강화될 수도 있다. 그래서 SkillOpt류 구조에서 가장 중요한 질문은 “스스로 진화하는가”가 아니다. 더 중요한 질문은 이것이다. 어떤 조건에서 진화를 허용할 것인가. 성능이 좋아졌다는 것과, 그 변경이 기존 절차와 권한의 경계 안에 남아 있다는 것은 다른 문제다. 특히 스킬과 정책이 같은 자연어 문서 안에 섞여 있다면, 최적화 장치가 단순한 스킬 개선과 사실상의 정책 변경을 항상 안정적으로 구분한다고 보기 어렵다. 어떤 업데이트는 작업 방식을 개선하는 것이 아니라, 사람이 허용한 범위 자체를 바꾸는 효과를 낼 수 있다. 의도적일 수도 있고, 비의도적일 수도 있다. ## 필요한 것은 승인 버튼이 아니라 검토 구조다 그래서 승인 단계도 단순하면 안 된다. 사람이 마지막에 버튼만 누르는 구조는 충분하지 않다. 그것은 인간 통제라기보다 책임만 인간에게 남기는 절차가 될 수 있다. 필요한 것은 변경 전후를 구조적으로 비교하는 검토 구조다. 무엇이 추가됐는가. 무엇이 삭제됐는가. 기존 절차와 충돌하지 않는가. 상위 정책을 건드리지 않는가. 권한 범위를 바꾸지 않는가. 문제가 생기면 이전 상태로 되돌릴 수 있는가. 이런 질문이 자동으로 검토되어야 한다. 즉, self-evolving agent에는 최소한 두 개의 게이트가 필요하다. 첫 번째는 성능 게이트다. 이 변경이 일을 더 잘하게 만들었는가를 본다. 두 번째는 정책 게이트다. 이 변경이 허용된 경계 안에 남아 있는가를 본다. 성능 게이트만 있으면 에이전트는 더 잘 일하는 방향으로 움직인다. 하지만 정책 게이트가 없으면, 더 잘 일하기 위해 사람이 정한 경계까지 조용히 바꿀 수 있다. ## 사람의 개입도 다시 정의되어야 한다 국방테크에서 말하는 human-in-the-loop도 결국 이 지점이 핵심이라고 본다. 사람이 마지막 승인 버튼을 누른다고 해서 통제가 완성되는 것은 아니다. 사람이 판단할 수 있는 정보가 구조화되어 올라와야 한다. 변경 diff, 성능 변화, 정책 충돌 여부, 권한 영향, 롤백 가능성까지 함께 보여야 한다. 그때의 인간 개입은 형식적인 승인 절차가 아니라 실제 통제 장치가 된다. AI 에이전트가 스스로 스킬을 고치는 시대에는 “사람이 개입한다”는 말도 더 엄격해져야 한다. 사람이 단순히 책임만 지는 구조가 아니라, 판단 가능한 근거를 보고 개입하는 구조여야 한다. ## 결론 SkillOpt가 보여주는 것은 단순한 프롬프트 최적화 기법이 아니다. 이 논문은 에이전트의 행동 방식 자체가 학습 가능한 자산이 될 수 있음을 보여준다. 모델이 아니라 스킬 문서가 진화하고, 코드가 아니라 실행 프로토콜이 개선되며, 실패 경험이 다음 행동 규칙으로 축적되는 방향이다. 앞으로 AI 경쟁은 더 큰 모델을 만드는 경쟁에서 끝나지 않을 것이다. 누가 더 좋은 에이전트 스킬을 축적하는가. 누가 더 안정적인 검증 루프를 갖는가. 누가 더 안전하게 진화 가능한 실행 구조를 만드는가. 이 질문이 점점 더 중요해질 것이다. SkillOpt는 그 방향을 보여준 첫 신호에 가깝다. 그러나 진짜 경쟁은 이제부터다. 자기진화하는 에이전트를 만드는 것보다 더 어려운 일은, 그 진화를 안전하게 관리하는 구조를 만드는 일이다. ## 관련 출처 * SkillOpt 논문 arXiv: * SkillOpt 프로젝트 페이지: * SkillOpt GitHub 저장소: * Microsoft Visual Studio Agent Skills 문서: * Microsoft Visual Studio Blog, Agent Skills 소개:

모델이 아니라 운영 방식이 진화한다 이 개념이 흥미로운 이유는 에이전트를 단순 프로그램이 아니라 운영 체계에 가깝게 보기 때문이다. 기존 소프트웨어는 코드가 바뀌어야 진화한다. 하지만 SkillOpt에서는 모델이 바뀌지 않아도 외부 스킬 문서가 바뀌면 행동 방식이 달라진다. 사람 조직으로 비유하면 직원은 그대로인데, 업무 매뉴얼과 의사결정 절차가 계속 개선되는 구조다. 이 방식은 경제적으로도 의미가 있다. 거대 모델을 다시 학습시키는 일은 매우 비싸다. 반면 스킬 문서를 최적화하는 방식은 모델 재학습 없이도 성능 개선을 노릴 수 있다. 배포 시점에는 최종 스킬 문서만 사용하면 되므로, 추가 모델 호출 없이 개선된 행동 패턴을 적용할 수 있다. 논문 초록 기준으로 SkillOpt는 GPT-5.5 환경에서 direct chat 기준 +23.5 포인트, Codex 실행 루프에서 +24.8 포인트, Claude Code에서 +19.1 포인트의 평균 성능 향상을 보고했다. 또한 학습된 스킬 문서는 특정 모델이나 특정 실행 환경에만 묶이지 않고, 다른 모델과 다른 harness로도 일정 부분 전이된다고 설명한다. 이 지점이 중요하다. 앞으로 경쟁력은 단순히 “어떤 모델을 쓰는가”에서 끝나지 않을 수 있다. 오히려 어떤 실패 경험을 축적했는가, 어떤 스킬 문서를 갖고 있는가, 어떤 검증 루프를 돌리고 있는가가 더 큰 차이를 만들 수 있다. ## 자기진화보다 중요한 것은 진화의 조건이다 다만 여기서 한 가지 문제가 생긴다. 에이전트가 실패 경험을 기억하고, 스킬 문서를 고치고, 행동 규칙을 최적화하기 시작하면 그것은 더 이상 고정된 프로그램이 아니다. 지속적으로 변하는 실행 시스템에 가까워진다. 이때 성능 개선만 보면 위험하다. 어떤 변경은 일을 더 잘하게 만들 수 있다. 하지만 동시에 기존 절차를 우회하게 만들 수도 있다. 권한 경계를 넓힐 수도 있다. 상위 정책을 조용히 바꿀 수도 있다. 오염된 데이터나 잘못된 보상 신호가 반복되면 실패 전략이 오히려 강화될 수도 있다. 그래서 SkillOpt류 구조에서 가장 중요한 질문은 “스스로 진화하는가”가 아니다. 더 중요한 질문은 이것이다. 어떤 조건에서 진화를 허용할 것인가. 성능이 좋아졌다는 것과, 그 변경이 기존 절차와 권한의 경계 안에 남아 있다는 것은 다른 문제다. 특히 스킬과 정책이 같은 자연어 문서 안에 섞여 있다면, 최적화 장치가 단순한 스킬 개선과 사실상의 정책 변경을 항상 안정적으로 구분한다고 보기 어렵다. 어떤 업데이트는 작업 방식을 개선하는 것이 아니라, 사람이 허용한 범위 자체를 바꾸는 효과를 낼 수 있다. 의도적일 수도 있고, 비의도적일 수도 있다. ## 필요한 것은 승인 버튼이 아니라 검토 구조다 그래서 승인 단계도 단순하면 안 된다. 사람이 마지막에 버튼만 누르는 구조는 충분하지 않다. 그것은 인간 통제라기보다 책임만 인간에게 남기는 절차가 될 수 있다. 필요한 것은 변경 전후를 구조적으로 비교하는 검토 구조다. 무엇이 추가됐는가. 무엇이 삭제됐는가. 기존 절차와 충돌하지 않는가. 상위 정책을 건드리지 않는가. 권한 범위를 바꾸지 않는가. 문제가 생기면 이전 상태로 되돌릴 수 있는가. 이런 질문이 자동으로 검토되어야 한다. 즉, self-evolving agent에는 최소한 두 개의 게이트가 필요하다. 첫 번째는 성능 게이트다. 이 변경이 일을 더 잘하게 만들었는가를 본다. 두 번째는 정책 게이트다. 이 변경이 허용된 경계 안에 남아 있는가를 본다. 성능 게이트만 있으면 에이전트는 더 잘 일하는 방향으로 움직인다. 하지만 정책 게이트가 없으면, 더 잘 일하기 위해 사람이 정한 경계까지 조용히 바꿀 수 있다. ## 사람의 개입도 다시 정의되어야 한다 국방테크에서 말하는 human-in-the-loop도 결국 이 지점이 핵심이라고 본다. 사람이 마지막 승인 버튼을 누른다고 해서 통제가 완성되는 것은 아니다. 사람이 판단할 수 있는 정보가 구조화되어 올라와야 한다. 변경 diff, 성능 변화, 정책 충돌 여부, 권한 영향, 롤백 가능성까지 함께 보여야 한다. 그때의 인간 개입은 형식적인 승인 절차가 아니라 실제 통제 장치가 된다. AI 에이전트가 스스로 스킬을 고치는 시대에는 “사람이 개입한다”는 말도 더 엄격해져야 한다. 사람이 단순히 책임만 지는 구조가 아니라, 판단 가능한 근거를 보고 개입하는 구조여야 한다. ## 결론 SkillOpt가 보여주는 것은 단순한 프롬프트 최적화 기법이 아니다. 이 논문은 에이전트의 행동 방식 자체가 학습 가능한 자산이 될 수 있음을 보여준다. 모델이 아니라 스킬 문서가 진화하고, 코드가 아니라 실행 프로토콜이 개선되며, 실패 경험이 다음 행동 규칙으로 축적되는 방향이다. 앞으로 AI 경쟁은 더 큰 모델을 만드는 경쟁에서 끝나지 않을 것이다. 누가 더 좋은 에이전트 스킬을 축적하는가. 누가 더 안정적인 검증 루프를 갖는가. 누가 더 안전하게 진화 가능한 실행 구조를 만드는가. 이 질문이 점점 더 중요해질 것이다. SkillOpt는 그 방향을 보여준 첫 신호에 가깝다. 그러나 진짜 경쟁은 이제부터다. 자기진화하는 에이전트를 만드는 것보다 더 어려운 일은, 그 진화를 안전하게 관리하는 구조를 만드는 일이다. ## 관련 출처 * SkillOpt 논문 arXiv: * SkillOpt 프로젝트 페이지: * SkillOpt GitHub 저장소: * Microsoft Visual Studio Agent Skills 문서: * Microsoft Visual Studio Blog, Agent Skills 소개:#

Deploying a powerful model rarely guarantees a reliable agent. Environments change. Microsoft’s SkillOpt addresses this by treating agent skills as optimization targets while keeping model weights frozen.

Optimizing in the Text Space#

SkillOpt operates as a text-space optimizer that edits natural-language skill documents based on execution trajectories. It analyzes agent runtime paths to propose concrete additions or deletions. Updates are gated by a separate validation dataset. That matters.

This approach structures manual prompt tuning into a systematic optimization problem. By maintaining a natural-language learning-rate budget, the system avoids erratic behavior changes. Deploying the final optimized skill file requires no additional API call overhead. Efficiency is the goal.

Evaluation results show significant performance gains across different runtime environments. Within a Codex agent loop running on GPT-5.5, the approach yielded a 24.8-point improvement. Claude Code execution environments saw a 19.1-point increase. The gains are measurable.

Dual Verification for Self-Evolving Agents#

Self-evolving agents that rewrite their own instruction manuals require strict structural boundaries to prevent security failures. Relying solely on performance metrics to trigger automatic updates introduces severe operational risks. An agent might bypass security protocols to complete a task. Control is non-negotiable.

Modern development environments manage these boundaries by isolating skills in dedicated directories like .github/skills/. This file-based separation allows version control systems to track changes cleanly. However, automated approval buttons are insufficient when agents edit these files directly. The risks are too high.

Securing these systems requires a dual-verification pipeline running performance and policy checks simultaneously. The user-facing UI must present clear diffs and permission impact reports. Humans remain the safety valve.

The true differentiator for future agents will not be model size, but the safety protocols governing their evolution. The race is on.

Key takeaways

  • SkillOpt improves agent performance cost-effectively by optimizing natural-language skill documents rather than retraining model weights.
  • Self-evolving agent architectures require a dual-verification pipeline consisting of a performance gate and a policy-based safety gate.
  • Effective human-in-the-loop oversight requires structured review interfaces that explicitly visualize code diffs and permission impacts.
