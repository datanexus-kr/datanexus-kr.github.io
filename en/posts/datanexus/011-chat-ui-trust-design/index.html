11. AI answers can be wrong. How should they appear on screen?

How the DataNexus Chat UI shows answer evidence. SQL, Runtime Trace, fallback state, permission scope, sources, and knowledge graphs are shown together as evidence behind an answer.

Up to post 10, I had been checking where SQL generation most often goes wrong.

I ran BIRD and Spider and changed models as well. I grouped wrong answers into types like COLUMN_BINDING, VALUE_BINDING, and DERIVED_METRIC. While reviewing those SQL generation errors, another question came up.

How should generated SQL and answer evidence appear on the screen?

The accuracy tests had missed something. If a data analysis AI is going to be used at work, users need to check the evidence. Even when the number is correct, nobody can be sure if they cannot tell where it came from. In the end, they run the query again.

For the DataNexus Chat UI, the answer text was not the only thing that mattered. Reviewable evidence mattered too. The evidence behind the number and the sentence should be shown together on the screen.

I first looked at other products#

As of May 2026, I looked again at domestic and global products.

Global products

  • Databricks Genie
  • Snowflake Cortex Analyst
  • Strategy(MSTR) AUTO
  • ThoughtSpot Spotter 3
  • Vanna.ai

Domestic products

  • Qurify(SmartMind)
  • MISO(GS Neotek & 52g)

I used one lens while looking at them.

What does a user need to see before trusting an AI answer?

The products expressed it differently, but they did not simply put an answer on the screen.

  • Databricks Genie uses Trusted assets to label answers that use verified SQL queries or SQL functions as Trusted. Genie Space users can ask questions about organizational data in natural language. In the Data and Instructions tabs, they can also check context such as tables, columns, instructions, SQL queries, and functions. Unity Catalog permissions also apply, so only data the user is allowed to see is reflected in the answer.

  • Snowflake Cortex Analyst stood out more for its API structure than its UI. The REST API separates response content into text, suggestion, and sql. SQL content can include fields such as verified_query_used, and the response also keeps warnings and response_metadata.question_category. For questions where SQL is hard to generate, it can return a suggestion instead of SQL.

  • Strategy(MSTR) AUTO was useful around Auto Dashboards and Strategy AI. Based on public docs, features such as derived metrics, selectors, auto narratives, and auto answers are getting stronger. The Interpretation area I saw in demos was also relevant. Leaving how the user question was interpreted as Metrics and Dimensions is close to what DataNexus needs in its Trace UI.

  • ThoughtSpot Spotter 3 overlapped more directly with this post. Spotter 3 says it will handle structured warehouse data together with unstructured work context from Slack, Salesforce, Jira, SharePoint, and similar tools. It pulls context from external tools as an MCP host. It also emphasizes checking its own answer and continuing into follow-up analysis.

  • Vanna.ai was useful around the learning loop and permission handling. The Vanna 2.0 docs explain a structure where the LLM is connected to tools such as SQL execution and chart generation. Tool Memory stores successful interactions, and user-specific permissions are reflected throughout the flow. The public README also shows streaming progress, SQL code blocks, tables, charts, and natural-language summaries together.

The domestic products approached the problem a little differently. I treated them more cautiously and stayed close to public product material and release notes.

  • Qurify is closer to a Text-to-SQL product for talking with databases in natural language. Based on the 0.4.4 release notes, users can edit, run, and copy generated SQL. They can also continue follow-up questions by citing execution results. CSV download and tool call argument display are available too. For DataNexus, the way it does not hide SQL and supports follow-up questions was useful.

  • MISO is closer to an AI app builder for work automation than a dedicated NL2SQL product. Based on the public site, it emphasizes on-premises enterprise AI platforms and Workflow App / Agent App cases. Report automation and document search cases are still useful references when designing operational AI screens for DataNexus.

After looking at these products, the elements DataNexus needed on screen became clearer. Writing a nicer answer mattered less than keeping SQL, source, validation state, permission scope, and fallback state visible.

This also made me revisit what the BI industry usually calls the semantic layer. DataNexus Glossary, Ontology, and Knowledge Graph are closer to pushing the semantic layer further toward operational knowledge.

One more thing was needed: a screen where users can check relationships directly.

A knowledge graph is closer to evidence structure than a pretty diagram#

At first, I thought of the knowledge graph as a separate analysis screen.

I pictured a screen where users zoom and pan around nodes and edges while exploring relationships. It looks good in a technical demo. In real work, though, a graph being visually impressive is not enough.

For a graph to earn its place on the screen, it has to answer questions like this.

  • Which concept did this answer reference?
  • Which data is that concept connected to?
  • Who defined the relationship?
  • Where is the evidence for this relationship?
  • If a formula or metric definition changes, which answers are affected?

Traditional search is good at finding keywords. If a document contains the word “net revenue”, search can find that document. But search alone has a hard time understanding how net revenue relates to gross revenue, returns, discounts, and allowances.

That is why ontology and knowledge graphs matter.

An ontology is close to a blueprint that organizes the concepts and relationships used inside an organization.

For example:

  • Concept: customer, product, order, net revenue
  • Relationship: a customer creates an order, net revenue is calculated from gross revenue
  • Property: order timestamp, payment method, order status
  • Rule: a refund can happen only after payment is completed
  • Metric: average order value, net revenue, year-over-year growth rate

A knowledge graph connects that blueprint to actual data. Concepts become nodes, and relationships become edges. Documents, tables, columns, metric definitions, and formulas are tied into one network.

One thing should not be misunderstood: this is not about replacing the existing RDBMS with a knowledge graph.

An RDBMS is strong at storing values and handling transaction processing. A knowledge graph is strong at explaining the meaning of values and business relationships. They play different roles.

This distinction mattered for DataNexus.

SQL calculates the value. The knowledge graph explains which business concept that value came from. One without the other was not enough.

A graph also needs evidence#

In a knowledge graph feature, concepts and evidence need to be shown together.

When a user clicks a node or relationship, they should be able to check where that relationship came from. For example, clicking the “net revenue” node should not just show connected nodes. It should also show the glossary term, document sentence, or related columns behind the definition.

When a graph works this way, it is no longer a simple relationship map. It becomes a UI for following the evidence behind an answer.

Suppose someone asks, “What is net revenue?” and the answer ends like this.

Net revenue is the amount after subtracting returns, discounts, and allowances from gross revenue.

Even if that sentence is correct, the user will ask again.

  • Where did this definition come from?
  • Is this our company’s definition?
  • Which columns is it connected to?
  • If this definition changes, which dashboards are affected?

So the answer needs at least this kind of information beside it.

source: Data Catalog Glossary
term: net revenue
source_id: urn:li:glossaryTerm:net_sales
related concepts: gross revenue, returns, discounts, allowances
linked columns: orders.amount, orders.refund_amount, orders.discount_amount
validation: PASS

Here, the knowledge graph is not decoration. It is the entry point for following the answer’s source.

A number alone is not enough to decide whether to trust it#

After looking at other products, I became more certain that evidence needs to appear together on the result screen.

Suppose the user asks:

What was the total order count in 2024?

If the AI only answers:

The total order count in 2024 was 42.

At first glance, nothing looks wrong. But these questions remain immediately.

  • Which table did it query?
  • How did it filter for 2024?
  • Were cancelled orders included?
  • Did this number come from the sample DB or the production DB?
  • Was SQL actually executed, or did the model just write a sentence?

A single number, “42”, is not enough to judge the answer.

Genie attaches a Trusted label and Snowflake keeps validation fields in the response JSON for the same reason. Without evidence on the screen, users either trust the answer without grounding or ignore it completely. Neither works in real business use.

So the DataNexus Chat UI does not show only the answer result. It shows the SQL query and execution order together. Source, fallback state, validation state, and permission scope appear in the same place.

Do not hide the SQL query either#

When building an NL2SQL product, hiding SQL is tempting.

Business users usually do not inspect SQL directly. The screen also gets more complex. The product experience of AI doing the analysis on their behalf can feel weaker too.

But in an enterprise data environment, SQL should not be completely hidden.

In DataNexus, SQL is not treated as an internal log that only developers see. It can be collapsed or expanded inside the answer.

The answer area shows the human-readable answer first.

The total order count in 2024 was 42.

Below that, SQL can be expanded or collapsed.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

Users want to see the result first, so SQL should not appear before the answer. But SQL should not disappear either.

SQL is the first piece of evidence someone can check when asking, “Where did this number come from?” Analysts or DBAs open it when needed. When an error happens, the same query should reproduce it.

Trace shows the path that produced the answer#

An AI answer does not end with one model call.

DataNexus has ontology and a catalog. DW/DM, NL2SQL, and validation logic also work together. An answer naturally passes through several steps.

Snowflake puts fields such as verified_query_used, question_category, and warnings in the response JSON for the same reason: to make that process traceable from the outside.

When a user question comes in, the system first classifies the intent. The processing order changes by question type. A metric definition goes to the catalog first, while an actual number goes to the DW/DM DB.

“What is net revenue?” should reach the catalog glossary before SQL runs.

“What was the total order count in 2024?” runs a SELECT against the DW/DM DB.

The screen should make this difference visible right away.

Not every internal log needs to be shown. For users, source, validation, and fallback are enough to start with. But the detailed Trace that operators and analysts can expand needs more information.

For example, a DB lookup answer could leave this trace.

Runtime Trace
trace_id: dnq_20260523_001
route: NL2SQL
source: DW/DM
source_id: mart.orders
semantic_context: Data Catalog Glossary + Ontology
verified_query: none
sql_execution: success
validation:
  schema_compliance: pass
  policy_check: pass
  source_binding: pass
permission_scope:
  user: sales_analyst
  rls_policy: sales_region_allowed
  masked_columns: []
fallback: false
warnings: []

If the answer came from the catalog, it looks like this.

Runtime Trace
trace_id: dnq_20260523_002
route: Glossary
source: Data Catalog Glossary
source_id: urn:li:glossaryTerm:net_sales
sql_execution: not_applicable
validation:
  source_binding: pass
  policy_check: pass
permission_scope:
  glossary_visibility: internal
fallback: false
warnings: []

The user screen only needs to show part of this.

source: DW/DM
validation: PASS
fallback: false

Strategy(MSTR) AUTO leaving referenced Metrics and Dimensions in the Interpretation area is dealing with the same problem. In an enterprise setting, it is hard to trust and use an answer when the criteria behind it are opaque.

Without Trace, users have to guess where the answer came from.

Permission scope is also part of the evidence#

Showing the source alone is not enough.

Tables or documents the user cannot access should not be used as evidence. Even for the same question, the answer and evidence can differ when the permission scope differs for sales, finance, or executive accounts.

That is why Trace needs to keep permission scope along with source.

permission_scope:
  user: sales_analyst
  allowed_domains: [Sales]
  row_level_filter: region IN ('KR-Seoul', 'KR-Gyeonggi')
  masked_columns: [customer_phone, customer_email]

This information does not need to be exposed as-is to every user. But operators and audit owners should be able to check it.

A trustworthy answer has to explain not only “which data did this come from?” but also “did this answer come from data this user is allowed to see?”

Do not hide fallback state#

A failure is more dangerous when it looks like success.

The DB connection drops, or SQL execution fails, but a sentence that looks correct still appears on the screen. Then the user may treat a wrong answer as correct.

Snowflake does not force SQL when the AI is not confident and can return suggestion content. Qurify lets users edit and run generated SQL directly. Both directions are useful because they do not hide failure or uncertainty outside the screen.

When DB lookup fails, fallback needs to appear. In DataNexus, this fallback is not hidden.

Sometimes the system still needs to show an answer after a failure. In a demo environment, there may be times when a prepared answer has to be shown. The more this happens, the more clearly the screen needs to mark whether it is a real DB result or fallback.

Runtime Trace
source: fallback demo response
sql_execution: failed
validation: FALLBACK
fallback: true
warnings:
  - DB connection failed. Demo response was used.

In a demo environment, missing this fallback label makes it easy to mistake a prepared answer for a live DB result.

Users should be able to see what the system did and what it could not do. Operators and developers should be able to trace where it failed and how often fallback happens.

A source makes the answer usable at work#

In DataNexus, the source that needs to be shown most often is the data catalog.

Take the term “net revenue”. Each company defines it differently. Some subtract only returns and discounts. Others include allowances or tax treatment too.

DDL does not contain that definition.

Tables only have columns like amount, discount_amount, and refund_amount. The business concept of net revenue lives outside the database.

That is why the term definition has to come from the catalog.

When someone asks “What is net revenue?” in the Chat UI, and the answer came from the catalog glossary, the screen should leave it as source: Data Catalog Glossary.

source: Data Catalog Glossary
term: net revenue
source_id: urn:li:glossaryTerm:net_sales
owner: Sales Data Domain

The user can confirm that the answer came from a glossary managed by the organization, not from the model’s general knowledge.

Ontology-based NL2SQL is not a way to make the model guess the right meaning on its own. It answers by bringing in the meaning the organization has already defined at runtime.

Data view and graph view should complement each other#

Putting a knowledge graph on the screen does not mean every user should start from the graph.

Most users look at the answer and number first. The graph comes after that.

So in DataNexus, the data view and graph view open separately but stay connected.

The data view shows the exact value, and the SQL view keeps how that value was calculated.

The total order count in 2024 was 42.

The calculation method is checked through SQL.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

When semantic relationships matter, the graph view opens.

Order
 ├─ has property: order_date
 ├─ has property: order_status
 └─ counted as: Order Count

When the company standard matters, the catalog view opens.

Glossary Term: Order Count
Definition: Number of order records created within the selected period
Owner: Sales Data Domain

If all four are expanded at once, the screen gets crowded. But each one needs to be available when needed.

Start with the answer. If the number looks strange, open SQL. If the business definition is unclear, go into the catalog. Open the graph only when relationships or impact scope matter.

The reason for this order is simple. Most users check the result first. When something feels suspicious, they follow the evidence.

Interactive editing needs review before direct application#

If a user says, “Change this node type to strategic customer” or “Add the urgent keyword to this relationship,” the knowledge system can keep being refined.

That direction is right. But in an enterprise environment, applying those changes immediately is risky. One term change can affect SQL generation, dashboards, reports, and permission policies.

So in DataNexus, conversational edits do not go straight into operational knowledge. They follow this flow.

user edit request
  → create change candidate
  → analyze impact scope
  → data owner review
  → apply to Glossary / Ontology after approval
  → resync related indexes

Users need a way to ask AI to edit knowledge. But an operational knowledge base should not be treated like a personal notebook. It is an organizational standard, so approvals and history need to remain.

That is why the Knowledge Graph UI becomes part of the screen design that creates trust.

The graph does more than show relationships. It has to become a safety layer for checking impact scope.

Accuracy and reviewability need to be checked separately#

In NL2SQL evaluation, I look at EX, or Execution Accuracy. It checks whether the generated SQL returns the same result as the expected answer.

But EX is an offline evaluation metric. It is measured on benchmarks that have a gold SQL query or a gold execution result.

In production runtime, there is usually no gold answer. The question the user just asked does not come with a prepared gold SQL query.

So PASS on the operational screen should not mean that EX passed. PASS should be judged by runtime validation.

PASS
- SQL execution or catalog lookup succeeded
- Schema Compliance passed
- permission / RLS policy passed
- source or verified query / glossary / graph node matched
- no critical warning

REVIEW
- SQL ran, but binding is uncertain
- verified query was not used
- source matching is weak
- question is ambiguous or router detoured
- data owner or analyst review is needed

FALLBACK
- SQL execution failed
- DB was not queried
- suggestion only
- demo response
- substitute answer, not live operational data

Accuracy and reviewability also need to be checked separately. Even when the SQL result is right, the screen can still be hard to trust and use.

The SQL is right, but the wording is stiff.

The number is right too. But if the screen does not show the basis for it, the user cannot make the next judgment.

The source exists, but if it is hard to find, the user has to ask again.

TrustSQL treats Text-to-SQL reliability as more than producing correct SQL for answerable questions. It also values the ability to abstain when a question cannot be answered. LinkedIn’s enterprise Text-to-SQL case also builds a knowledge graph from schema, documents, query logs, wiki, code, and domain knowledge. It shows a direction where chatbot UI supports different user intents and follow-up questions.

DataNexus calls this criterion the Taste Gate. Even when EX passes, if evidence is missing or the source is unclear, the answer fails the Taste Gate.

MetricWhat it checks
EXWhether SQL execution matches the gold answer in an offline benchmark
CQWhether the ontology passes Competency Questions
Schema ComplianceWhether SQL stays inside the allowed schema and policy
Hallucination RateWhether it invents columns, terms, or sources
Query Router AccuracyWhether the question went to the right flow among SQL, Graph, Glossary, and Vector
Runtime ValidationWhether execution, policy, source, and warning checks pass at runtime
Taste GateWhether the user can review the answer and move on to the next judgment

If EX passes but the user still asks, “Can I trust this?”, the product experience has failed.

On the other hand, even when SQL fails, the user can still understand the system state if fallback is clearly marked and the review path remains open.

That is why EX can pass and still be checked again at the Taste Gate.

Moving this principle onto the screen#

I summarized the screen structure like this.

I started with the answer card. The top row shows trust state and source, and the middle shows the business summary and chart. SQL and Trace open from the bottom accordion.

+---------------------------------------------------------------+
| [Status: PASS / REVIEW / FALLBACK] | [Source: Data Catalog]   |
+---------------------------------------------------------------+
| "Net revenue this quarter was KRW 120B."                     |
| [Chart: Bar / Line auto-rendered]                             |
+---------------------------------------------------------------+
| ▶ Show Evidence: SQL & Trace                                  |
+---------------------------------------------------------------+
| [Tabs] 1. SQL  2. Data Lineage  3. Metric Logic  4. Graph      |
+---------------------------------------------------------------+
| [👍 / 👎]       [Request Revision]  [Export Query]            |
+---------------------------------------------------------------+

An executive can stop after checking the top two blocks. An analyst opens the accordion and reads SQL. A DBA checks the passed-through tables in the Data Lineage tab. A data owner looks at term and metric relationships in the Graph tab.

The feedback buttons connect to Vanna’s Self-Learning loop and the DataNexus review queue. Revision feedback remains as a training-data candidate.

Start with the summary. When more review is needed, open SQL, data lineage, or the knowledge graph. Do not dump everything at once.

Do not show an answer without a source. At least one of SQL, document ID, glossary term, or graph node should stay attached.

The status labels are PASS, REVIEW, and FALLBACK.

The label alone should help users decide the next action.

PASS: usable for business judgment now
REVIEW: open the result and verify once
FALLBACK: not a real DB result or not a normal execution result

Here, PASS does not mean EX passed.

PASS means execution succeeded, policy validation passed, the source matched, and there was no critical warning.

Exposing raw confidence scores from the model is another option. But an uncalibrated number shifts interpretation work onto the user. Three labels that clearly say “ready to use,” “verify before acting,” and “not a real result” are easier to act on.

Keep the answer text short. It should read more like a colleague giving a quick explanation than a report. SQL can be collapsed, but it should not be removed. Trace and status labels are the minimum device for distinguishing a DB result, a catalog answer, and fallback.

If the answer does not show its evidence, it becomes just another plausible chatbot.

The problem I want to solve in the DataNexus Chat UI is less about “making AI write better answers” and more about letting users check where the answer came from.

In the next post, I will cover the flow where users ask again and rewrite answers on top of this screen.
