11. Designing Chat UIs for Uncertain AI Answers

How the DataNexus Chat UI shows answer evidence: SQL, Runtime Trace, fallback state, permission scope, sources, and the knowledge graph alongside the answer.

Up to post 10, I had been analyzing the types of errors that most often appear during SQL generation.

I used BIRD and Spider benchmarks, tested several models, and grouped wrong answers into types such as COLUMN_BINDING, VALUE_BINDING, and DERIVED_METRIC. After analyzing those SQL error profiles more deeply, I shifted the research focus.

“Then how should generated SQL and answer evidence appear on the screen?”

I had focused on accuracy tests and had missed what real users need to check on the screen. A data analysis AI used in a business environment has to make it clear where the number came from. If there is no way to check the evidence, users eventually have to run the SQL again even when the result looks correct.

For the DataNexus Chat UI, polishing the LLM’s text output was not enough. I decided to place the SQL query, source, permission scope, validation state, and Fallback state next to the final response. Only then can users trust the data in meeting materials or practical decisions.

How other products display evidence#

As of May 2026, I reviewed publicly available documents and demo screens from major domestic and global products.

Global products:

  • Databricks Genie
  • Snowflake Cortex Analyst
  • Strategy(MSTR) AUTO
  • ThoughtSpot Spotter 3
  • Vanna.ai

Domestic products:

  • Qurify(SmartMind)
  • MISO(GS Neotek & 52g)

I focused on one question: “What does a user need to check before using this answer at work?” Each product answered it differently, but none of them relied only on a single-line text response.

  • Databricks Genie uses Trusted assets to mark answers that use verified SQL queries or functions with a ‘Trusted’ label. Users ask questions in natural language inside Genie Space and inspect tables, columns, guidelines, SQL queries, and other context in the Data and Instructions tabs. Unity Catalog permissions also apply, so answers use only data the user is allowed to see.

  • Snowflake Cortex Analyst gave direct clues for screen design through its REST API response structure. The response content types are text, suggestion, and sql, and SQL content may include fields such as verified_query_used. The API also returns warnings and response_metadata.question_category, so ambiguous questions that are hard to turn into SQL can receive alternative questions as suggestion content.

  • Strategy(MSTR) AUTO showed, through Auto Dashboards and Strategy AI documentation and demos, how features such as derived metrics, selectors, auto narratives, and auto answers are expanding. The Interpretation area in the demo was especially useful for DataNexus Trace UI design because it explained which Metric and Dimension the user’s question was mapped to.

  • ThoughtSpot Spotter 3 was the closest fit for the DataNexus goal of presenting answer evidence on screen. Spotter 3 handles not only structured data warehouse data, but also information scattered across work tools such as Slack, Salesforce, Jira, and SharePoint. Its flow of bringing scattered context into the answer, letting the user inspect the result, and then moving into follow-up analysis was particularly helpful for DataNexus design.

  • Vanna.ai focuses on the learning loop and permission handling. According to the Vanna 2.0 docs , the LLM is connected to tools for SQL execution and chart generation, and Tool Memory stores successful question-SQL-result pairs in a reusable form. User permissions are applied across the whole pipeline. Progress state, SQL code, tables, charts, and natural-language summaries all appear in one flow.

For domestic products, I relied on public product pages and release notes.

  • Qurify is a Text-to-SQL product for interacting with databases in natural language. As of version 0.4.4, it shows generated SQL on the screen and lets users edit, run, and copy it. Its way of continuing follow-up questions by citing execution results was also useful for DataNexus design.

  • MISO is closer to an enterprise AI platform and workflow app builder than to a single NL2SQL product. I used its public report automation and document search cases as benchmarks while shaping the operational AI screens for DataNexus.

Those benchmarks shaped the DataNexus screen direction. Before polishing the final response, the UI needed a pipeline where users could first check SQL, source, validation state, permission scope, and Fallback state.

I also had to revisit the semantic layer. A typical semantic layer maps business terms and metric definitions to SQL. DataNexus Glossary, Ontology, and Knowledge Graph go beyond table- and column-level metadata and include business rules and relationships between concepts. If those relationships stay only inside the system, users have to go back to documents. The answer card needed an interface where users could open them directly.

Using knowledge graphs to verify sources#

DataNexus defines ‘answer evidence’ as a network that connects the concepts used in the answer to sources, related tables, and columns. At first, I also considered a separate analysis screen where users could follow nodes and edges in a knowledge graph. But in real business environments, visual polish matters less than the information users can inspect after a click.

A knowledge graph in the UI should answer questions like these immediately.

  • Which business concept was this answer based on?
  • Which physical data, such as tables or columns, is that concept connected to?
  • Who defined this relationship, and where is the documentary evidence?
  • If the metric definition changes, which answers are affected?

While traditional keyword search can detect terms such as “net revenue”, it struggles to infer how that metric is tied to gross revenue, returns, discounts, and allowances. In business queries, the contextual links between concepts matter more than simple term matching. DataNexus uses ontology and a knowledge graph to bring those links onto the screen.

In DataNexus, ontology acts as a reference map for enterprise-standard terms and relationships.

  • Concept: customer / product / order / net revenue
  • Relationship: a customer creates an order / net revenue is calculated from gross revenue
  • Property: order timestamp, payment method, order status
  • Rule: a refund can occur only after payment is completed
  • Metric: average order value, year-over-year growth rate

The knowledge graph maps this reference map to physical data. Concepts become nodes, relationships become edges, and documents, tables, columns, and formulas connect into one network.

The goal is not to replace the RDBMS. It handles transactions, while the knowledge graph explains what that data means in business terms. In DataNexus, it was important to show the value calculated by SQL and the meaning explained by the knowledge graph side by side.

Direct source inspection within the graph#

A knowledge graph interface is not enough if it only visualizes concept nodes. It also has to show the sources and metadata behind those concepts.

When a user clicks a node or edge, the evidence behind that relationship should open immediately. For example, when the user clicks the “net revenue” node, the screen should expand step by step into adjacent nodes, the Data Catalog Glossary entry, the source passage in the original document, and the mapped DB columns. That is how the user can check the answer all the way through.

Even with a clear text summary of “What is net revenue?”, practitioners still need the evidence behind it.

Net revenue is the amount after subtracting returns, discounts, and allowances from gross revenue.

Even if the response is accurate, practitioners ask for the next level of verification.

  • Where did this business logic come from?
  • Is it the current enterprise standard?
  • Which physical columns were used in the calculation?

So the answer card needs at least this kind of metadata.

 - source: Data Catalog Glossary
 - term: net revenue
 - source_id: urn:li:glossaryTerm:net_sales
 - related concepts: gross revenue / returns / discounts / allowances
 - linked columns: orders.amount, orders.refund_amount, orders.discount_amount
 - validation: PASS

Users use this information as a clue and follow the evidence inside the knowledge graph.

Standalone metrics alone cannot build trust#

While benchmarking global products, it became clear that data analysis result screens need to show evidence for trust alongside the answer.

Suppose a user asks:

What was the total order count in 2024?

A number-only response can look complete:

The total order count in 2024 was 12,486.

The response is technically correct, but practitioners immediately ask questions like these.

  • Which mart table was queried?
  • What physical timestamp defined the 2024 period condition?
  • Were cancelled and returned orders included in the count?
  • Was this operational data or staging data?

An uncontextualized metric like “12,486” lacks the grounding required for validation, making it unsuitable for official executive reports. The DataNexus Chat UI places the SQL query, execution process (Trace), validation state, and access scope below the natural-language answer so users have criteria for judgment.

The case for keeping SQL on the screen#

When designing an NL2SQL UI, it is tempting to hide SQL completely for the readability of business users. The concern is that displaying SQL clutters the interface and obscures the AI’s autonomy. But in an enterprise environment, transparency matters, so SQL cannot be hidden completely.

DataNexus does not leave SQL only in internal logs. It implements SQL as a block that can be collapsed and expanded inside the answer.

  • Primary natural-language answer layer:

The total order count in 2024 was 12,486.

  • Secondary expandable SQL layer:
SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

The query remains collapsed by default. When number verification is needed, an analyst or DBA can open the query immediately, rerun it in the same environment, and narrow the error scope.

For users who are not comfortable reading SQL, the screen first shows an Interpretation summary that explains how the system parsed and mapped the user’s intent. This area helps users catch VALUE_BINDING errors.

FieldInterpreted value
Period2024-01-01 - 2024-12-31
Metricnet revenue
Formulagross revenue - returns - discounts - allowances
Dataorders
Permissioncurrent user permission scope

Users do not have to read the full SQL to first check whether their intent was reflected correctly. If something is wrong, they can fix the condition directly on the screen and rerun it.

The screen also shows SQL Inspect information based on a dry run to reduce risks before executing the query.

SQL Inspect

 - EXPLAIN summary: Index Scan
 - Estimated rows: 124,503
 - Full-scan warning: none
 - Estimated cost: low
 - Policy check: passed

Trace lets users check the answer flow#

Trace is the execution record that shows which internal pipeline processed the user’s question. The DataNexus answer process does not end with a single model call. It goes through ontology validation, catalog mapping, and question-type classification before sending the question to the right path.

The system routes requests based on question type.

  • “What is the definition of net revenue?” -> query the catalog glossary immediately without running SQL.
  • “What was the total order count in 2024?” -> run a SELECT statement against the data warehouse (DW).

The detailed Trace layer keeps logs like these for audit and debugging.

  • Database lookup Trace example:
 - Runtime Trace
 - trace_id: dnq_20260523_001
 - route: NL2SQL
 - source: DW/DM
 - source_id: mart.orders
 - semantic_context: Data Catalog Glossary + Ontology
 - verified_query: none
 - sql_execution: success
 - validation:
   - schema_compliance: pass
   - policy_check: pass
   - source_binding: pass
 - permission_scope:
   - user: sales_analyst
   - rls_policy: sales_region_allowed
   - masked_columns: []
 - fallback: false
 - warnings: []
  • Data catalog lookup Trace example:
 - Runtime Trace
 - trace_id: dnq_20260523_002
 - route: Glossary
 - source: Data Catalog Glossary
 - source_id: urn:li:glossaryTerm:net_sales
 - sql_execution: not_applicable
 - validation:
   - source_binding: pass
   - policy_check: pass
 - permission_scope:
   - glossary_visibility: internal
 - fallback: false
 - warnings: []

Here, sql_execution: not_applicable is not an error. It means the path was a metadata lookup that did not need SQL execution. In this case, the system verifies data integrity through validation.source_binding. The user screen shows only summary indicators such as Source: DW/DM, Validation: PASS, and Fallback: False instead of complex logs.

Granular routing by execution path#

To improve operational stability, DataNexus defines the runtime execution path (route) in more detail. Even for an answer that passed system validation and reached ‘PASS’, the trust level differs depending on whether it reused a verified fixed query or executed Raw SQL generated at runtime.

routeMeaning
semantic_routedInterpreted the question through the semantic layer and ontology rules, then chose the path
verified_query_matchedMatched an already verified and approved SQL query or function
raw_sql_generatedDid not match a verified asset, so generated and executed new SQL at runtime
raw_sql_fallbackThe recommended path failed and the system detoured into Raw SQL generation
glossary_onlyChecked only the catalog and glossary definitions without querying the DB
graph_traversalFollowed nodes and edges in the knowledge graph to check concept relationships and impact scope
fallback_demoUsed the demo response layer when there was no DB connection
unsupported_metricNo organizational metric definition existed, so the answer was held or confirmation was requested

The system has to record this internal path and keep it visible enough for users to follow how the AI answer was processed. When the system is uncertain or detours into an exception path, users can see that state and respond accordingly.

Reduce ambiguity before answering#

Showing evidence after the answer is necessary, but the system also has to reduce ambiguity at the question stage. Real work questions are usually abstract. So before generating SQL right away, the system first shows ontology-based interpretation candidates and narrows the intent.

[Question] Show me 2024 net revenue

DataNexus checks the query conditions.

 ○ net_revenue_v1: gross revenue - returns - discounts - allowances
 ○ net_revenue_KR: Korean accounting standard, includes tax treatment
 ○ net_revenue_global: IFRS standard

[Run with this definition] [Choose manually]

The detailed logic chosen by the user is kept in the session and conversation context. Internally, the system uses this state structure.

{
  "persisted_context": {
    "accepted_metric": "urn:li:glossaryTerm:net_sales",
    "accepted_period": "2024",
    "accepted_filters": ["region = KR"],
    "rejected_interpretations": ["gross_sales_only"],
    "scope": "current_thread_only"
  }
}

The scope: current_thread_only setting keeps the selected metric criteria and filter conditions active for the current conversation. The user can ask follow-up questions without losing the flow. The state is named persisted_context so the schema reads as session state rather than a generic carryover flag.

Permission scope as core evidence#

Even if the source is correct, the answer must not include information outside the user’s security permissions. The same natural-language question can return a different data scope depending on department or role. Trace maps permission-scope information together with the source.

permission_scope:
  - user: sales_analyst
  - allowed_domains: [Sales]
  - row_level_filter: region IN ('KR-Seoul', 'KR-Gyeonggi')
  - masked_columns: [customer_phone, customer_email]

The standard UI omits the granular details of row-level security (RLS) filters and masking policies. Instead, the backend records them so data security auditors can inspect them later in the Inspect layer.

Fallback states and infrastructure failures#

The most dangerous failures are silent: an infrastructure outage can masquerade as a successful response when stale cached data appears without warning.

Rather than burying infrastructure issues in a normal answer, DataNexus surfaces database communication outages and query timeouts directly on the screen.

 - Runtime Trace
 - source: fallback demo response
 - sql_execution: failed
 - validation: FALLBACK
 - fallback: true
 - warnings:
   - DB connection failed. Demo response was used.

In production, DataNexus separates FALLBACK · demo_response from FALLBACK · sql_execution_failed. The UI must clearly distinguish between a simple demo mode and a real infrastructure failure. When access constraints or missing data sources prevent execution, the system catalogs the state as ABSTAIN rather than a generic fallback.

User labelSecondary qualifierMeaning
PASSverified_query_matchedMatches a verified query or function asset
PASSsemantic_routedPassed through the semantic layer and organizational ontology rules
PASSglossary_exact_matchMatches an approved glossary definition in the data catalog
REVIEWraw_sql_onlySQL ran, but it did not match a verified asset
REVIEWlow_binding_confidenceColumn or value mapping confidence is low and needs user review
REVIEWambiguous_metricTwo or more metric definitions can be mapped
FALLBACKsql_execution_failedSQL syntax error or execution engine failure
FALLBACKdemo_responseShows static guide data, not a live DB query result
FALLBACKsuggestion_onlyCould not generate SQL, so only alternative question candidates are returned
ABSTAINinsufficient_contextInternal information or metadata is insufficient for an answer
ABSTAINpermission_blockedSecurity policy or access permission prevents the answer

Anchoring answers in verified data sources#

In business queries, the most important reference point is the data catalog. A KPI such as “net revenue” can include or exclude different items depending on the company’s business domain and accounting rules.

A typical DDL schema defines only physical column structures, such as amount or discount_amount. It does not carry business context. The term definition tied to actual business rules has to come from the official data catalog at runtime.

 - source: Data Catalog Glossary
 - term: net revenue
 - source_id: urn:li:glossaryTerm:net_sales
 - owner: Sales Data Domain

When the data domain owner (owner) and glossary ID are shown together, users can check the result against the enterprise standard instead of relying on guesswork.

View values and relationships inside one answer card#

A knowledge graph or lineage view can be a good verification tool, but users do not need to start from a complex network map every time. Most users prefer to see the summary number first and then trace the detailed evidence backward.

DataNexus groups summary, SQL, catalog, and graph views into tabs inside one answer card.

  • [Default screen] Summary view
The total order count in 2024 was 12,486.
  • [Tab 1] SQL view
SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';
  • [Tab 2] Graph view
Order
 ├─ has property: order_date
 ├─ has property: order_status
 └─ counted as: Order Count
  • [Tab 3] Catalog view
 - Glossary Term: Order Count
 - Definition: Number of order records created within the selected period
 - Owner: Sales Data Domain

The UI usually starts with a compact summary card so the evidence layers do not overwhelm the layout. When number verification or domain-rule checking is needed, the user opens the lower layers one by one.

A review layer for conversational knowledge base edits#

When a user requests a knowledge base edit directly through the chat interface, such as “change this node type to strategic customer,” the system can create a change candidate object for updating knowledge assets.

In an enterprise environment, however, it is risky to apply conversation results directly to the knowledge base. A metadata change can trigger a cascading effect across the downstream SQL generation pipeline and dashboard reports.

DataNexus puts staging and approval steps around conversational edit requests.

user edit request
  → create change candidate (Draft)
  → impact analysis
  → data owner review and policy check
  → apply to Glossary / Ontology master after final approval
  → resync related knowledge graph indexes

Approval queues and change history help prevent incorrect edits from entering the knowledge base and help preserve the integrity of data assets.

Accuracy and reviewability are different problems#

In offline benchmarking, we usually measure EX (Execution Accuracy), which checks whether the SQL generated by the model matches the result of the gold SQL. But in real-time runtime environments, there is no predefined Gold Answer, so offline metrics cannot be used as-is.

The operational PASS label does not simply mean EX passed with no SQL syntax error. It follows runtime validation criteria that check whether policy and source validation pipelines were passed.

 PASS
   - SQL execution engine has no errors and data catalog mapping is complete
   - schema policy check passed
   - data security policy and row-level security (RLS) filters passed
   - matches an official glossary or verified query asset
   - no important warnings

 REVIEW
   - SQL ran, but Binding confidence is low
   - generated a new query instead of using an approved verified query asset
   - mapping uncertainty stemming from low metadata density or intermittent source connectivity
   - routing shifted to an alternative path because the question was ambiguous
   - data owner or analyst review is needed

 FALLBACK
   - SQL execution failed because of a syntax error or connection timeout
   - falling back to the internal cache due to blocked database queries
   - routed to suggestion-only state or a demo response
   - result did not come from a real operational data source

Even if SQL execution succeeds and the value is correct (EX passed), it is hard to use for practical judgment if the screen does not show which data and which criteria produced the value.

Research such as TrustSQL also sees Text-to-SQL reliability not only as the ability to return the right answer, but also as the ability to abstain from ambiguous questions that cannot be answered. Enterprise cases such as LinkedIn Text-to-SQL also combine domain knowledge and schema context into a knowledge graph and verify answers in multiple stages.

DataNexus does not evaluate this with one score. It checks whether the UI gives users enough clues to decide what to do next. I call this review principle the “Practitioner Sanity Check.”

CriterionWhat it checks
EXWhether generated SQL execution matches the gold data in an offline benchmark
CQWhether the ontology can answer key business competency questions
Schema ComplianceWhether generated SQL follows internal schema and security guidelines
Hallucination RateWhether it avoids inventing non-existent columns, fake business terms, or false sources
Query Router AccuracyWhether the question was sent to the right engine among SQL, Graph, Glossary, and Vector
Runtime ValidationWhether execution, security policy, metadata source, and warning logs pass at runtime
Practitioner Sanity CheckWhether the user can move into practical judgment after seeing evidence in the result card

Applying this principle to the actual screen#

Based on the principles above, I redesigned the final answer card layout. The top fixes the trust state badge and primary source. The middle automatically renders the key summary data and chart. SQL and Trace for deeper verification are placed in collapsible tabs at the bottom.

+---------------------------------------------------------------+
| [Status: PASS / REVIEW / FALLBACK / ABSTAIN] | [Source: Data Catalog] |
+---------------------------------------------------------------+
| "This quarter's net revenue is KRW 120B."                    |
| [Chart: Bar / Line auto-rendered]                             |
+---------------------------------------------------------------+
| ▶ Show Evidence: SQL & Trace                                  |
+---------------------------------------------------------------+
| [Tabs] 1. SQL  2. Data Lineage  3. Metric Logic  4. Graph      |
+---------------------------------------------------------------+
| [👍 / 👎]       [Request revision]  [Export]                  |
+---------------------------------------------------------------+

Executives can make a quick judgment from the top status label and summary number. Working data analysts or DBAs can open the lower Inspect layer and check SQL, data lineage, and the knowledge graph layer by layer.

The feedback area at the bottom does not inject collected user corrections directly into real-time learning. It sends them to a governance review queue. It is a safety mechanism.

LayerContent
Primary cardcore natural-language answer / status badge / primary source / chart
Inspect layerdetailed SQL / Runtime Trace log / permission scope / Data Lineage / knowledge graph and source highlights

Evidence data should go beyond a simple document link. It follows the DataHub URN and physical table.column FQN (Fully Qualified Name) and narrows the evidence down to specific values and source passages.

However, if reliability labels such as Trusted or PASS are used too often without conditions, users may believe the label without checking the content. That is why DataNexus turns on labels only when runtime validation criteria are met.

 - PASS: ready to use for practical judgment or report citation
 - REVIEW: needs lower-level highlights and SQL review for number validation
 - FALLBACK: alternative data or demo response is shown because of infrastructure outage or a similar issue
 - ABSTAIN: no answer because of permission limits or missing data source

The natural-language response at the top of the card gives a concise summary of the metric and its primary criterion. SQL and Trace stay collapsed by default so they do not clutter the interface. Still, the path must remain on the screen so users can follow the data flow whenever needed.

The operator dashboard does more than collect simple metrics. It tracks which citation users clicked, which detailed route frequently falls into Fallback, and keeps adjusting the overall AI automation scope.

The biggest question I ran into while planning the BI Agent system was this: “Can users trust the data shown on the screen and use it in real work without second-guessing it?” If the screen cannot show evidence, polished text alone will not earn trust in an enterprise environment.
