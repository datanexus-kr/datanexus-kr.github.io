11. How Should We Show AI Answers That Can Be Wrong?

How the DataNexus Chat UI shows the evidence behind an answer. After reviewing domestic and global products, I decided to show SQL, Runtime Trace, fallback state, sources, and knowledge-graph evidence together.

Up to post 10, I had been checking where SQL generation most often goes wrong.

I ran BIRD and Spider. I changed models. I grouped wrong answers into types like COLUMN_BINDING, VALUE_BINDING, and DERIVED_METRIC. While looking at those SQL generation errors, another question came up.

How should a correct SQL query be shown on the screen?

The accuracy experiments had missed something. If a data analysis AI is going to live inside a real work interface, people need a way to check the evidence. Even when the number is correct, it feels unsafe if nobody can tell where it came from. If the SQL is right but the screen is hard to review, people go back to the query editor.

For the DataNexus Chat UI, a well-written answer was not enough. The screen had to show the evidence behind the number and the sentence.

I first looked at other products#

Before designing the screen, I looked at products already in use.

Global products

  • Databricks Genie
  • Snowflake Cortex Analyst
  • Strategy(MSTR) AUTO
  • Vanna.ai

Domestic products

  • Qurify(SmartMind)
  • MISO(GS Neotek & 52g)

I used the same lens for all of them.

How does this product make an AI answer trustworthy?

The details differed, but they shared one pattern: they did not just throw an answer onto the screen.

  • Databricks Genie puts a ‘Trusted’ badge beside the answer, with ‘Fix it’ and ‘Request review’ paths nearby.

  • Snowflake Cortex Analyst is closer to an API than a UI, but it leaves validation fields such as verified_query_used and question_category in the response JSON. When confidence is low, it returns suggestion instead of SQL.

  • Strategy(MSTR) AUTO shows the question interpretation and referenced Metrics/Dimensions in its Interpretation area. Even when data retrieval fails, it still shows the SQL and interpretation result.

  • Vanna.ai was useful for its four-block response and feedback loop. SQL/Table/Chart/Summary can be composed separately, and thumbs up/down feedback is sent to Training Data. When RAG is wrong, the result screen creates material for fixing it.

The domestic products approached the problem a little differently.

  • Qurify rechecks SQL execution errors through a Self-Correction loop. Its knowledge graph tab also stood out because it lets users visually explore relationships between data.

  • MISO uses the Ask → Look → Investigate → Voice → Evolve flow to prevent premature judgment.

After looking at these products, the layer DataNexus needed became clearer. Writing a nicer answer was less important than leaving SQL, source, validation state, and fallback state on the screen.

One more thing kept coming up: a screen that shows the relationships themselves.

A knowledge graph is closer to evidence structure than a pretty diagram#

At first, I thought of the knowledge graph as a separate analysis screen.

Nodes, edges, and a canvas where users can zoom or pan around relationships. It looks good in a technical demo. In a real work screen, though, a graph being visually impressive is not enough.

The important question is whether it can answer things like this.

  • Which concept did this answer reference?
  • Which data is that concept connected to?
  • Who defined the relationship?
  • Where is the evidence for this relationship?
  • If a formula or metric definition changes, which answers are affected?

Traditional search is good at finding keywords. If a document contains the word “net revenue”, search can find that document. But it is hard for search alone to understand how net revenue relates to gross revenue, returns, discounts, and allowances.

That is why ontology and knowledge graphs matter.

An ontology is close to a blueprint. It defines the concepts and relationships used inside the organization.

For example:

  • Concept: customer, product, order, net revenue
  • Relationship: a customer creates an order, net revenue is calculated from gross revenue
  • Property: order timestamp, payment method, order status
  • Rule: a refund can happen only after payment is completed
  • Metric: average order value, net revenue, year-over-year growth rate

A knowledge graph connects that blueprint to actual data. Concepts become nodes, and relationships become edges. Documents, tables, columns, metric definitions, and formulas enter the same network.

The important point is that this does not replace an RDBMS.

An RDBMS is strong at storing values and handling transactions. A knowledge graph is strong at exposing meaning and relationships. They are not competitors. They do different jobs.

This distinction mattered for DataNexus.

SQL calculates the value.

The knowledge graph explains which business concept that value came from.

One without the other is not enough.

A graph also needs evidence#

What I found useful in Qurify’s knowledge graph feature was not the graph itself, but something closer to a Concept-Evidence Graph.

When a user clicks a node or relationship, they should be able to see where that relationship came from. For example, clicking the “net revenue” node should not just show connected nodes. It should also show the glossary term, document sentence, or related columns behind the definition.

When a graph works this way, the screen changes meaning.

It is not just showing a network.

It becomes a UI for following the evidence behind an answer.

Suppose someone asks, “What is net revenue?” and the answer ends like this.

Net revenue is the amount after subtracting returns, discounts, and allowances from gross revenue.

Even if that sentence is correct, the user will ask again.

  • Where did this definition come from?
  • Is this our company’s definition?
  • Which columns is it connected to?
  • If this definition changes, which dashboards are affected?

So the answer needs at least this kind of information beside it.

source: Data Catalog Glossary
term: net revenue
related concepts: gross revenue, returns, discounts, allowances
linked columns: orders.amount, orders.refund_amount, orders.discount_amount
validation: PASS

Here, the knowledge graph is not decorative. It is a path into the evidence behind the answer.

A result that looks correct is still hard to trust from the number alone#

After looking at those products, the need to keep evidence on the screen became clearer.

Suppose the user asks:

What was the total order count in 2024?

If the AI only answers:

The total order count in 2024 was 42.

At first glance, nothing looks wrong. But the user will want to know more.

  • Which table did it query?
  • How did it filter for 2024?
  • Were cancelled orders included?
  • Did this number come from the sample DB or the production DB?
  • Was SQL actually executed, or did the model just write a sentence?

A single number, “42”, is not enough to judge the answer.

This is why Genie has a ‘Trusted’ badge and Snowflake leaves validation fields in the response JSON. Without evidence on the screen, users either accept the answer blindly or ignore it completely. Neither works in real business use.

So the DataNexus Chat UI does not show only the answer result. It keeps the SQL query, execution order, context source, fallback usage, and validation state visible together.

Do not hide the SQL query either#

When building an NL2SQL product, hiding SQL is tempting.

Business users usually do not inspect SQL directly. Showing it also makes the screen more complex. More than anything, it can weaken the feeling that AI is doing the analysis for them.

But in an enterprise data environment, SQL should not be completely hidden.

Snowflake handles this with a collapse/expand panel. The default screen stays simple, and SQL opens only when someone needs it.

DataNexus follows the same direction. SQL is not treated as an internal log that only developers see. It sits inside the answer in a collapsible area.

The answer card shows the human-readable answer first.

The total order count in 2024 was 42.

Below that, SQL can be expanded or collapsed.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

People want the result first. SQL should not appear before the answer. But SQL should not disappear either.

SQL is the first clue when someone asks, “Where did this number come from?” Analysts or DBAs need to check it when necessary and reproduce the issue with the same query when something goes wrong.

Trace shows the order behind the answer#

An AI answer does not end with one model call.

DataNexus has ontology and a catalog. DW/DM, NL2SQL, and validation logic also move together. An answer naturally passes through several steps.

Snowflake puts fields such as verified_query_used and question_category in the response JSON for the same reason: the order can be traced from the outside.

When a user question comes in, the system first classifies the intent. The processing order changes depending on whether the question is asking for a metric definition, an actual number, a data location, or impact analysis.

“What is net revenue?” should reach the catalog glossary before SQL runs.

“What was the total order count in 2024?” runs a SELECT against the DW/DM DB.

The user needs to see that difference on the screen.

Not every internal log needs to be shown. Three fields are enough for a first pass: source, validation, and fallback.

Runtime Trace
source: DW/DM
validation: PASS
fallback: false

If the answer came from the catalog, it looks like this.

Runtime Trace
source: Data Catalog Glossary
validation: PASS
fallback: false

Strategy(MSTR) AUTO shows which Metrics/Dimensions the AI referenced in its Interpretation area. In an enterprise setting, it is hard to trust and use an answer when the criteria behind it are opaque.

Without Trace, users have to guess where the answer came from.

Fallback does not hide failure#

A failure is more dangerous when it looks like success.

The DB connection drops, or SQL execution fails, but a sentence that looks correct can still appear on the screen.

Then the user may believe a wrong answer.

Snowflake does not force SQL when the AI is not confident. It returns suggestion content and makes that explicit. Qurify analyzes SQL failures and retries through its Self-Correction loop. Both products avoid hiding failure.

DataNexus does not hide fallback either.

Sometimes the system still needs to show an answer after a failure. In a demo environment, it may show a prepared answer. If so, it is even more important to mark whether it is a real DB result or fallback.

Runtime Trace
source: fallback demo response
validation: REVIEW
fallback: true

During demos, missing this fallback label could make a prepared answer look like a live DB result.

Users should be able to see what the system did and what it could not do. Operators and developers should be able to trace where it failed and how often fallback happens.

A source makes the answer usable at work#

In DataNexus, the source that needs to appear most often is the data catalog.

Take the term “net revenue”. Each company defines it differently. Some subtract only returns and discounts. Others include allowances or tax treatment too.

DDL does not contain that definition.

Tables only have columns like amount, discount_amount, and refund_amount. The business concept of net revenue lives outside the database.

That is why the term definition has to come from the catalog.

When someone asks “What is net revenue?” in the Chat UI, and the answer came from the catalog glossary, the screen should leave it as source: Data Catalog Glossary.

source: Data Catalog Glossary
term: net revenue

The user knows this answer did not come from the model’s general knowledge. It came from a glossary managed by the organization.

Ontology-based NL2SQL is not the model guessing the right meaning on its own. The runtime brings in the meaning the organization has already defined.

Data view and graph view should complement each other#

Putting a knowledge graph on the screen does not mean every user should start from the graph.

Most users look at the answer and number first. The graph comes after that.

So in DataNexus, the data view and graph view should be separate but connected.

The data view shows the exact value.

The total order count in 2024 was 42.

The SQL view shows how the value was calculated.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

The graph view shows semantic relationships.

Order
 ├─ has property: order_date
 ├─ has property: order_status
 └─ counted as: Order Count

The catalog view shows the organizational standard.

Glossary Term: Order Count
Definition: Number of order records created within the selected period
Owner: Sales Data Domain

If all four are expanded at once, the screen gets crowded. But each one needs to be available when needed.

Start with the answer. Open SQL when review is needed. Open the catalog when the business definition is unclear. Open the graph when relationships or impact scope matter.

That order feels natural.

Interactive editing should go to a review queue first#

Another useful point in Qurify’s knowledge graph approach is interactive editing.

If a user says, “Change this node type to strategic customer” or “Add the urgent keyword to this relationship,” the knowledge system can keep getting refined.

The direction is right.

But in an enterprise environment, applying those changes immediately is risky. One term change can affect SQL generation, dashboards, reports, and permission policies.

So in DataNexus, conversational edits should not go straight into operational knowledge. This flow is safer.

user edit request
  → create change candidate
  → analyze impact scope
  → data owner review
  → apply to Glossary / Ontology after approval
  → resync related indexes

AI should be able to help edit knowledge. But an operational knowledge base is not a personal notebook. It is an organizational standard, so approvals and history need to remain.

This is where Knowledge Graph UI meets Trust UX.

The graph is not just a screen that shows relationships. It becomes a safety layer that shows how far a change will spread.

Accuracy and screen review are separate#

In NL2SQL evaluation, I look at EX, or Execution Accuracy. It checks whether the generated SQL returns the same result as the expected answer.

But even when EX passes, the screen can still be hard to trust.

The SQL is right, but the wording is stiff.

The number is right, but the screen does not show the basis for it, so the user cannot make the next judgment.

The source exists, but if it is hard to find, the user has to ask again.

So DataNexus separates accuracy from screen review.

I temporarily called this screen review check the Taste Gate. Separate from whether the answer is correct, it asks whether the user can review the answer and move on to the next judgment.

MetricWhat it checks
EXWhether the SQL execution result matches the expected answer
CQWhether the ontology holds up against Competency Questions
Schema ComplianceWhether generated SQL stays inside the allowed schema and policy
Hallucination RateWhether it invents columns, terms, or sources
Query Router AccuracyWhether the question went to the right flow among SQL/Graph/Glossary/Vector
Taste GateWhether the user can review the answer and move on to the next judgment

If EX passes but the user still asks, “Can I trust this?”, the product experience is not good enough.

On the other hand, even when SQL fails, the user can still understand the system state if fallback is clearly marked and the review path stays open.

That is why EX can pass and still be checked again at the Taste Gate.

Moving this principle onto the screen#

I summarized the screen structure like this.

The first area I touched was the answer card. The top row shows trust state and source. The middle has the business summary and chart. SQL and Trace open from the bottom accordion.

+-----------------------------------------------------------+
| [Status: PASS / REVIEW / FALLBACK] | [Source: Data Catalog] |
+-----------------------------------------------------------+
| "Net revenue this quarter was KRW 120B."                  |
| [Chart: Bar / Line auto-rendered]                         |
+-----------------------------------------------------------+
| ▶ Show Evidence: SQL & Trace                              |
+-----------------------------------------------------------+
|  [Tabs] 1. SQL  2. Data Lineage  3. Computation  4. Graph |
+-----------------------------------------------------------+
| [👍 / 👎]       [Request Revision]  [Export Query]        |
+-----------------------------------------------------------+

An executive can stop at the top two blocks. An analyst opens the accordion and reads SQL. A DBA checks the passed-through tables in the Data Lineage tab. A data owner looks at the term and metric relationships in the Graph tab.

The feedback buttons are the same kind of path as Vanna’s Self-Learning loop. Revision feedback remains as a training-data candidate.

When DataNexus is attached on top of an existing in-house ERP or MES, replacing the screen is not the right move. It should sit on top of the current work screen.

+-----------------------------------------------------------+
| [DataNexus Overlay Bar]  (pinned to top, always visible)  |
+-----------------------------------------------------------+
| [Search] "Show me current inventory for Product A"        |
|                                                           |
|  +-------------------------------------------+            |
|  | Answer summary...                          |  ← popup   |
|  | [Source: DW/DM] [Status: PASS]             |            |
|  | [View SQL] [View Trace] [View Graph]       |            |
|  +-------------------------------------------+            |
+-----------------------------------------------------------+

The user does not move to a separate analytics page. They query directly on top of the screen they are already using. Because the existing system is not heavily modified, rollout discussions become simpler.

Start with the summary. When more review is needed, open SQL, data lineage, or the knowledge graph. Do not dump everything at once.

Do not ship an answer without a source. At least one of SQL, document ID, glossary term, or graph node should stay attached.

The first status labels were PASS, REVIEW, and FALLBACK. The label should let the user move to the next judgment without reading a long explanation.

What the result screen should show#

In short, this is what should remain.

  • Keep the answer text short. It should read more like a colleague giving a quick explanation than a report. AI should not just jump to the conclusion. It should leave steps for checking the evidence.

  • SQL can be collapsed, but it should not be removed. If the answer has no SQL, such as a catalog term definition, do not fabricate one.

  • Trace and status labels belong together. The screen should distinguish a DB result, a catalog answer, and fallback. PASS, REVIEW, and FALLBACK are enough.

  • The source shows what the answer is based on. Today, the sources are DW/DM tables and Data Catalog Glossary Terms. Later, documents, data lineage, and knowledge graph nodes can be added.

  • The knowledge graph is not a separate visualization decoration. It is a path for exploring answer evidence. It should show how Concepts, Relationships, Properties, Rules, and Metrics connect to actual data.

  • Interactive editing is useful, but it should not update operational knowledge immediately. Change candidates, impact scope, data owner approval, and resync flow are needed.

If the answer does not show its evidence, it eventually becomes a chatbot without grounding.

The problem I want to solve in the DataNexus Chat UI is not “making AI write better answers.”

It is making sure users can review the answer, doubt it, and trace it again when needed.
