11. AI answers can be wrong. How should they appear on screen?

How the DataNexus Chat UI shows answer evidence. SQL, Runtime Trace, fallback state, permission scope, sources, and knowledge graphs are shown together as evidence behind an answer.

Up to post 10, I had been checking where SQL generation most often goes wrong.

I ran BIRD and Spider, changed models, and grouped wrong answers into types like COLUMN_BINDING, VALUE_BINDING, and DERIVED_METRIC. After staring at those SQL generation errors for a while, another question came up.

How should generated SQL and answer evidence appear on the screen?

By focusing on accuracy tests, I had not looked closely enough at the screen flow for checking evidence. If a data analysis AI is going to be used at work, users need to check the evidence. Even when the number is correct, nobody can be sure if they cannot tell where it came from. In the end, they run the query again.

For the DataNexus Chat UI, writing a clean answer was not enough. Users needed the evidence behind the number and the sentence on the same screen.

Trust here does not come from the tone of the answer. It comes from a screen where users can check the answer themselves. They need to see SQL, sources, permission scope, validation state, and fallback state beside the answer. If that information is missing, even a correct number is hard to use at work.

I first looked at other products#

As of May 2026, I reviewed domestic and global products again.

Global products

  • Databricks Genie
  • Snowflake Cortex Analyst
  • Strategy(MSTR) AUTO
  • ThoughtSpot Spotter 3
  • Vanna.ai

Domestic products

  • Qurify(SmartMind)
  • MISO(GS Neotek & 52g)

While checking product docs and demos, I kept returning to the same question.

What does a user need to see before trusting an AI answer?

The products expressed it differently, but they did not simply put an answer on the screen.

  • Databricks Genie uses Trusted assets to label answers that use verified SQL queries or SQL functions as Trusted. Genie Space users can ask questions about data stored inside the company in natural language. In the Data and Instructions tabs of a Genie Space, they can check context such as tables, columns, instructions, SQL queries, and functions. Unity Catalog permissions also apply, so only data the user is allowed to see is used in the answer.

  • Snowflake Cortex Analyst mattered more for its API response structure than for its UI. The fields make it clearer what the screen can show as evidence. The REST API separates response content into text, suggestion, and sql. SQL content can include fields such as verified_query_used, and the response also includes warnings and response_metadata.question_category. For questions where SQL is hard to generate, it can return a suggestion instead of SQL.

  • Strategy(MSTR) AUTO was useful in its Auto Dashboards and Strategy AI docs and demos. In the public docs, features such as derived metrics, selectors, auto narratives, and auto answers continue to expand. The Interpretation area I saw in demos was especially useful. Showing how a user question was mapped to Metrics and Dimensions helped shape the Trace UI in DataNexus.

  • ThoughtSpot Spotter 3 was closest to this post’s question: how should answer evidence appear on screen? Spotter 3 handles structured warehouse data together with information scattered across tools such as Slack, Salesforce, Jira, and SharePoint. The direction in the public material was clear. It pulls work-tool context into the answer, checks the result, and then moves into follow-up analysis.

  • Vanna.ai was useful for checking how the learning loop and permission handling work. The Vanna 2.0 docs explain a structure where the LLM is connected to tools such as SQL execution and chart generation. Tool Memory stores successful question-SQL-result bundles in a form that can be reused. User-specific permissions are applied throughout the flow. The public README also shows streaming progress, SQL code blocks, tables, charts, and natural-language summaries in one flow.

The domestic products approached the problem a little differently. I did not have access to internal material, so I stayed with what was available in public product pages and release notes.

  • Qurify can be read as a Text-to-SQL product for talking with databases in natural language. Based on the 0.4.4 release notes, users can edit, run, and copy generated SQL. They can also continue follow-up questions by citing execution results. CSV download and tool call argument display are available too. For DataNexus, its visible SQL flow and follow-up handling were useful references.

  • MISO is more of an AI app builder for work automation than a dedicated NL2SQL product. Based on the public site, it emphasizes on-premises enterprise AI platforms and Workflow App / Agent App cases. Its report automation and document search cases were useful when shaping operational AI screens for DataNexus.

After organizing these product patterns, the elements DataNexus needed on screen became clearer. Before refining the answer text, the screen had to let users check SQL, source, validation state, permission scope, and fallback state.

This also made me restate what the BI industry usually calls the semantic layer. A semantic layer usually connects business terms and metric definitions to SQL or BI screens. DataNexus Glossary, Ontology, and Knowledge Graph go one step further. They tie column- and table-level metadata to business rules and relationships between concepts.

One more thing was needed: a screen for checking relationships directly.

A knowledge graph is a screen for following evidence#

By evidence, I mean the network that lets users follow the concepts used in an answer, where those concepts came from, and which tables or columns are connected to them. At first, I thought of the knowledge graph as a separate analysis screen.

I pictured a screen where users zoom and pan around nodes and edges while exploring relationships. It looks good in a technical demo. In real work, though, a graph does not become useful just because it is visually polished.

For a graph to earn its place on the screen, it has to answer questions like this.

  • Which concept did this answer reference?
  • Which data is that concept connected to?
  • Who defined this relationship, and where is the evidence for it?
  • If a formula or metric definition changes, which answers are affected?

Traditional search is good at finding keywords. If a document contains the word “net revenue”, search can find that document. But search alone has a hard time understanding how net revenue relates to gross revenue, returns, discounts, and allowances.

For business answers, the relationships around a term can matter more than the document that contains the term. To follow those relationships on screen, DataNexus needs ontology and a knowledge graph.

An ontology is the blueprint for concepts and relationships used inside an organization.

For example:

  • Concept: customer, product, order, net revenue
  • Relationship: a customer creates an order, net revenue is calculated from gross revenue
  • Property: order timestamp, payment method, order status
  • Rule: a refund can happen only after payment is completed
  • Metric: average order value, net revenue, year-over-year growth rate

A knowledge graph connects that blueprint to actual data. Concepts become nodes, and relationships become edges. Documents, tables, columns, metric definitions, and formulas are tied into one network.

This is different from trying to replace the existing RDBMS with a knowledge graph.

An RDBMS is the right tool for storing values and handling transactions. A knowledge graph explains what those values mean in business terms and how they relate to other concepts.

This distinction mattered for DataNexus.

SQL can calculate the value. To explain which business concept that value came from, DataNexus also needed the knowledge graph. I could not treat one as a substitute for the other.

A graph also needs visible sources#

On a screen that shows a knowledge graph, concepts and evidence need to appear together.

When a user clicks a node or relationship, the screen should let them check where that relationship came from right away. For example, clicking the “net revenue” node cannot stop at showing connected nodes. It also has to show the glossary term, document sentence, or related columns behind the definition.

When a graph works this way, users do more than look at a few connected nodes. They can follow the definition, columns, and document sentences behind the answer on the same screen.

Suppose someone asks, “What is net revenue?” and the answer stops like this, with no evidence for the user to inspect.

Net revenue is the amount after subtracting returns, discounts, and allowances from gross revenue.

Even if that sentence is correct, the user will ask again.

  • Where did this definition come from?
  • Is this our company’s definition?
  • Which columns is it connected to?
  • If this definition changes, which dashboards are affected?

So the answer needs at least this kind of information beside it.

source: Data Catalog Glossary
term: net revenue
source_id: urn:li:glossaryTerm:net_sales
related concepts: gross revenue, returns, discounts, allowances
linked columns: orders.amount, orders.refund_amount, orders.discount_amount
validation: PASS

Here, users follow the answer’s source through the knowledge graph.

A number alone is not enough to trust an answer#

Genie’s Trusted label, Snowflake’s validation fields, and Qurify’s visible SQL flow made the direction clearer: the result screen needs to show evidence together with the answer.

Suppose the user asks:

What was the total order count in 2024?

If the AI only answers:

The total order count in 2024 was 42.

The sentence looks like an answer. In practice, users do not stop there. These questions come up immediately.

  • Which table did it query?
  • How did it filter for 2024?
  • Were cancelled orders included?
  • Did this number come from the sample DB or the production DB?
  • Was SQL actually executed, or did the model just write a sentence?

A single number, “42”, does not tell the user whether it is safe to put that value into a meeting deck.

Genie’s Trusted label and Snowflake’s validation fields in the response JSON come from the same concern. Without evidence on the screen, users may trust the answer without grounding or stop using it at all. For a work tool, both outcomes are exactly what the interface should avoid.

So the DataNexus Chat UI does not stop at the answer result. It also shows the SQL query and execution order together. Source, fallback state, validation state, and permission scope appear in the same place.

Why SQL should stay visible#

When designing an NL2SQL product, it is tempting to hide SQL from the screen.

Business users usually do not inspect SQL directly. The screen also gets more complex. It can also make the product feel less like AI is doing the analysis on the user’s behalf.

But in an enterprise environment, SQL should not be completely hidden.

In DataNexus, SQL is not treated like an internal log that only developers inspect. It can be collapsed or expanded inside the answer.

The answer area places the natural-language answer first.

The total order count in 2024 was 42.

Below that, the screen places an expandable SQL block.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

Users want to see the result first. If SQL appears above the answer, the screen feels heavy. But when someone needs to inspect the number, SQL should be available on the same screen.

SQL is the first piece of evidence someone checks when asking, “Where did this number come from?” Analysts or DBAs open it when needed, and when an error happens, they can reproduce it with the same query.

The next step after showing SQL is showing how the SQL was formed. Not every user reads SQL. The first thing users need to check is how the question was bound to values.

Interpreted criteria

Period: 2024-01-01 - 2024-12-31
Metric: net revenue
Formula: gross revenue - returns - discounts - allowances
Data: orders
Permission: current user's permission scope

This table connects directly to the VALUE_BINDING error taxonomy in DataNexus. Users should be able to check how “2024” became a date condition, how “net revenue” became a formula, and how “VIP customer” became a segment rule. If the binding is wrong, they should be able to fix the period or metric definition inline and run again before reading the full SQL.

The SQL tab also needs dry-run information. Showing a query is not enough to tell whether it is safe to run. Estimated rows, full-scan risk, policy check results, and cost level help users see execution risk before running it.

SQL Inspect

EXPLAIN summary: Index Scan
Estimated rows: 124,503
Full-scan warning: none
Estimated cost: low
Policy check: passed

Trace lets users inspect the answer flow#

Here, Trace means the execution record that shows how a user question was processed. This definition matters because in the DataNexus architecture, an AI answer does not end with one model call. The system checks ontology and the catalog, then moves through DW/DM lookup, NL2SQL, and validation logic depending on the question type.

The Snowflake fields mentioned earlier, such as verified_query_used, question_category, and warnings, serve the same purpose. They provide fields the caller can use to trace the processing path from the response.

When a user question comes in, the system first classifies the intent. The processing order changes by question type. If the user asks for a metric definition, the system queries the catalog first. If the question needs an actual number, the route goes to the DW/DM DB.

“What is net revenue?” should reach the catalog glossary before SQL runs.

“What was the total order count in 2024?” runs a SELECT against the DW/DM DB.

This difference needs to be visible on the screen right away.

Not every internal log needs to be exposed. For users, source, validation, and fallback are enough as a starting point. But the detailed Trace that operators and analysts open needs more information.

For example, a DB lookup answer needs this kind of record.

Runtime Trace
trace_id: dnq_20260523_001
route: NL2SQL
source: DW/DM
source_id: mart.orders
semantic_context: Data Catalog Glossary + Ontology
verified_query: none
sql_execution: success
validation:
  schema_compliance: pass
  policy_check: pass
  source_binding: pass
permission_scope:
  user: sales_analyst
  rls_policy: sales_region_allowed
  masked_columns: []
fallback: false
warnings: []

If the answer came from the catalog, it looks like this.

Runtime Trace
trace_id: dnq_20260523_002
route: Glossary
source: Data Catalog Glossary
source_id: urn:li:glossaryTerm:net_sales
sql_execution: not_applicable
validation:
  source_binding: pass
  policy_check: pass
permission_scope:
  glossary_visibility: internal
fallback: false
warnings: []

The user screen only needs to show part of this.

source: DW/DM
validation: PASS
fallback: false

Route needs finer categories#

In the initial design, the route centered on NL2SQL and Glossary. In production screens, the route needs to be more granular. Even inside PASS, an answer based on a verified query and an answer based on newly generated raw SQL do not carry the same level of trust.

route enum v2

semantic_routed
verified_query_matched
raw_sql_generated
raw_sql_fallback
glossary_only
graph_traversal
fallback_demo
unsupported_metric

If this distinction is used as a simple reassurance badge, it loses meaning. It records whether this answer used the semantic layer, matched a verified query, or fell back to raw SQL. Raw fallback will happen in production. A trust UI should show the path the answer actually took instead of implying that every answer used the best path.

The Interpretation area in Strategy(MSTR) AUTO, mentioned earlier, looks like a screen built in the same context. It shows how the question was interpreted as Metrics and Dimensions so people can inspect it later. In an enterprise setting, AI that cannot show the criteria behind an answer is hard to use. Without Trace, users end up going back to SQL or documents because they cannot verify the source from the answer.

Reduce ambiguity before answering#

Adding evidence after the answer is not enough. Real work questions are often unclear from the start. A request like “show me net revenue” hides period, accounting standard, deduction rules, and data source. In that case, DataNexus should show interpretation candidates before generating SQL.

[Question] Show me 2024 net revenue

DataNexus checks once more.

○ net_revenue_v1: gross revenue - returns - discounts - allowances
○ net_revenue_KR: Korean accounting standard, includes tax treatment
○ net_revenue_global: IFRS standard

[Run with this definition] [Choose manually]

This dialog is a step for confirming intent when several business definitions exist. The ontology creates the candidates here. It gathers candidate definitions, linked columns, permission scope, and validation state, then presents them in a form the user can choose.

The selected criteria also need to carry forward inside the same thread.

{
  "carry_forward": {
    "accepted_metric": "urn:li:glossaryTerm:net_sales",
    "accepted_period": "2024",
    "accepted_filters": ["region = KR"],
    "rejected_interpretations": ["gross_sales_only"],
    "scope": "current_thread_only"
  }
}

As the conversation gets longer, users expect the “net revenue definition I just selected” to remain in effect. DataNexus should show whether it is carrying that criterion into the next question or interpreting the new question from scratch.

Permission scope has to be visible too#

The source alone does not settle the trust problem.

Tables or documents the user cannot access should not be used as evidence. Even for the same question, sales, finance, and executive accounts may see different data. That can change both the answer and the evidence.

That is why Trace records permission scope along with source.

permission_scope:
  user: sales_analyst
  allowed_domains: [Sales]
  row_level_filter: region IN ('KR-Seoul', 'KR-Gyeonggi')
  masked_columns: [customer_phone, customer_email]

This information does not need to be shown to every user. But operators and audit owners need to be able to check it later.

A trustworthy answer should first show which data it came from. It also needs to explain whether the answer stayed inside the user’s permission scope.

Fallback state should appear on screen#

A failed result is most dangerous when it looks like success.

For example, the DB connection may drop while the screen still shows a sentence like “The total order count in 2024 was 42.” Without a failure label, the user may treat that value as a real query result.

Snowflake can return suggestion content instead of forcing SQL when the AI is not confident. Qurify keeps a flow where users edit and run generated SQL. The implementations differ, but both keep failure or uncertainty visible on the screen.

When DB lookup fails, DataNexus shows the fallback state on the screen.

There are cases where the system still needs to show an answer after a failure. In a demo environment, a prepared answer may be displayed. In those cases, the screen has to clearly mark whether the answer came from a real DB result or fallback.

Runtime Trace
source: fallback demo response
sql_execution: failed
validation: FALLBACK
fallback: true
warnings:
  - DB connection failed. Demo response was used.

In a demo environment, missing this fallback label makes it easy to mistake a prepared answer for a live DB result.

Users need to know how far the system got. Operators and developers need to track the failure point and fallback frequency.

Operational labels can be split one level further. FALLBACK · demo_response and FALLBACK · sql_execution_failed mean different things to the user. If the system lacks enough information or the user’s permission blocks the answer, it is better to separate that state as ABSTAIN.

A visible source makes the answer usable at work#

In DataNexus, the source that needs to be shown most often is the data catalog.

Take the term “net revenue”. Each company has slightly different criteria. One company may subtract only returns and discounts. Another may include allowances or tax treatment in the calculation.

DDL describes the shape of tables and columns, but it does not carry this kind of business rule.

The table exposes columns like amount, discount_amount, and refund_amount. A separate business definition has to decide how those columns become net revenue.

That is why the term definition has to come from the catalog.

When someone asks “What is net revenue?” in the Chat UI, and the answer came from the catalog glossary, the screen should show source: Data Catalog Glossary with it.

source: Data Catalog Glossary
term: net revenue
source_id: urn:li:glossaryTerm:net_sales
owner: Sales Data Domain

The user confirms whether the answer came from a glossary managed by the organization.

In ontology-based NL2SQL, meaning is not left only to the model’s guess. The system brings in the meaning the organization has already defined at runtime and connects it to the answer.

View values and relationships inside one answer card#

Putting a knowledge graph on the screen does not mean every user should start from the graph. Most users look at the answer and number first, and the graph follows after that.

In DataNexus, the answer card lets users switch between summary, SQL, catalog, and graph tabs. The value appears first, and the SQL tab explains how that value was calculated.

The total order count in 2024 was 42.

The calculation method is checked through SQL.

SELECT COUNT(*) AS order_count
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND order_date < DATE '2025-01-01';

When semantic relationships matter, users move to the graph tab.

Order
 ├─ has property: order_date
 ├─ has property: order_status
 └─ counted as: Order Count

When the company standard matters, users check the term definition in the catalog tab.

Glossary Term: Order Count
Definition: Number of order records created within the selected period
Owner: Sales Data Domain

If summary, SQL, catalog, and graph are all expanded at once, the screen gets heavy. So the answer summary comes first. From there, users open SQL when a number looks strange, the catalog when a business definition matters, and the graph when relationships or impact scope need inspection. Users usually check the result first and follow the evidence only when something feels suspicious.

Conversational edits go through review before they apply#

If a user says, “Change this node type to strategic customer” or “Add the urgent keyword to this relationship,” the system can create a change candidate for operational knowledge.

The direction is right. But in an enterprise environment, applying changes immediately is risky. One term change can affect SQL generation, dashboards, reports, and permission policies.

So in DataNexus, conversational edits do not go straight into operational knowledge. They go through this process.

user edit request
  → create change candidate
  → analyze impact scope
  → data owner review
  → apply to Glossary / Ontology after approval
  → resync related indexes

A way to ask AI for knowledge edits improves operational efficiency. Still, treating an operational knowledge base like a personal notebook breaks the approval flow. It changes organizational standards, so approvals and change history have to be managed together.

In an operational screen, the graph cannot stop at showing relationships. It also has to help people check change impact scope.

Accuracy and reviewability should not be merged into one metric#

NL2SQL evaluation usually measures EX, or Execution Accuracy. It checks whether the generated SQL returns the same result as the expected answer.

But EX is an offline evaluation metric. It is measured on benchmarks that have a gold SQL query or a gold execution result.

In production runtime, there is usually no gold answer. A question asked in real time does not come with a prepared gold SQL query.

So PASS on the operational screen should not mean the same thing as EX. EX compares benchmark answers offline, while runtime validation checks whether the current question passed execution, policy, and source checks in production.

PASS
- SQL execution or catalog lookup succeeded
- Schema Compliance passed
- permission / RLS policy passed
- source or verified query / glossary / graph node matched
- no critical warning

REVIEW
- SQL ran, but binding is uncertain
- verified query was not used
- source matching is weak
- question is ambiguous or router detoured
- data owner or analyst review is needed

FALLBACK
- SQL execution failed
- DB was not queried
- suggestion only
- demo response
- substitute answer, not live operational data

Even when the SQL result is right, the screen can still make the answer hard to use at work. If the natural-language answer reads like a stiff report sentence, the reader has to reinterpret it. If the number is correct but the screen does not show the calculation basis, the user cannot decide whether to put it into a report or meeting deck. If the source is hard to find, the user ends up asking a data owner or asking the system again.

TrustSQL explains that Text-to-SQL reliability requires correct SQL when a question can be answered and abstention when it cannot. LinkedIn’s enterprise Text-to-SQL case also builds a knowledge graph from schema, documents, query logs, wiki, code, and domain knowledge. The case also shows a chatbot UI handling different user intents and follow-up questions together.

DataNexus calls this criterion the Taste Gate. Even when EX passes, an answer with missing evidence or an unclear source does not pass the Taste Gate.

MetricWhat it checks
EXWhether SQL execution matches the gold answer in an offline benchmark
CQWhether the ontology passes Competency Questions
Schema ComplianceWhether SQL stays inside the allowed schema and policy
Hallucination RateWhether it invents columns, terms, or sources
Query Router AccuracyWhether the question went to the right flow among SQL, Graph, Glossary, and Vector
Runtime ValidationWhether execution, policy, source, and warning checks pass at runtime
Taste GateWhether the user can review the answer and move on to the next judgment

If EX passes but the user still cannot trust the answer right away, the product has not done its job. On the other hand, even when SQL fails, a clear fallback label and visible evidence let the user understand the system state. That is why an answer that passes EX still goes through the Taste Gate.

Applying this principle to the actual screen#

The final screen structure looks like this.

I rebuilt the answer card layout first. The top row contains trust state and source, the middle contains the business summary and chart, and SQL and Trace sit at the bottom of the card as sections the user can expand when needed.

+---------------------------------------------------------------+
| [Status: PASS / REVIEW / FALLBACK / ABSTAIN] | [Source: Data Catalog] |
+---------------------------------------------------------------+
| "Net revenue this quarter was KRW 120B."                     |
| [Chart: Bar / Line auto-rendered]                             |
+---------------------------------------------------------------+
| ▶ Show Evidence: SQL & Trace                                  |
+---------------------------------------------------------------+
| [Tabs] 1. SQL  2. Data Lineage  3. Metric Logic  4. Graph      |
+---------------------------------------------------------------+
| [👍 / 👎]       [Request Revision]  [Export Query]            |
+---------------------------------------------------------------+

An executive can start the business judgment from the status and summary near the top. Analysts, DBAs, and data owners who need more detail move down to the tabs. Analysts open the SQL tab in the expandable area to inspect queries, DBAs check the tables used in the Data Lineage tab, and data owners review terms and metric relationships in the Graph tab.

The feedback buttons play a role similar to Vanna’s Self-Learning loop by collecting user corrections as learning material. In DataNexus, that feedback goes to the review queue as a candidate instead of being applied directly.

This structure needs to separate the default card from the Inspect layer. The default card should contain only the core answer, PASS/REVIEW/FALLBACK/ABSTAIN label, primary source, and chart. The Inspect layer should hold SQL, Trace, permission scope, Lineage, Metric Logic, Graph, and Source highlights.

LayerContent
Primary cardCore answer, status label, primary source, chart
Inspect layerSQL, Trace, permission scope, Lineage, Metric Logic, Graph, Source highlight

Sources also should not stop at one document-level link. Citation grain means deciding how far down the evidence should go. Document-level, paragraph-level, sentence-level, and number-level sources all lead to different review behavior. Because DataNexus has DataHub URNs and table.column FQNs, it should move toward sentence- and number-level evidence.

But if words like Trusted, PASS, or verified are attached too easily, users may trust the label without checking the content. Labels should appear only when their validation conditions are met, and operational metrics should still track how trustworthy the source is and whether it may have been poisoned.

The summary appears first, and users who need more detail can open SQL, data lineage, or the knowledge graph in order. If every piece of evidence is expanded from the start, the review tools overwhelm the answer.

An answer without a source should not appear on the screen. It has to connect to at least one of SQL, document ID, glossary term, or graph node.

The default status labels are PASS, REVIEW, and FALLBACK. A state where not answering is safer is separated as ABSTAIN.

The label alone should help users decide what to do next.

PASS: usable for business judgment now
REVIEW: open the result and verify once
FALLBACK: not a real DB result or not a normal execution result
ABSTAIN: no answer because context is insufficient or permission blocks it

Here, PASS does not mean EX passed. The screen uses PASS only when execution succeeded, policy validation passed, the source matched, and there was no critical warning.

Exposing raw confidence scores from the model is another option. But an uncalibrated score is hard for users to interpret. PASS, REVIEW, and FALLBACK make it clearer whether the answer is ready to use, needs a check, or is not a real DB result.

Keep the answer text short. It should read like a colleague giving a quick explanation. SQL can be collapsed, but it should not disappear from the screen. Trace and status labels distinguish a DB result, a catalog answer, and fallback.

The operator screen needs to look beyond one answer card. It should show which answers users inspected, which citations they clicked, which routes often fall back, and whether REVIEW labels later became real errors.

The question that remains while designing the DataNexus Chat UI is simple: what does the user need to check before using this answer at work? Refining the answer sentence comes after the screen shows those checks clearly.
